# 결제_배송_분쟁_설계_확인사항

## 목적

이 문서는 Haggle의 다음 단계인 `결제`, `배송 추적`, `분쟁 처리`를
`다른 팀이나 외부 개발자가 봐도 표준처럼 느껴지는 구조`로 만들기 전에
합의해야 할 항목을 고정하기 위한 문서다.

현재 기준:
- 작업 기준 브랜치: `codex/commerce-foundation`
- 기준 베이스: `origin/main`
- 방향:
  - 결제는 `x402` 중심
  - `Stripe` 는 fallback / merchant 선택지
  - 배송과 분쟁은 Haggle 고유 로직이 아니라 `표준 엔티티 + 이벤트 모델` 중심

## 설계 원칙

### 1. 결제, 배송, 분쟁은 엔진 부속 기능이 아니다
- 협상 엔진은 `가격/조건 합의`까지 책임진다.
- 결제는 `settlement`.
- 배송은 `fulfillment`.
- 분쟁은 `post-order resolution`.
- 따라서 각 영역은 별도 상태 머신과 별도 이벤트 계약을 가져야 한다.

### 2. Core 와 Provider 를 분리한다
- Haggle Core 는 공통 인터페이스와 상태 전이만 정의한다.
- 실제 처리 수단은 provider 로 분리한다.
- 예:
  - `payment provider = x402 | stripe`
  - `carrier provider = ups | fedex | dhl | easyship`
  - `dispute evidence store = haggle | third-party`

### 3. 외부 표준처럼 보여야 한다
- 네이밍은 vendor-neutral 해야 한다.
- 엔티티는 transport-agnostic 해야 한다.
- 이벤트는 idempotent 해야 한다.
- 상태 전이는 테스트 가능해야 한다.

## 1. 결제

### 1.1 결제 방향
- 1순위: `x402`
- 2순위: `Stripe`

### 1.2 왜 x402가 중심이어야 하는가
- agent-to-agent commerce 문맥에 더 자연스럽다.
- payment capability 를 machine-readable 하게 노출하기 쉽다.
- HNP/UCP 표준화 방향과 더 잘 맞는다.

### 1.3 Stripe 를 병행하는 이유
- 일반 사용자와 카드 결제 진입장벽이 낮다.
- 판매자 온보딩 속도가 빠르다.
- MVP 이후 확장 시 현실적인 fallback 이 된다.

### 1.4 확인해야 할 결정
- buyer 가 checkout 시 payment rail 을 고르는가
- seller 가 listing 수준에서 허용 rail 을 고르는가
- 기본 결제수단이 `x402 only` 인가, `x402 preferred + stripe enabled` 인가
- payment authorization 과 capture 를 분리할 것인가
- escrow 를 결제 계층에 넣을 것인가, 별도 settlement 계층으로 둘 것인가

### 1.5 권장 방향
- listing / seller 레벨에서 허용 payment rails 선언
- 기본값은 `x402 preferred`
- Stripe 는 `fallback_enabled` 옵션으로 노출
- Haggle Core 는 아래 엔티티만 고정
  - `PaymentIntent`
  - `PaymentAuthorization`
  - `PaymentSettlement`
  - `Refund`

## 2. 배송 추적

### 2.1 배송은 무엇을 표준화해야 하는가
- carrier 자체가 아니라 배송 상태 모델
- tracking event envelope
- estimated / actual timestamps
- buyer, seller, system 이 보는 canonical shipment state

### 2.2 최소 Shipment 모델
- `shipment_id`
- `order_id`
- `carrier`
- `tracking_number`
- `tracking_url`
- `status`
- `events[]`
- `eta`
- `delivered_at`

### 2.3 canonical shipment status 권장안
- `LABEL_PENDING`
- `LABEL_CREATED`
- `IN_TRANSIT`
- `OUT_FOR_DELIVERY`
- `DELIVERED`
- `DELIVERY_EXCEPTION`
- `RETURN_IN_TRANSIT`
- `RETURNED`

### 2.4 확인해야 할 결정
- Haggle 이 label 생성까지 하는가
- tracking 은 partner webhook pull-through 인가
- 배송 이벤트는 polling 보다 webhook 우선인가
- negotiation 결과 중 배송 조건 변경이 order snapshot 에 어떻게 잠기는가

## 3. 분쟁 처리

### 3.1 분쟁은 reputation field 가 아니라 case system 이어야 한다
- 현재 엔진 문서에는 `n_dispute_losses` 같은 reputation input 이 있다.
- 운영 시스템으로 가려면 실제 `case` 엔티티가 필요하다.

### 3.2 최소 Dispute 모델
- `dispute_id`
- `order_id`
- `reason_code`
- `status`
- `opened_by`
- `opened_at`
- `evidence[]`
- `resolution`
- `resolved_at`

### 3.3 canonical dispute status 권장안
- `OPEN`
- `UNDER_REVIEW`
- `WAITING_FOR_BUYER`
- `WAITING_FOR_SELLER`
- `RESOLVED_BUYER_FAVOR`
- `RESOLVED_SELLER_FAVOR`
- `PARTIAL_REFUND`
- `CLOSED`

### 3.4 확인해야 할 결정
- dispute trigger 는 언제 열리는가
  - 미배송
  - 설명과 상이
  - 파손
  - 사기 의심
- 분쟁 증거는 어떤 형식을 받는가
  - 이미지
  - 텍스트
  - tracking snapshot
  - payment proof
- resolution authority 는 누구인가
  - seller self-resolution
  - Haggle mediation
  - provider-led resolution

## 4. 표준화 방식

### 4.1 문서 세트 분리 권장
- `결제_코어_프로필.md`
- `배송_이벤트_모델.md`
- `분쟁_상태_모델.md`
- `결제_배송_분쟁_적합성_테스트.md`

### 4.2 transport-agnostic envelope 필요
- `id`
- `type`
- `version`
- `occurred_at`
- `subject_id`
- `idempotency_key`
- `provider`
- `payload`

### 4.3 provider-specific 필드는 extension 으로 격리
- x402 전용 필드
- Stripe 전용 필드
- carrier 전용 raw payload

## 5. 구현 순서 권장

1. canonical entity / state machine 문서 확정
2. DB schema 초안
3. provider interface 추상화
4. x402 adapter
5. Stripe adapter
6. shipment event ingestion
7. dispute case lifecycle
8. conformance tests

## 6. 이번 브랜치에서 먼저 정할 질문

우선 아래 6개를 확정하면 구현이 빨라진다.

1. 결제 기본 rail 은 `x402 preferred + stripe optional` 로 갈지
2. 결제 authorization / settlement / refund 를 분리할지
3. 배송은 `tracking-only` 부터 시작할지, label creation 까지 할지
4. 분쟁은 `manual review 중심` 으로 시작할지, 일부 rule-based automation 을 넣을지
5. dispute 결과가 reputation 으로 언제 반영되는지
6. Core 표준 문서에서 provider-specific 필드를 어느 수준까지 허용할지
