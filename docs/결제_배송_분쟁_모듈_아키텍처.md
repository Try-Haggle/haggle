## 목적

이 문서는 Haggle의 다음 단계인 `결제`, `배송 추적`, `분쟁 처리`를
한 저장소 안의 부속 기능이 아니라, 필요하면 각각 독립 서비스로 운영할 수 있는

핵심 원칙은 단순하다.

- 각 영역은 자체 상태 머신을 가진다.
- 각 영역은 자체 이벤트를 발행한다.
- 각 영역은 provider 구현체와 core 계약을 분리한다.
- monorepo 안에서는 패키지로 시작하되, 나중에 그대로 서비스로 분리 가능해야 한다.

## 패키지 경계

현재 추가한 기준 패키지는 아래와 같다.

- `@haggle/commerce-core`
- `@haggle/payment-core`
- `@haggle/shipping-core`
- `@haggle/dispute-core`

### `@haggle/commerce-core`
- 공통 이벤트 envelope
- 서비스 경계 정의
- 도메인 공통 계약
- approval policy / settlement approval 상태

### `@haggle/payment-core`
- payment rail 추상화
- payment intent / authorization / settlement / refund 타입
- payment 상태 전이
- 결제 rail 우선순위 정의

### `@haggle/shipping-core`
- shipment / shipment event 타입
- canonical tracking state
- 배송 상태 전이

### `@haggle/dispute-core`
- dispute case / evidence / resolution 타입
- 분쟁 상태 전이

## 서비스 분리 기준

각 모듈은 아래 조건을 만족해야 독립 서비스로 분리하기 쉽다.

1. 자체 DB 테이블을 가진다
2. 자체 상태 머신을 가진다
3. 자체 webhook / event ingestion 을 가진다
4. 외부 provider adapter 를 core 와 분리한다
5. 다른 모듈과는 이벤트나 stable contract 로만 연결한다

## 결제

### 원칙
- `x402` 가 기본 rail 이다
- `Stripe` 는 fallback / merchant 옵션이다
- Haggle Core 는 provider-neutral 해야 한다
- payment 단계는 rail 선택 단계가 아니라, 이미 확정된 rail 실행 단계에 가깝다

### 결제 전에 먼저 고정해야 하는 것
- seller approval policy
- buyer explicit approval
- hold / reservation policy
- shipment info SLA

즉 실제 흐름은 아래에 가깝다.

1. 협상 후보안 도달
2. `MUTUALLY_ACCEPTABLE`
3. buyer가 아래 중 하나 선택
   - 바로 승인
   - soft hold
   - 계속 협상
4. seller가 가격 예약을 걸면 `RESERVED_PENDING_APPROVAL`
5. buyer 승인
6. seller policy 확인
7. auto seller면 즉시 `APPROVED`
8. manual seller면 `AWAITING_SELLER_APPROVAL`
9. seller 승인 후 결제 실행

### 고정할 엔티티
- `PaymentIntent`
- `PaymentAuthorization`
- `PaymentSettlement`
- `Refund`

추가로 공통 계층에서 고정할 엔티티:
- `SettlementApproval`
- `SellerApprovalPolicy`

### provider 분리
- core:
  - 확정된 rail 실행
  - 상태 전이
  - canonical event
- adapter:
  - `x402`
  - `Stripe`

### 권장 운영 형태
- `payment-core` 는 shared contract
- 실제 운영 서비스는 나중에 예:
  - `payment-api`
  - `payment-worker`

## 승인 정책

### seller 설정
seller 는 리스팅 또는 계정 레벨에서 approval policy 를 설정할 수 있어야 한다.

- `AUTO_WITHIN_POLICY`
  - 정책 범위 안이면 buyer 승인 순간 최종 완료
  - 완료 후 seller 에게 알림
  - 그 시점부터 배송 정보 입력 SLA 시작

- `MANUAL_CONFIRMATION`
  - buyer 승인 후 seller 승인까지 받아야 최종 완료
  - seller 승인 시점부터 배송 정보 입력 단계 진입

### 배송 정보 입력 SLA
- approval 완료 시점으로부터 계산한다
- 예:
  - `shipment_input_due_days = 2`
  - `shipment_input_due_at = approved_at + 2 days`

## hold 와 reservation

### 핵심 원칙
- `hold` 와 `reservation` 은 같은 개념이 아니다
- 기본 `hold` 는 구매자 보류 상태이며 가격 보장을 의미하지 않는다
- 가격 보장은 seller가 명시적으로 잠가준 `reservation` 에서만 발생한다

### soft hold
soft hold 는 구매자가 거래를 바로 승인하지 않고 잠깐 보류하는 상태다.

- 상태: `HELD_BY_BUYER`
- 목적:
  - 다른 거래와 비교
  - 다른 seller와의 카운터 오퍼 재료로 사용
  - 시간 경과에 따른 효용 변화 관찰
- 특징:
  - `held_snapshot_price` 와 `held_snapshot_utility` 는 저장한다
  - 재개 시점에는 항상 `현재 효용` 기준으로 다시 계산한다
  - 예전 가격으로 자동 복귀하지 않는다

즉 soft hold 는 `좋아 보이는 제안의 스냅샷 저장` 이지, 계약적 약속이 아니다.

### reserved hold
reserved hold 는 seller가 일정 시간 동안 가격을 묶어주는 명시적 약속이다.

- 상태: `RESERVED_PENDING_APPROVAL`
- 필수 메타데이터:
  - `held_snapshot_price_minor`
  - `held_at`
  - `expires_at`
- 특징:
  - 만료 전까지는 해당 가격으로 buyer 승인이 가능하다
  - 만료 후에는 자동 보장이 끝나고 재계산 대상이 된다
  - 기본값으로 두면 안 되고, seller 또는 marketplace policy가 명시적으로 허용해야 한다

### 상태 머신 권장안
- `NEGOTIATING`
- `MUTUALLY_ACCEPTABLE`
- `HELD_BY_BUYER`
- `RESERVED_PENDING_APPROVAL`
- `AWAITING_SELLER_APPROVAL`
- `APPROVED`
- `DECLINED`
- `EXPIRED`

### 표준 해석
- `MUTUALLY_ACCEPTABLE`
  - seller 와 buyer 모두 거래 가능한 가격/조건대에 들어왔다는 뜻
  - 아직 최종 완료는 아님
- `HELD_BY_BUYER`
  - buyer가 승인 대신 보류를 선택
  - 이후 더 협상하다 실패해도 과거 hold 가격으로 자동 복귀하지 않음
- `RESERVED_PENDING_APPROVAL`
  - seller가 가격을 일정 시간 잠금
  - 이 경우에만 expiry 전 복귀 또는 즉시 승인 가능

### 권장 저장 필드
- `hold_kind`
- `held_snapshot_price_minor`
- `held_snapshot_utility`
- `held_at`
- `hold_reason`
- `resume_reprice_required`
- `expires_at`

### negotiation 에 반영해야 하는 요소
- auto approval seller 는 전환 속도 이점이 있다
- manual seller 는 일반적으로 friction 이 크다
- 하지만 수동 seller 라도 과거 응답이 빠르면 무조건 패널티를 주지 않는다

즉 engine / checkout layer 에서는 아래 지표를 쓸 수 있다.
- `seller_approval_mode`
- `median_response_minutes`
- `p95_response_minutes`
- `reliable_fast_responder`

이건 shipping/dispute 가 아니라 `commerce-core` 의 approval policy 로 관리하는 것이 맞다.

## 경쟁 강도 정책

### 핵심 원칙
- 외부에서 주장한 개별 `counter offer` 가격은 직접 압박 신호로 쓰지 않는다
- 경쟁 강도는 플랫폼이 검증한 수요/공급/비교 가능 리스팅 변화로 계산한다
- seller 와 buyer 는 같은 `competitive_pressure` 를 보더라도 반대로 해석할 수 있다

### seller 측 상승 압력
다음 상황에서는 seller 가 더 높은 가격을 기대할 여지가 생긴다.

- 같은 상품에 대한 활성 buyer 관심이 늘어남
- 수동 승인 seller 이지만 실제 응답 이력이 빠름
- 최근 hold 상태 buyer 가 많아 재진입 가능성이 큼

이 경우 manual seller 는 경매처럼 점진적으로 가격을 올리거나, 카운터 오퍼를 더 강하게 낼 수 있다.
단, 이건 `검증된 buyer 수요 압력` 기반이어야 하며, buyer 가 주장한 임의의 경쟁 가격 기반이어서는 안 된다.

### buyer 측 하락 압력
다음 상황에서는 buyer 가 더 낮은 가격을 기대할 여지가 생긴다.

- 더 싼 동일/유사 제품 리스팅이 계속 등장
- seller 대안 수가 증가
- 현재 listing 의 시장 내 위치가 불리해짐

이 경우 seller 카운터 오퍼는 점진적으로 내려갈 수 있다.

### 권장 계산 입력
- `active_buyer_interest_count`
- `active_seller_alternatives`
- `cheaper_comparable_listing_count`
- `demand_pressure`
- `supply_pressure`
- `competitive_pressure`

이 값들은 `commerce-core` 의 market pressure 계약에서 생성하고, engine 은 최종 정규화된 `competitive_pressure` 만 받는 것이 바람직하다.

## 승인 후 미결제 / 미이행 패널티

### 원칙
- buyer 와 seller 모두 승인 이후의 불이행에 대한 비용을 가져야 한다
- 이 패널티는 dispute 와 별도지만, 장기적으로는 온체인 신뢰 레이어에 앵커링 가능해야 한다
- penalty 요약값은 engine 의 `V_r` 와 `V_s` 에 간접 반영될 수 있어야 한다

### 패널티 대상 예시
- `BUYER_APPROVED_BUT_NOT_PAID`
- `SELLER_APPROVED_BUT_NOT_FULFILLED`
- `SHIPMENT_INFO_SLA_MISSED`
- `DISPUTE_LOSS`

### 신뢰 요약값
- `successful_settlements`
- `approval_defaults`
- `shipment_sla_misses`
- `dispute_losses`
- `settlement_reliability`

### 온체인 신뢰 프로필
장기적으로는 위 요약값을 `온체인 신뢰 프로필` 로 앵커링할 수 있어야 한다.

권장 필드:
- `wallet_address`
- `reputation_score`
- `settlement_reliability`
- `successful_settlements`
- `approval_defaults`
- `shipment_sla_misses`
- `dispute_wins`
- `dispute_losses`
- `expertise[]`
- `onchain_reference`

### 전문 분야(expertise)
신뢰는 단일 숫자만으로 충분하지 않다. seller 나 buyer 가 어떤 분야에서 강한지도 기록해야 한다.

예:
- `electronics`
- `luxury`
- `fashion`
- `collectibles`
- `automotive`
- `general`

각 분야마다 아래를 누적할 수 있다.
- `score`
- `successful_orders`
- `dispute_wins`
- `dispute_losses`

즉 "전자기기 거래에는 강하지만 명품 거래에는 약한 seller" 같은 표현이 가능해야 한다.

### 엔진 반영
- `settlement_reliability` 는 `V_r` 의 선택적 입력으로 사용한다
- approval 후 결제 미완료가 반복되면 buyer `settlement_reliability` 가 내려간다
- seller 의 배송 입력 SLA 위반이 반복되면 seller `settlement_reliability` 가 내려간다
- 이 값은 공개 평판과 별도로 관리한다
- `dispute_wins` / `dispute_losses` 와 `expertise` 는 상대방 모델과 risk/routing 계층에서 후속 활용 가능하다

## 모듈 간 연결 방식

직접 함수 호출보다 아래 방식이 우선이다.

1. stable type contract
2. idempotent event
3. provider-specific extension field

예:
- negotiation candidate reached
  -> `MUTUALLY_ACCEPTABLE`
- buyer hold
  -> `HELD_BY_BUYER`
- seller reserve with expiry
  -> `RESERVED_PENDING_APPROVAL`
- buyer approved
  -> auto seller: `APPROVED`
  -> manual seller: `AWAITING_SELLER_APPROVAL`
- approved
  -> payment intent created
- order created
  -> shipment created
- shipment delivery exception
  -> dispute candidate created
- dispute resolved with refund
  -> refund requested
