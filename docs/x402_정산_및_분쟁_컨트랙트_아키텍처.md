# x402_정산_및_분쟁_컨트랙트_아키텍처

## 목적

이 문서는 Haggle 이 `Base + USDC + x402` 를 사용할 때,
결제 정산과 분쟁 앵커링을 어떤 스마트 컨트랙트 경계로 가져갈지 정의한다.

핵심 목표는 세 가지다.

- seller 직접 수령
- Haggle fee 분리
- dispute 결과의 검증 가능성 확보

## 전제

- 기본 결제 rail 은 `x402`
- 기본 네트워크는 `Base`
- 기본 자산은 `USDC`
- buyer authorization 은 `human_wallet` 과 `agent_wallet` 를 모두 지원
- confirmation 은 facilitator + chain verification 의 hybrid 모델

## 왜 escrow 를 기본으로 두지 않는가

- 물리 상품 거래에서는 분쟁과 배송 확인이 결제와 길게 얽힌다
- Haggle 이 자금을 오래 보관하는 구조는 운영/법적 부담을 키울 수 있다
- 따라서 MVP와 초기 표준화 단계에서는 `custody/escrow` 가 아니라
  `execution + fee split + dispute anchor` 중심으로 설계한다

즉 초기 방향은:

- 돈은 seller 에게 직접 간다
- Haggle 은 fee wallet 로 수수료를 받는다
- dispute 는 offchain case system 이 처리한다
- 그 결과만 onchain 에 앵커링한다

## 컨트랙트 경계

### 1. HaggleSettlementRouter

역할:

- payment execution request 를 받는다
- seller payout amount 와 Haggle fee amount 를 분리한다
- approval snapshot hash 를 바인딩한다
- optional reservation id 를 바인딩한다
- execution event 를 발생시킨다

필수 입력:

- `order_id`
- `payment_intent_id`
- `buyer_id`
- `seller_id`
- `buyer_authorization_mode`
- `buyer_wallet`
- `seller_wallet`
- `haggle_fee_wallet`
- `gross_amount`
- `seller_amount`
- `haggle_fee_amount`
- `approval_snapshot_hash`

출력:

- `execution_id`
- `router_reference`
- `tx_hash`
- `status`

### 2. HaggleDisputeRegistry

역할:

- dispute case hash 저장
- evidence root hash 저장
- resolution hash 저장
- onchain reference 반환

필수 입력:

- `order_id`
- `dispute_case_id`
- `evidence_root_hash`
- `resolution_hash`

출력:

- `onchain_reference`
- `anchored_at`

## 상태 연결

1. 협상 완료 후 approval snapshot 고정
2. payment intent 생성
3. buyer authorization 완료
4. settlement router 실행
5. 정산 성공 시 payment settled
6. 분쟁이 없으면 종료
7. 분쟁 발생 시 offchain case 진행
8. resolution hash 를 dispute registry 에 앵커링
9. trust ledger 갱신

## hold / reservation 연결

- 일반 `soft hold` 는 router 와 직접 연결되지 않는다
- `reserved` 상태만 `reservation_id` 로 router 에 바인딩될 수 있다
- 즉 가격 보장과 onchain execution 을 섞는 경우에도,
  명시적으로 예약된 상태만 binding 대상이 된다

## human wallet vs agent wallet

### human wallet
- buyer 가 직접 wallet signature 수행
- 일반 사용자 흐름
- wallet UI/지갑 앱 서명이 필요

### agent wallet
- agent 가 programmatic wallet 로 서명
- MCP/UCP 자동 거래 흐름
- 별도 spending policy 와 risk control 필요

컨트랙트/정산 레벨에서는 둘 다 `buyer_authorization_mode` 로 동일하게 기록한다.

## 신뢰 데이터 연결

SettlementRouter 와 DisputeRegistry 는 trust ledger 와 연결된다.

- 승인 후 미결제
- 배송 SLA 미준수
- 분쟁 승리/패배
- 성공 정산

이후 요약값은 `OnchainTrustProfile` 과 `settlement_reliability` 에 반영된다.
