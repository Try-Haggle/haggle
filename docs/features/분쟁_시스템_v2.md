## 2. 3-Tier 구조

### 비용 공식

```
Tier 1:  max(거래액 × 0.5%, $3)    — AI 판사 (단독)
Tier 2:  max(거래액 × 2%,  $12)    — 배심원 재판 (Panel)
Tier 3:  max(거래액 × 5%,  $30)    — 배심원 대재판 (Grand Panel)
```

### 에스컬레이션 배수

| 구간 | 배수 | 의미 |
|------|------|------|
| T1 → T2 | **4x** | AI 결과 불복 시 상당한 비용 증가 |
| T2 → T3 | **2.5x** | 패널 결과 불복 시 추가 비용 |

### 전환점

$600 이하: 최소 비용 적용 ($3 / $12 / $30)
$600 이상: 비율 적용 (0.5% / 2% / 5%)
→ 3개 Tier 전환점이 모두 $600으로 일치

## 3. 법원 비유: 배심원 재판 모델

### 역할 구조

```
실제 법원              →    해글 분쟁
──────────                  ──────────
원고측 변호사           →    구매자측 AI
피고측 변호사           →    판매자측 AI
판사                   →    T1: AI 단독 판결
배심원                 →    T2/T3: 일반 사용자 다수결
전문가 증인            →    Skill (외부 API, 예: skill-legit)
```

### 왜 배심원 모델인가

- 전문 리뷰어 모집은 비현실적 (스케일 불가)
- 일반 사용자도 **AI가 쟁점을 정리해주면** 충분히 판단 가능
- 전문성이 필요한 경우 → Skill로 외부 전문가 API 호출 (증거로 활용)

### 배심원 자격

```
거래 5건 이상 완료 AND Trust Score 50 이상
```

별도 모집, 훈련, 자격증 불필요. 기존 사용자가 곧 배심원.

### AI의 역할

1. 양측 증거 수집 및 정리
2. 구매자측 AI: 구매자에게 유리한 주장 구성
3. 판매자측 AI: 판매자에게 유리한 주장 구성
4. 쟁점 요약: "핵심 질문: X인가 Y인가?"
5. 배심원에게 간결한 선택지 제시

### 배심원 투표 화면 (예시)

```
┌──────────────────────────────────────────┐
│  분쟁 #1234 — iPhone 15 Pro, $800        │
│                                          │
│  📱 구매자 주장 (AI 요약):                 │
│  "배터리 건강도 95%라고 했는데 82%임"       │
│  [사진] [배터리 스크린샷]                   │
│                                          │
│  📱 판매자 주장 (AI 요약):                 │
│  "판매 후 사용으로 인한 자연 감소"           │
│  [리스팅 캡처] [배송 시점 스크린샷]          │
│                                          │
│  ⚖️ 핵심 질문:                            │
│  배터리 건강도 13% 차이가 환불 사유인가?     │
│                                          │
│  [구매자 승] [판매자 승] [부분 환불 __%]    │
└──────────────────────────────────────────┘
```

## 4. 비용 분배

### 고정 비율 원칙

```
Tier 1:  전액 → 플랫폼 (AI 판결, 배심원 없음)
Tier 2/3: 30% → 플랫폼, 70% → 배심원
```

왜 고정 비율인가:
- **투명**: "70%는 배심원, 30%는 플랫폼" — 한 문장으로 설명 가능
- **공정**: 거래액, 투표 결과와 무관하게 항상 동일 비율
- **예측 가능**: 배심원이 사전에 보상 계산 가능
- **모듈화 용이**: 다른 플랫폼은 비율만 변경

### 1인당 보상 계산

```
1인당 = (분쟁비용 × 70%) ÷ 다수파 수
```

| 거래액 | T2 비용 | 배심원 풀 (70%) | 다수파 | **1인당** |
|-------|---------|---------------|-------|----------|
| $100 | $12 | $8.40 | 2 | **$4.20** |
| $500 | $12 | $8.40 | 2 | **$4.20** |
| $1K | $20 | $14.00 | 3 | **$4.67** |
| $3K | $60 | $42.00 | 5 | **$8.40** |
| $5K | $100 | $70.00 | 5 | **$14.00** |
| $10K | $200 | $140.00 | 6 | **$23.33** |
| $25K | $500 | $350.00 | 9 | **$38.89** |
| $50K | $1K | $700.00 | 13 | **$53.85** |
| $100K | $2K | $1,400.00 | 19 | **$73.68** |

### 박빙 투표 규칙

마진 1 (예: 3-2, 4-3) 일 때: 1인분을 플랫폼이 추가 확보.

```
$1K T2, 5명, 3-2 결과:
  풀 $20 × 70% = $14 배심원 풀
  다수파 3명, 박빙(-1) → 2인분 지급
  1인당 = $14 ÷ 3 × 2/3 = $3.11
  플랫폼 = $6 (기본 30%) + $4.67 (박빙 1인분) = $10.67
```

### 다수파만 보상

- 다수 의견 배심원만 보상 수령
- 소수파 = $0
- 고등급 배심원이 소수파 → 가중 DS 감점 (vote_weight 비례)

## 5. 배심원 배정

### 알고리즘: Efraimidis-Spirakis (ES)

```
각 배심원에게 key = random^(1/weight) 생성
→ 상위 N개 선택
→ seed = hash(dispute_id) (결정론적, 감사 가능)
```

성능: 50,000명 풀에서 45명 선택 < 5ms

### 배정 파이프라인

```
1. Pre-filter
   - 슬롯 초과 → 제외
   - 이해충돌 → 제외 (거래 당사자 관련)
   - 배심원 자격 미달 → 제외

2. Idle-first (무조건 놀고 있는 사람 먼저)
   0건 처리 중인 사람만으로 풀 구성
     → 충분하면 (≥ N명) → 여기서 ES 선택 → 끝
     → 부족하면 → 1건 처리 중 추가 → 2건 → ...

3. Weight 기반 ES 선택
   weight = tierWeight × performanceBoost
```

### tierWeight

| 배심원 별 | 가중치 | ⭐ 대비 |
|---------|--------|-------|
| ⭐ (0-30점) | 1.0 | 1.0x |
| ⭐⭐ (31-50점) | 1.2 | 1.2x |
| ⭐⭐⭐ (51-70점) | 1.5 | 1.5x |
| ⭐⭐⭐⭐ (71-85점) | 1.8 | 1.8x |
| ⭐⭐⭐⭐⭐ (86+점) | 2.2 | 2.2x |

최대 격차 2.2배. **별 수가 배정 자격을 제한하지 않음** — 확률만 조절.
⭐ 배심원도 $100K 분쟁에 배정될 수 있음 (확률 낮을 뿐).

### performanceBoost

```
performanceBoost = 1.0
  + 0.2 × (zone_hit_rate - 0.5)       정확할수록 ↑
  + 0.1 × (1 - response_hours / 48)    빠를수록 ↑
  + 0.1 × min(completed_cases / 100, 1) 경험 많을수록 ↑
```

범위: ~0.8 ~ 1.4. 잘하는 사람 = 더 자주 뽑힘 (벌칙이 아닌 보상).

### 슬롯 시스템

전원 동일 한도: **3슬롯**. 등급 무관.

분쟁이 슬롯 비용을 결정:

## 6. 판매자 보증금

### 구조

- Tier 2/3 에스컬레이션 시 **판매자만** 보증금 예치
- 구매자는 별도 보증금 없음 (이미 에스크로가 구매자의 stake)
- 보증금 금액 = 분쟁 비용과 동일

### 미예치 시

- 기한 내 미예치 → 판매자 자동 패소 (Default Judgment)
- 기한: T2 = 48시간, T3 = 72시간

### 결과별 처리

| 결과 | 보증금 |
|------|--------|
| 판매자 패소 | 몰수 → 플랫폼 수익 |
| 판매자 승소 | 전액 환불 |

## 7. 정산 (Settlement)

### 자금 흐름

분쟁 비용 출처:
- **구매자 패소** → 에스크로 대금에서 분쟁 비용 차감
- **판매자 패소** → 판매자 보증금에서 분쟁 비용 차감

분쟁 비용 행선지:
- 배심원 70% (T2/T3) 또는 플랫폼 100% (T1)
- 플랫폼 30% (T2/T3)

### 결과별 정산

**구매자 승:**

| 항목 | 금액 |
|------|------|
| 구매자 | 에스크로 전액 환불 |
| 판매자 | $0 |
| 분쟁 비용 출처 | 판매자 보증금 |
| 배심원 | 분쟁비용 × 70% |
| 플랫폼 | 분쟁비용 × 30% + 보증금 잔액 |

**판매자 승:**

| 항목 | 금액 |
|------|------|
| 구매자 | $0 |
| 판매자 | 에스크로 - 분쟁비용 + 보증금 환불 |
| 분쟁 비용 출처 | 에스크로 (구매자 대금) |
| 배심원 | 분쟁비용 × 70% |
| 플랫폼 | 분쟁비용 × 30% |

**부분 환불:**

| 항목 | 금액 |
|------|------|
| 구매자 | 환불 금액 |
| 판매자 | 에스크로 - 환불 금액 |
| 분쟁 비용 출처 | 판매자 보증금 |
| 배심원 | 분쟁비용 × 70% |
| 플랫폼 | 분쟁비용 × 30% + 보증금 잔액 |

## 8. 품질 관리

### Trust Score 연동

| 이벤트 | Trust Score 영향 |
|--------|----------------|
| 분쟁 패소 | 하락 |
| 분쟁 과다 신청 | 하락 |
| 반복 패소 | 분쟁 제기 제한 가능 |

### 배심원 별 연동

| 이벤트 | DS 영향 |
|--------|--------|
| 다수파 (맞춤) | 점수 상승 |
| 소수파 (틀림) | 점수 하락 × vote_weight |
| 빠른 응답 | performanceBoost 상승 |
| 소액 대충 판단 | 동일 감점 (소액 무시 방지) |

고등급 배심원 소수파 감점 가중:

```
⭐ 틀림: -1 × 0.63 = -0.63
⭐⭐⭐⭐⭐ 틀림: -1 × 2.0 = -2.0
```

→ 높은 등급 = 높은 투표 가중치 = 높은 책임

## 9. 전문가 증인: Skill 연동

### 패턴

전문성이 필요한 분쟁 → 외부 API를 Skill로 연결 → 결과를 증거로 배심원에게 제공

### 기존 구현

`@haggle/skill-legit` — LegitApp 명품 감정 서비스 연동 (125 tests)

### 확장 예시

| Skill | 분야 | 외부 서비스 |
|-------|------|-----------|
| skill-legit | 명품 감정 | LegitApp |
| skill-mechanic (미래) | 차량 점검 | 차량 검사 API |
| skill-art-verify (미래) | 미술품 | 아트 감정 API |

### 비용

Skill 호출 비용 = 별도 (분쟁 비용 풀에서 차감 or 요청자 부담)

## 10. 모듈화 — "분쟁의 Stripe"

### 비전

Stripe가 결제를 표준화했듯, Haggle이 분쟁 해결을 표준화.
다른 플랫폼이 API로 연결하여 즉시 분쟁 시스템 구축.

### DisputeModuleConfig

```typescript
interface DisputeModuleConfig {
  // 비용
  tier1_rate: number;             // default 0.005
  tier2_rate: number;             // default 0.02
  tier3_rate: number;             // default 0.05
  tier1_min_cents: number;        // default 300
  tier2_min_cents: number;        // default 1200
  tier3_min_cents: number;        // default 3000

  // 분배
  platform_share: number;         // default 0.30
  close_vote_penalty: number;     // default 1

  // 배심원
  juror_min_transactions: number;  // default 5
  juror_min_trust_score: number;   // default 50
  reviewer_counts: ReviewerCountConfig;
  max_concurrent_slots: number;    // default 3
  tier_weights: TierWeightConfig;

  // AI
  ai_plaintiff_enabled: boolean;   // default true
  ai_defendant_enabled: boolean;   // default true
  ai_expert_witness: boolean;      // default true

  // 네트워크
  use_shared_pool: boolean;        // true면 해글 공유 배심원 풀
  haggle_network_fee: number;      // 해글 네트워크 사용 수수료
  external_experts: ExpertServiceConfig[];
}
```

### 플랫폼별 설정 예시

**프리랜서 마켓플레이스:**

```typescript
{
  tier2_rate: 0.02,
  tier2_min_cents: 500,     // $5 (소액 거래 많음)
  platform_share: 0.25,     // 배심원에게 75%
  juror_min_transactions: 3, // 낮은 진입 장벽
}
```

**중고차 플랫폼:**

```typescript
{
  tier2_rate: 0.015,         // 고액이라 낮은 비율
  tier2_min_cents: 3000,     // $30
  platform_share: 0.35,      // 전문 인프라 비용
  max_concurrent_slots: 2,   // 복잡한 분쟁 = 집중
}
```

**B2B 도매:**

```typescript
{
  tier2_rate: 0.008,
  tier3_rate: 0.02,
  tier2_min_cents: 5000,     // $50
  platform_share: 0.40,
  max_concurrent_slots: 1,   // 고액 = 1건 집중
}
```

### 수익 모델

| 채널 | 설명 |
|------|------|
| 자체 P2P 거래 수수료 | GMV × 1.5% |
| 자체 분쟁 platform_share | 분쟁비용 × 30% |
| 외부 플랫폼 API 수수료 | 건당 or GMV 비율 |
| 공유 배심원 네트워크 수수료 | 네트워크 사용료 |

### 네트워크 효과

```
플랫폼 A, B, C가 해글 모듈 사용
→ 공유 배심원 풀 확대
→ 배심원 수입 다각화 (여러 플랫폼 분쟁 리뷰)
→ 더 많은 플랫폼 유치
→ 선순환
```
