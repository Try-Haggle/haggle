# 배송_추적_모듈_설계

## 목적

이 문서는 Haggle의 배송 추적 모듈을 독립 서비스로 분리 가능한 수준으로 정의한다.
핵심 기준은 `canonical shipment state`, `shipment input SLA`, `tracking ingestion 분리`다.

## 원칙

- carrier raw status 를 그대로 domain 상태로 사용하지 않는다.
- approval 완료 이후부터 배송 입력 SLA 가 시작된다.
- tracking 수집과 사용자 표시를 분리한다.
- 배송 불이행은 trust ledger 와 dispute candidate 생성으로 이어질 수 있어야 한다.

## 선행 조건

배송 모듈은 아래 이후에 활성화된다.

- buyer approval 완료
- 필요 시 seller approval 완료
- order snapshot 고정
- shipment input due 시각 계산

## 핵심 엔티티

- `Shipment`
- `ShipmentEvent`

관련 공통 엔티티:

- `SellerApprovalPolicy`
- `SettlementApproval`
- `TrustPenaltyRecord`

## canonical 상태

- `LABEL_PENDING`
- `LABEL_CREATED`
- `IN_TRANSIT`
- `OUT_FOR_DELIVERY`
- `DELIVERED`
- `DELIVERY_EXCEPTION`
- `RETURN_IN_TRANSIT`
- `RETURNED`

## shipment input SLA

approval 완료 이후 seller 는 일정 기간 안에 배송 정보를 입력해야 한다.

예:
- `shipment_input_due_days = 2`
- `shipment_input_due_at = approved_at + 2 days`

위반 시:
- `SHIPMENT_INFO_SLA_MISSED` penalty 발행
- seller `settlement_reliability` 하락 가능
- 필요 시 dispute candidate 생성

## tracking ingestion 분리

### shipping-core
- canonical shipment status
- 상태 전이
- shipment event contract

### delivery ingest layer
- carrier webhook 수신
- raw event 정규화
- canonical event 발행

## trust 연동

배송 모듈은 seller 신뢰도에 직접 영향을 준다.

- SLA 반복 위반
- 배송 미입력
- delivery exception 반복

이 값은 결제 모듈의 settlement reliability 집계와 합쳐져
seller 의 장기 trust profile 에 반영될 수 있다.

## dispute 연동

아래 상황은 dispute 후보가 될 수 있다.

- `DELIVERY_EXCEPTION`
- `LABEL_PENDING` 장기 지속
- seller 승인 후 배송 미입력
- delivered claim 불일치
