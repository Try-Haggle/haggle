# 결제_모듈_설계

## 목적

이 문서는 Haggle의 결제 모듈을 독립 서비스로 분리 가능한 수준으로 정의한다.
핵심 기준은 `x402 우선`, `Stripe fallback`, `approval 이후 실행`, `trust ledger 연동`이다.

## 원칙

- 결제는 협상 단계의 선택 UI가 아니라, 이미 확정된 rail의 실행 단계다.
- 기본 rail은 `x402` 이고 `Stripe` 는 fallback 또는 merchant 정책 옵션이다.
- provider adapter 와 canonical payment state 를 분리한다.
- 승인 후 미결제는 trust penalty 로 기록되어야 한다.

## 선행 조건

결제 시작 전에 아래가 고정되어야 한다.

- `MUTUALLY_ACCEPTABLE`
- buyer explicit approval
- seller approval policy
- hold / reservation 상태 정리
- 최종 settlement snapshot

즉 실제 흐름은 아래와 같다.

1. 협상 후보안 도달
2. `MUTUALLY_ACCEPTABLE`
3. buyer 승인
4. seller policy 확인
5. auto seller면 즉시 `APPROVED`
6. manual seller면 `AWAITING_SELLER_APPROVAL`
7. approval 완료
8. payment intent 생성
9. rail 실행

## rail 정책

### x402
- 기본 rail
- Base + USDC 를 기본 전제로 둔다
- onchain settlement 또는 x402-compatible execution layer 우선
- 장기적으로 trust profile 과 자연스럽게 연결 가능
- buyer authorization 은 `human_wallet` 과 `agent_wallet` 를 모두 지원한다

### Stripe
- fallback
- merchant preference 또는 buyer environment 제약 시 사용
- provider reference 를 canonical settlement 와 분리 저장

## provider 분리

### core
- `selected_rail` 실행
- canonical state transition
- idempotent event 발행

### adapter
- `ai.haggle.x402`
- `ai.haggle.stripe`

## x402 authorization mode

### human wallet
- 일반 구매자가 자기 지갑으로 직접 서명
- 웹/앱 기본 UX
- self-custody 가 명확하다

### agent wallet
- 에이전트가 programmatic wallet 로 서명
- MCP/UCP 기반 자동 구매에 적합하다
- 별도 policy / spending rule 이 필요하다

둘 다 지원하되, canonical payment intent 에는
`buyer_authorization_mode` 가 반드시 기록되어야 한다.

## x402 settlement 방향

Haggle 은 초기 구현에서 `x402 exact` 를 사용하되,
장기적으로는 `Settlement Router + Dispute Registry` 구조로 확장한다.

핵심은 아래와 같다.

- seller 는 자신의 USDC wallet 으로 직접 payout 을 받는다
- Haggle 은 별도 fee wallet 로 수수료를 받는다
- 분쟁 시스템은 escrow 보관보다 `onchain anchor + offchain resolution` 을 우선한다
- 즉 자금을 오래 custody 하는 구조보다, 정산 이벤트와 분쟁 결과를 검증 가능하게 남기는 구조를 택한다

이 방향은 `x402-exec` 같은 future execution spec 과도 자연스럽게 이어질 수 있지만,
현재 core 구현은 RFC 상태의 기능에 의존하지 않는다.

## 보안 전제조건

결제는 아래 전제조건이 충족될 때만 실행되어야 한다.

- settlement approval 상태가 `APPROVED`
- manual seller 인 경우 `seller_approved_at` 존재
- buyer approval 존재
- 요청 actor 가 실제 buyer 또는 seller participant
- `selected_rail`, `amount`, `currency` 가 settlement snapshot 과 일치

즉 payment intent 는 임의 생성이 아니라,
`승인 완료된 settlement snapshot` 에서만 파생되어야 한다.

## webhook 보안

provider webhook 는 raw payload 를 받기 전에 서명 검증이 필요하다.

- `x402` : provider signature header 검증
- `Stripe` : `stripe-signature` 검증

검증 실패 시 canonical payment state 를 변경하면 안 된다.

초기 mock 단계에서도 최소한:
- 서명 헤더 존재 확인
- provider 구분
- received_at 기록

은 유지해야 한다.

## smart contract 경계

실제 자금 실행과 분쟁 앵커링은 아래 두 경계로 분리한다.

- `HaggleSettlementRouter`
  - seller payout
  - Haggle fee split
  - approval snapshot hash binding
  - optional reservation binding
- `HaggleDisputeRegistry`
  - dispute case hash
  - evidence root hash
  - resolution hash
  - onchain reference 반환

초기 구현에서는 contract interface 만 고정하고,
business dispute resolution 은 offchain case system 이 수행한다.

초기 API 구현은 아래 전략을 따른다.

- x402 mode 가 `mock` 이면 mock adapter 사용
- x402 mode 가 `real` 이면 `RealX402Adapter` 사용
- 현재 router/dispute registry 는 scaffold contract 로 연결
- 실제 chain ABI 와 verifier 는 다음 단계에서 교체

## x402 buyer signing flow

실제 x402 실행 경로는 아래처럼 동작한다.

1. `POST /payments/prepare`
2. `GET /payments/:id/x402/requirements`
3. buyer 가 로컬 wallet 또는 agent wallet 로 서명
4. `POST /payments/:id/x402/submit-signature`
5. server 가 facilitator `/settle` 를 호출
6. 성공 시 canonical payment state 를 `SETTLED` 로 전이

`verify_only` 는 디버깅/사전점검 용도로만 허용하고,
정식 결제 완료 판정은 `/settle` 성공을 기준으로 한다.

## 승인 후 미결제 패널티

아래 상황은 trust ledger 에 기록되어야 한다.

- `BUYER_APPROVED_BUT_NOT_PAID`
- `SELLER_APPROVED_BUT_NOT_FULFILLED`

이 기록은 이후 `settlement_reliability` 로 집계되고,
engine 의 `V_r` 에 선택적으로 반영된다.

## 온체인 신뢰 연동

결제 모듈은 장기적으로 아래와 연결되어야 한다.

- `OnchainTrustProfile`
- `wallet_address`
- `onchain_reference`
- `settlement_reliability`
- `approval_defaults`

즉 결제는 돈만 처리하는 모듈이 아니라,
`합의 이후 실제 이행`을 신뢰 데이터로 남기는 핵심 소스다.
