## 1. 골리앗 vs 다윗: 마켓플레이스 AI 업데이트 분석

### 1.1 골리앗들의 최근 AI 업데이트

**Meta (Facebook Marketplace)**
- 사진 한 장으로 AI 자동 리스팅 (제목, 설명, 가격 제안)
- AI 자동 응답 (구매자 질문에 자동 대응)
- 간편 배송 라벨 생성
- 톤 조절 기능 (Recommended, Friendly, Professional, Concise)
- 차량 AI 인사이트 (엔진, 안전등급, 가격 비교)
- 구매자-판매자 채팅에 AI 추천 질문
- eBay/Poshmark 파트너 리스팅 통합
- 미국+캐나다 일일 리스팅 350만 건

**eBay**
- "Magical Listing" AI 업그레이드 (사진 기반 자동 리스팅)
- Depop $1.2B 인수 (2026 Q2 완료 예정)
- AI 벤처 투자 확대 (Vardera 등)

**기타**
- Mercari, Poshmark, Depop: 유사 AI 리스팅 기능 이미 제공 중

### 1.2 골리앗들의 한계 (Haggle의 기회)

| 골리앗이 잘하는 것 | 골리앗이 절대 못하는 것 |
|-------------------|---------------------|
| 사진→리스팅 자동화 | **협상 자체가 없음** — 가격은 판매자가 정하고 구매자가 수락하는 구조 |
| 소셜 그래프 기반 신뢰 | **수수료 구조를 못 바꿈** — eBay 13%, Meta 배송 수수료는 수익 모델 |
| 압도적 유저 볼륨 | **결제 혁신 없음** — 기존 카드/PayPal 레일에 종속 |

### 1.3 다윗의 전투 전략

1. **싸움의 판을 바꿔라** — "리스팅 편의성" 경쟁 안 함. 가격 결정 과정 자체를 혁신
2. **수수료 격차를 무기로** — $1,000 거래 시 eBay $130 vs Haggle $15 = $115 차이
3. **프로토콜 레이어에서 싸워라** — 플랫폼(벽 안에 가둠) vs 프로토콜(어디서든 연결)
4. **골리앗의 AI 업데이트를 활용** — 플랫폼이 AI-ready가 될수록 Haggle이 꽂힐 소켓 증가

## 2. 플랫폼 → 네트워크 → 프로토콜 전환 전략

### Phase 1: 플랫폼 (Month 0–6)

- 닭과 달걀 문제 해결 — ChatGPT Apps에서 직접 거래 발생
- HNP를 내부적으로 운영, 외부 비공개
- 협상 이력, 가격 패턴, ZOPA 데이터 축적
- **목적: 프로토콜을 증명할 실거래 데이터 생성**
- **함정: 여기서 멈추면 그냥 또 하나의 마켓플레이스**

### Phase 2: 네트워크 (Month 6–18)

- MCP 서버 공개 — 어떤 AI 에이전트든 Haggle에 연결 가능
- Skills Marketplace 오픈 — 서드파티 인증, 배송, 감정 Skill
- 크로스플랫폼 거래 시작
- **핵심 지표: Haggle 자체 리스팅 비중 100% → 50% 이하로 떨어져야 성공**
- **비유: Visa 자체 카드 → Visa 네트워크에 다른 은행이 붙는 것**

### Phase 3: 프로토콜 (Month 18–36)

- HNP가 Haggle 없이도 돌아가는 상태
- 전환 조건: 충분한 거래 데이터 + 복수의 독립 구현체 + 거버넌스 분리
- HNP 스펙 오픈소스 공개 (RFC 스타일)
- Haggle는 "best implementation"으로 수익화 (Red Hat 모델)
- **역설: Haggle 회사가 사라져도 HNP가 돌아가면 → 그때 Haggle 가치가 가장 높음**

### 수익 모델 전환

| Phase | 수익원 | 비유 |
|-------|--------|------|
| 플랫폼 | 거래 수수료 1.5% | eBay |
| 네트워크 | 수수료 + API 사용료 + Skills 수수료 | Stripe |
| 프로토콜 | 프리미엄 엔진 + 데이터 인텔리전스 + B2B 라이센스 | Visa / Red Hat |

### 각 단계의 "졸업 시험"

- **플랫폼 → 네트워크:** "외부 에이전트가 Haggle 없이는 P2P 협상을 못 한다"는 의존성
- **네트워크 → 프로토콜:** "Haggle 회사가 사라져도 HNP로 거래가 계속 돌아간다"

### 가장 위험한 실수

**너무 빨리 프로토콜화하는 것.** TCP/IP도 ARPANET 위에서 먼저 돌아간 뒤 표준이 됨. 비전은 프로토콜, 실행은 플랫폼 — 이 긴장을 유지하는 게 핵심.

## 3. AI 협상 분야에 사업이 없는 이유

### 구조적 이유 6가지

1. **기술 타이밍이 이제야 맞음** — 자연어 이해 + 전략적 추론 + 실시간 대화가 동시에 가능해진 게 최근. MCP 같은 에이전트 간 통신 표준 성숙이 2024-2025년
2. **기존 플랫폼에겐 자살 행위** — eBay AI 협상 도입 시 평균 거래가 하락 → 수수료 매출 감소. "Buy It Now" 90%인 구조를 뒤집는 것은 10년간의 UX 방향 역행
3. **"협상"이 사업 카테고리로 인식 안 됨** — VC에게 카테고리 자체가 없음 → 오히려 기회 (카테고리 정의자 = 1등)
4. **양면 문제 극도로 어려움** — 협상은 상대 없으면 가치 0. 콜드스타트 문제가 다른 AI 서비스보다 심함
5. **신뢰 문제** — AI에 금전적 의사결정 위임은 챗봇과 차원이 다른 신뢰 요구 (빠르게 변화 중)
6. **결제 인프라 부재** — 동적 가격 협상 후 조건부 에스크로는 기존 결제가 지원 안 함. x402 + USDC가 최근에야 해결

### 결론

기술(LLM+MCP+x402), 시장 인식(AI 에이전트 경제), 규제(GENIUS Act)가 **동시에 맞아떨어지는 윈도우가 지금 막 열림.** 6개월 전엔 너무 일렀고, 2년 후엔 누군가 이미 자리잡았을 것.

## 7. LLM 비용 하락과 Model-Agnostic 전략

### LLM 가격 하락 트렌드

GPT-4급 성능이 2년 전 대비 이미 50배 이상 하락. "만약"이 아니라 "언제"의 문제. **그리고 이 변화를 시스템이 자동으로 흡수해야 한다.**

### 3계층 추상화: 변화를 흡수하는 구조

```
┌───────────────────────────────────────────┐
│  Layer A: Protocol Contract (불변)         │  ← 모델이 뭐든 이 스키마를 준수
│  NegotiationMove / RefereeCoaching 스키마  │
│  검증 규칙 7개, 수렴 보장 조건              │
├───────────────────────────────────────────┤
│  Layer B: Skill Interface (안정적)         │  ← Skill이 뭐든 이 인터페이스 구현
│  assessContext / generateOffer /           │
│  evaluateOffer                            │
├───────────────────────────────────────────┤
│  Layer C: Model Adapter (교체 가능)        │  ← 6개월마다 교체해도 위 2개 불변
│  프롬프트 템플릿, 토큰 전략, 출력 파싱      │
└───────────────────────────────────────────┘
```

**새 모델 도입 = Layer C에 어댑터 파일 1개 추가.** Protocol, Referee, 테스트 — 전부 그대로.

### LLM 발전이 주는 기회도 자동 흡수

모델이 똑똑해지면 코칭을 줄이고 자유도를 높인다:

| 모델 능력 | 코칭 레벨 | LLM 자유도 |
|-----------|----------|-----------|
| BASIC (현재 Grok Fast) | DETAILED — 구체적 권장가, 단계별 가이드 | 낮음 |
| ADVANCED (미래 GPT-5) | LIGHT — 힌트만, 자유롭게 전략 수립 | 높음 |

심판의 **검증 기준은 동일** — 코칭 상세도만 달라짐. 모델이 발전할수록 심판이 할 일은 줄고, LLM의 창의성은 증가.

### 전환 방향

**Phase 1: 엔진 = 심판 (MVP)**
- LLM이 협상을 실행, 엔진은 Coach → Validate → Guide
- Faratin 수학이 코칭으로 제공됨 — 차단이 아닌 유도
- DefaultEngineSkill이 LLM 장애 시 수학적 fallback 제공

**Phase 2: 데이터 > 로직**
- 엔진 로직은 commodity, 협상 데이터는 아님
- 카테고리별 ZOPA 분포, 성공/실패 전략 패턴, 시간대별 양보 곡선
- 누구나 협상 봇을 만들 수 있어도 데이터는 Haggle만 보유

**Phase 3: 엔진 오픈소스 → 프로토콜 가속**
- Haggle 엔진 자체를 오픈소스로 공개
- HNP 호환 에이전트를 누구나 제작 가능
- Haggle = 프로토콜 거버넌스 + 프리미엄 데이터 + 정산 인프라로 수익화

### 수익 모델 전환

| LLM 비용 | Haggle 수익원 | 핵심 자산 |
|----------|-------------|----------|
| 비쌈 | 엔진 효율 + 거래 수수료 | 심판 엔진 |
| 중간 | 데이터 인텔리전스 + 수수료 | 협상 데이터 + Skill 생태계 |
| 매우 쌈 | 프로토콜 수수료 + 데이터 라이센스 + 인증/신뢰 | HNP 표준 + 데이터 + Trust |

### 핵심 원칙

**"이 엔진이 영원하다"가 아니라 "이 엔진이 데이터를 모아주는 도구다"로 설계.** 모든 협상의 모든 라운드를 구조화된 데이터로 저장하는 게 엔진 로직보다 장기적으로 훨씬 더 가치있음. **그리고 LLM이 발전해도 Protocol Contract는 불변 — 이것이 프로토콜의 의미.**

## 8. Living Context: LLM이 변화를 "느끼는" 아키텍처

### 핵심 문제

LLM은 영속적 메모리가 없다. 매 호출마다 context window에 들어있는 것만 안다. **"LLM이 변화를 느낀다" = 시스템이 LLM의 context를 동적으로 조립한다.**

### Context Assembly Pipeline (6개 레이어)

```
┌───────────────────────────────────────────────────┐
│  LLM Context Window (매 라운드 새로 조립)           │
│                                                   │
│  L0: Protocol Rules (불변)                         │
│      NegotiationMove 스키마, 수렴 규칙, 출력 형식   │
│                                                   │
│  L1: Model Instructions (모델 교체 시 변경)        │
│      모델별 최적화된 지시사항                       │
│                                                   │
│  L2: Skill Context (세션 시작 시 로드)              │
│      활성 Skill의 전략, 전술, 도메인 지식, 제약     │
│      → Skill이 바뀌면 LLM의 "성격"이 바뀜          │
│                                                   │
│  L3: Referee Coaching (매 라운드 갱신)              │
│      Faratin 권장가, 상대 패턴, 수렴 속도, 힌트     │
│      → 심판의 수학적 코칭                          │
│                                                   │
│  L4: Session History (누적)                        │
│      이전 라운드 가격 이력, 발언 요약, 합의 이력    │
│      → 대화 맥락 유지                              │
│                                                   │
│  L5: Skill Signals (이벤트 기반)                    │
│      실시간 시세 변동, 경쟁 세션 정보               │
│      → 외부 변화 실시간 반영                        │
└───────────────────────────────────────────────────┘
```

### 변화 인식 매트릭스

| 변화 유형 | 어떤 레이어가 변하나 | LLM이 느끼는 방식 |
|-----------|---------------------|-----------------|
| 모델 교체 (Grok → GPT-5) | L1 | 지시 스타일 변경, 자유도 변경 |
| Skill 교체 (Default → SneakerHawk) | L2 | 전략, 전술, 도메인 지식 변경 |
| Skill 업데이트 (v1 → v2) | L2 | changelog로 새 기능 인식 |
| 라운드 진행 | L3 + L4 | 수학적 코칭 갱신 + 이력 누적 |
| 외부 시세 변동 | L5 | "StockX 가격 $20 하락" 신호 |
| 코칭 레벨 변경 | L3 | 모델 능력에 따라 상세/간략 |

### Skill이 LLM에 "연속적으로" 인식되는 메커니즘

서드파티가 SneakerHawkSkill v2를 만들면:

```
SkillManifest.llm_context = {
  system_instructions: "스니커즈 리셀 전문 협상...",
  tactics: ["ANCHORING_LOW", "BUNDLE_PRESSURE", ...],
  domain_knowledge: ["Jordan 1 프리미엄 20-30%", ...],
  constraints: ["DS 확인 없이 ACCEPT 금지", ...],
  changelog: ["v2: BUNDLE_PRESSURE 전술 추가", ...],
}
```

이 `llm_context`가 L2에 직접 주입 → LLM이 "스니커즈 전문가"처럼 행동.
Skill 개발자가 `llm_context`를 잘 작성하면 = LLM이 그 Skill을 정확히 이해.

### 그래도 안 되는 것 (심판이 필요한 이유)

LLM이 아무리 발전해도 **대체 불가능한 3가지**:

1. **수학적 결정론** — 온도 0이어도 LLM은 100% 동일 답 보장 불가. 분쟁 시 재현 가능성 필수
2. **검증 가능한 상태 전이** — `OFFER → COUNTER → ACCEPT → SETTLE` 각 전이의 프로토콜 유효성 검증. LLM은 "이해"하지만 "보장"하지 않음
3. **양쪽 공정성 증명** — Haggle이 양쪽 에이전트 운영. 편들기 안 했다는 증명에 감사 가능한 규칙 기반 시스템 필수

### 심판의 확장된 역할: 차단이 아닌 유도

```
기존 설계: LLM 생성 → 심판 검증 → ✅ 통과 / ❌ 차단
새 설계:   심판 코칭 → LLM 생성 → 심판 검증 → ✅ 통과 / 🔄 교정+재시도
```

| 상황 | 기존 (차단) | 새 설계 (유도) |
|------|-----------|--------------|
| 가격 범위 초과 | "거부됨" | "거부. Faratin 권장가 $680으로 수정 권장" (HARD) |
| 양보 정체 4라운드 | "경고" | "교정: $20+ 양보 또는 비가격 조건 추가 권장" + 재시도 (SOFT) |
| 편향 감지 | "경고" | "한쪽만 양보 요구 중. 중립적 제안으로 수정 권장" + 재시도 (SOFT) |

**SOFT violation → 교정 가이드 + 1회 재시도 → 재실패 시 DefaultEngineSkill fallback**

**엔진 = "협상의 코치+심판". LLM이 아무리 발전해도 심판은 대체 불가. 그리고 심판은 차단만 하지 않는다 — 코칭하고, 유도하고, 교정한다.**

## 9. Skills Marketplace 폭발 시나리오

### 엔진이 심판이 되면 전략이 상품이 된다

```
현재:  Haggle가 전략도 만들고 실행도 함 (폐쇄)
미래:  Haggle는 심판만, 전략은 누구나 만들 수 있음 (개방)
```

### 전략 Skill 예시

```yaml
SneakerHawkStrategy:
  creator: "@reseller_mike"
  category: sneakers
  description: "한정판 스니커즈 구매 특화. 리셀 마진 25%+ 확보율 82%"
  tactics: [ANCHORING_LOW, SCARCITY_COUNTER, BUNDLE_PRESSURE]
  price: 50 HGC
  rating: 4.8 (1,247 uses)
  win_rate: 78.3%

MacBookFlipperPro:
  creator: "@tech_deals_jane"
  category: electronics
  description: "중고 맥북 프로 구매. 감가상각 데이터 기반 최적가 산출"
  tactics: [CONDITION_LEVERAGE, COMPARABLE_ANCHOR, PATIENCE_PLAY]
  price: 30 HGC
  rating: 4.6 (892 uses)
  win_rate: 71.2%
```

### Skill 주입 플로우

```
1. 사용자 "맥북 프로 사고 싶어" 입력
2. Marketplace에서 카테고리 상위 전략 Skill 추천
3. 사용자가 Skill 선택 (HGC 지불)
4. Skill의 전술 세트 + 파라미터 → LLM 프롬프트에 주입
5. LLM이 그 전략으로 협상 실행
6. 엔진(심판)이 모든 제안을 검증
7. 결과 데이터가 Skill의 win_rate에 반영
```

### 활성화 메커니즘

1. **데이터 피드백 루프** — 사용량 ↑ → 성과 데이터 정확 ↑ → 좋은 Skill 부각 → 자기 강화
2. **크리에이터 인센티브** — 파워 셀러/리셀러/전문가가 노하우를 Skill로 판매. 새로운 직업 탄생
3. **카테고리 세분화 크라우드소싱** — 스니커즈 전문가가 스니커즈 전략, 차량 딜러가 차량 전략 제작
4. **LLM 모델별 최적화 경쟁** — 같은 전략도 GPT/Claude/Grok 최적화 버전 → 변종 증가 → 볼륨 증가

### 전략 진화 (EvoEngine 커뮤니티 확장)

```
SneakerHawk v1 (구매) → win_rate 78%
    ↓ 이기는 판매 전략 등장
SneakerShield v1 (판매) → win_rate 72%
    ↓ 학습하여 업데이트
SneakerHawk v2 → 끝없는 전략 진화
```

### HGC 경제 순환

```
거래 → 에그 드랍 → 버디 → HGC → Skill 구매 → 더 좋은 거래 → 순환
```

### 수익 레이어

| 레이어 | 수익원 | 마진 |
|--------|--------|------|
| 프로토콜 (심판) | 거래 수수료 1.5% | 99%+ |
| Skills Marketplace | 판매 수수료 20% | 100% |
| 프리미엄 데이터 | Skill 성과 분석, 카테고리 인텔리전스 | 95%+ |
| HGC 생태계 | Skill 구매 = HGC 소비 = 코인 수요 | 간접 수익 |

## 11. 최종 권장 방향: 4원칙 기반 심판 아키텍처

### 4원칙 프로토콜

모든 설계 결정의 토대. LLM이 어떤 모델이든, Skill이 누가 만들었든, 이 원칙 위에서 합의가 가능하다.

| 원칙 | 의미 | 구현 |
|------|------|------|
| **투명성** | 모든 과정이 기록되고 열람 가능 | 감사 로그 (coaching + 응답 + 검증 + 교정) |
| **합리성** | 모든 제안이 수학적으로 설명 가능 | Faratin 양보 곡선 기반 코칭 |
| **중립성** | buyer/seller에게 동일 규칙 적용 | 동일 코칭 로직 + 편향 감지 |
| **표준성** | 어떤 LLM/Skill이든 참여 가능 | NegotiationMove / RefereeCoaching 스키마 |

### 5계층 프로토콜 스택

```
Layer 5: Presentation  — 사용자 인터페이스 (웹, MCP, API)
Layer 4: Skill         — 협상 전략 실행 (Default / LLM / 3rd-Party)
Layer 3: Referee       — 검증 + 코칭 + 유도 (engine-core 수학)
Layer 2: Protocol      — HNP 상태 전이, 메시지 형식, 세션 관리
Layer 1: Settlement    — 스마트 컨트랙트, 에스크로, USDC 정산
```

각 계층은 독립 교체 가능 — Skill을 바꿔도 Referee 그대로, 모델을 바꿔도 Protocol 그대로.

### 심판 = Coach + Referee (차단 + 유도)

```
① COACH:    engine-core 수학 → RefereeCoaching 생성
② PLAY:     Skill이 coaching 참고하여 자유롭게 응답 생성
③ VALIDATE: 심판 검증 (HARD → 차단, SOFT → 교정+재시도)
④ RECORD:   coaching + 응답 + 검증 전체 감사 로그 기록
```

**심판은 차단만 하지 않는다.** 매 라운드 수학적 코칭을 제공하고, SOFT violation 시 교정 가이드와 재시도 기회를 준다. 재시도 실패 시 engine-core 수학(DefaultEngineSkill)이 fallback.

### 7개 검증 규칙

| # | 규칙 | 심각도 |
|---|------|--------|
| V1 | 가격이 p_limit 범위 초과 | HARD (차단) |
| V2 | 프로토콜 상태 전이 위반 | HARD (차단) |
| V3 | 타임아웃/데드라인 초과 | HARD (차단) |
| V4 | 양보 방향 역전 | SOFT (교정+재시도) |
| V5 | 수렴 정체 (4라운드+) | SOFT (교정+재시도) |
| V6 | 편향 감지 (한쪽만 양보) | SOFT (교정+재시도) |
| V7 | 과도한 양보 (Faratin 2x 초과) | SOFT (교정+재시도) |

### Living Context: LLM이 변화를 느끼는 방법

LLM은 stateless지만, **6개 레이어로 동적 조립되는 context**를 통해 변화를 "느낀다":

- **L0 Protocol Rules** (불변) — 모델이 뭐든 동일한 규칙
- **L1 Model Instructions** (모델별) — 모델 교체 시 이것만 변경
- **L2 Skill Context** (세션별) — Skill의 전략, 전술, 도메인 지식 주입
- **L3 Referee Coaching** (라운드별) — 매 라운드 수학적 코칭 갱신
- **L4 Session History** (누적) — 대화 맥락 유지
- **L5 Skill Signals** (이벤트) — 외부 시세 변동, 경쟁 정보 실시간 반영

**Skill 개발자가 `SkillManifest.llm_context`를 잘 작성하면 → L2가 바뀌면 → LLM이 그 Skill을 "느낌".**
**새 모델 어댑터 1개 추가하면 → L1이 바뀌면 → LLM이 새 능력을 "느낌".**

### 3-Tier 아키텍처

| Tier | 방식 | Haggle LLM 비용 | 대상 |
|------|------|-----------------|------|
| 1 | MCP Agent (사용자 자체 AI) | **$0** | 파워 유저, AI 에이전트 |
| 2 | Built-in Agent (Haggle 내장) | ~$0.005/세션 | 일반 사용자 |
| 3 | Skills Marketplace | Skill 수수료 | 전문 전략 (Phase 2+) |

### 리스크 완화

- DefaultEngineSkill이 LLM 장애 시 수학적 fallback 보장
- 최소 2개 모델 지원 (Grok + GPT/Claude fallback)
- HARD violation은 LLM 출력을 무조건 차단
- 간단한 협상은 DefaultEngineSkill만으로 LLM 없이 가동

### 한 줄 요약

> **엔진은 심판으로, LLM은 선수로, Skill은 코치로.**
> 심판은 차단만 하지 않는다 — **코칭하고, 유도하고, 교정한다.**
> LLM은 변화를 **"느낀다"** — 6개 레이어 Living Context를 통해.
> 모델이 발전하면 코칭을 줄이고, Skill이 진화하면 인식이 확장된다.
> **프로토콜(스키마+규칙)은 불변, 나머지는 전부 교체 가능.**
> Haggle의 진짜 moat는 엔진 로직이 아니라 **프로토콜 + 판례 DB + Skill 생태계.**

## 부록: 핵심 인사이트 요약

### 시장 타이밍
- x402 Foundation 출범 (1주일 전), Visa/Mastercard/Stripe 모두 AI 결제 진입
- 결제 인프라는 골리앗이 깔아주고, Haggle은 그 위에서 협상만 하면 됨
- P2P AI-to-AI 가격 협상 영역은 **완전한 공백**

### 경쟁 포지션
- B2B 조달 협상 (Pactum 등)은 존재하나 P2P 소비자 영역은 없음
- Visa 리서치: 기업 53%가 AI-to-AI 협상 허용 의향 → 시장 교육 진행 중
- 골리앗(eBay/Meta)의 AI 업데이트는 "리스팅 생성" 혁신, "거래 성사 과정" 혁신 아님

### 아키텍처 방향
- MVP부터 LLM 실행 + 엔진 검증 구조로 시작
- 바이너리 엔진 개발에 시간 투자하지 말 것
- 모든 협상 데이터를 구조화 저장 — 이것이 장기 자산
- Skills Marketplace를 자연스럽게 열 수 있는 구조 확보

### 장기 비전
- 플랫폼 → 네트워크 → 프로토콜 순차 전환
- "협상 서비스 회사" → "협상 전략의 앱스토어"
- Apple이 앱을 안 만들어도 앱스토어로 돈 버는 것과 같은 구조
