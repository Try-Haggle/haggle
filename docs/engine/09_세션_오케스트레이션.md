## 1. Master Strategy

### 1.1 개념

**하나의 상품에 하나의 전략.** Master Strategy는 사용자가 특정 상품을 구매(또는 판매)하기 위해 설정하는 협상 전략의 단일 원본이다. LLM 또는 통계 기반으로 생성되며, 동일한 전략이 여러 세션에 공유된다. 세션마다 상대방 정보(r-score, 가격, 거래 이력)가 다르므로, 같은 전략이라도 세션별로 다른 NegotiationContext가 생성된다.

```
┌─────────────────────────────────────────────────────────┐
│                   Master Strategy                        │
│              (LLM 또는 Stats로 생성)                      │
│                                                          │
│  weights, p_target, p_limit, α, β, U_threshold,         │
│  U_aspiration, n_threshold, persona, agent_stats,        │
│  anchor_ratio, market_utilization, r_score_minimum,      │
│  i_completeness_minimum, late_round_aggression_modifier, │
│  kappa, lambda, use_utility_space                        │
└──────────┬──────────────┬──────────────┬─────────────────┘
           │              │              │
           ▼              ▼              ▼
    ┌─────────────┐┌─────────────┐┌─────────────┐
    │  Session A   ││  Session B   ││  Session C   │
    │             ││             ││             │
    │ seller r=4.8 ││ seller r=3.2 ││ seller r=4.5 │
    │ price=$120   ││ price=$135   ││ price=$115   │
    │ history=3    ││ history=0    ││ history=1    │
    └─────────────┘└─────────────┘└─────────────┘
    같은 전략,       같은 전략,       같은 전략,
    다른 데이터      다른 데이터      다른 데이터
```

### 1.2 MasterStrategy Protobuf

```protobuf
message MasterStrategy {
  string id = 1;
  string user_id = 2;
  UtilityWeights weights = 3;
  float p_target = 4;
  float p_limit = 5;
  float alpha = 6;
  float beta = 7;
  float t_deadline = 8;
  float v_t_floor = 9;       // 시간 효용 바닥값. "천천히" = 0.8, 기본 = 0.0
  int32 n_threshold = 10;
  string persona = 11;
  float u_threshold = 12;
  float u_aspiration = 13;
  int32 llm_calls_used = 14;
  google.protobuf.Timestamp created_at = 15;
  google.protobuf.Timestamp expires_at = 16;
  // v1.0.1 NEW
  AgentStats agent_stats = 17;
  StatsPreset stats_preset = 18;
  float anchor_ratio = 19;
  float market_utilization = 20;
  float r_score_minimum = 21;
  float i_completeness_minimum = 22;
  float late_round_aggression_modifier = 23;
  float gamma = 24;
  float cross_pressure_sensitivity = 25;
  // v1.0.2 NEW
  float kappa = 26;
  float lambda = 27;
  bool use_utility_space = 28;
}
```

**주요 필드 설명:**

| 필드 | 설명 |
|------|------|
| `weights` | 4차원 효용 함수 가중치 (w_p, w_t, w_r, w_n) |
| `p_target` | 목표 가격 (이상적 구매가) |
| `p_limit` | 한계 가격 (이 가격 이상이면 거래 불가) |
| `alpha` | 시간 압박 곡선 형태 파라미터 |
| `beta` | Faratin 양보 곡선 형태 파라미터 |
| `v_t_floor` | 시간 효용 바닥값. "천천히" 전략이면 0.8, 기본 0.0 |
| `u_threshold` | 수락 임계값 (U_total ≥ 이 값이면 NEAR_DEAL) |
| `u_aspiration` | 목표 효용값 (이 이상이면 즉시 ACCEPT 권고) |
| `agent_stats` | 에이전트 스탯 (Aggression, Patience, Risk 등) |
| `kappa` | AC_next 수락 조건 민감도 |
| `lambda` | 효용 공간 역제안 감쇠 계수 |
| `use_utility_space` | true이면 효용 공간 기반 역제안, false이면 기존 Faratin 곡선 |

### 1.3 전략이 변경되는 경우

Master Strategy는 기본적으로 불변이지만, 아래 상황에서는 갱신된다:

| 상황 | 변경 방식 | LLM 필요? |
|------|-----------|-----------|
| 시장가 소폭 변동 (<10%) | 규칙 기반 조정 (`p_target`, `p_limit` 비례 이동) | 아니오 |
| 시장가 대폭 변동 (>10%) | Strategy Skill 재계산 (전체 파라미터 재생성) | 가능 |
| 사용자 수동 변경 | 직접 반영 (UI에서 파라미터 수정) | 아니오 |
| 세션 교착 (STALLED, 규칙 실패) | LLM 재전략화 (β, persona 등 변경) | 예 |
| 페르소나 전환 | 프리셋 교체 (stats_preset 기반 파라미터 일괄 변경) | 아니오 |
| 번들/조건부 제안 수신 | 새 조건 분석 (weights, p_target 재산정) | 예 |
| 스탯 변경 (v1.0.1) | Stats → 파라미터 재변환 (AgentStats → α, β 등 매핑) | 아니오 |

> **원칙:** LLM 호출은 규칙 기반으로 처리할 수 없는 경우에만 발생한다. Hot Path 비율을 극대화한다.

## 2. Session Orchestrator

### 2.1 책임

Session Orchestrator는 사용자의 모든 활성 세션을 관리하는 상위 계층이다.

| 책임 | 설명 |
|------|------|
| **이벤트 매칭** | 새 리스팅이 등록되면 활성 전략과 매칭하여 세션 후보 생성 |
| **일괄 평가** | `batchEvaluate()`로 N개 리스팅을 한 번에 평가하고 순위 산출 |
| **세션 생명주기** | CREATED → ACTIVE → NEAR_DEAL → ACCEPTED 전이 관리 |
| **토폴로지 감지** | 사용자의 세션 집합에서 1:1, 1:N, N:M 구조를 자동 판별 |
| **Top N 관리** | 상위 N개 세션만 활성 유지, 나머지는 WAITING 상태로 대기 |
| **크로스 프레셔** | 1:N 토폴로지에서 경쟁 세션 존재를 활용한 양보 압박 |
| **재평가 트리거** | 시장가 변동, 새 리스팅 등장 시 전략/세션 재평가 실행 |

### 2.2 토폴로지 자동 감지

사용자의 활성 세션 집합을 분석하여 협상 토폴로지를 자동으로 판별한다:

```python
def detect_topology(user_sessions: list) -> Topology:
    unique_items = set(s.item_category for s in user_sessions)
    unique_counterparties = set(s.counterparty_id for s in user_sessions)

    if len(user_sessions) == 1:
        return Topology.ONE_TO_ONE
    elif len(unique_items) == 1 and len(unique_counterparties) > 1:
        return Topology.ONE_TO_N
    elif len(unique_items) > 1 and len(unique_counterparties) > 1:
        return Topology.N_TO_M
    else:
        return Topology.INDEPENDENT
```

**토폴로지별 동작:**

| 토폴로지 | 설명 | 크로스 프레셔 |
|----------|------|--------------|
| `ONE_TO_ONE` | 단일 상대와 단일 협상 | 없음 |
| `ONE_TO_N` | 동일 상품, 복수 판매자와 병렬 협상 | 활성 (경쟁 세션 수 반영) |
| `N_TO_M` | 복수 상품, 복수 판매자와 동시 협상 | 상품별 그룹 내에서 활성 |
| `INDEPENDENT` | 서로 무관한 세션들 | 없음 |

### 2.3 세션 상태 머신

```
                         ┌──────────────────────────────┐
                         │                              │
                         ▼                              │
┌─────────┐    ┌─────────┐    ┌───────────┐    ┌──────────┐
│ CREATED │───▶│ ACTIVE  │───▶│ NEAR_DEAL │───▶│ ACCEPTED │
└─────────┘    └────┬────┘    └─────┬─────┘    └──────────┘
                    │               │
                    │               ├──────────▶ EXPIRED
                    │               │
                    ├──────────────▶ EXPIRED
                    │
                    ▼
               ┌─────────┐
               │ STALLED │──────────────────▶ SUPERSEDED
               └────┬────┘                   (다른 세션에서
                    │                         거래 성사)
                    ▼
               ┌─────────┐
               │ RESUMED │──────▶ ACTIVE (복귀)
               └─────────┘

               ┌─────────┐
               │ WAITING │  (Top N 밖, 승격 대기)
               └────┬────┘
                    │  Top 세션 탈락 시
                    ▼
               ┌─────────┐
               │ CREATED │──────▶ ACTIVE
               └─────────┘
```

**상태 전이 규칙:**

| 현재 | 트리거 | 다음 | 조건 |
|------|--------|------|------|
| CREATED | 첫 제안 발송 | ACTIVE | 세션 초기화 완료 |
| ACTIVE | U_total ≥ U_threshold | NEAR_DEAL | 효용값이 수락 임계치 이상 |
| ACTIVE | 2라운드 연속 양보 없음 | STALLED | `rounds_no_concession ≥ 2` |
| ACTIVE | t_elapsed > t_deadline | EXPIRED | 데드라인 초과 |
| STALLED | 재전략화 완료 | RESUMED | LLM 또는 규칙 기반 전략 갱신 |
| STALLED | 사용자 개입 | RESUMED | 사용자가 수동으로 파라미터 변경 |
| NEAR_DEAL | 사용자 승인 | ACCEPTED | 사용자가 최종 가격 확인 |
| NEAR_DEAL | t_elapsed > t_deadline | EXPIRED | 승인 전 데드라인 초과 |
| ACTIVE / STALLED | 다른 세션 ACCEPTED | SUPERSEDED | 동일 전략의 다른 세션에서 거래 성사 |
| WAITING | Top 세션 탈락 | CREATED → ACTIVE | 순위 상승으로 자동 승격 |

> **SUPERSEDED 규칙:** 1:N 토폴로지에서 한 세션이 ACCEPTED되면, 동일 상품의 나머지 세션은 자동으로 SUPERSEDED 처리된다.

### 2.4 세션별 컨텍스트 적용

동일한 Master Strategy가 각 리스팅의 데이터와 결합되어 세션별로 고유한 NegotiationContext를 생성한다:

```python
def apply_master_to_listing(
    master: MasterStrategy,
    listing: ListingData,
    session: SessionState
) -> NegotiationContext:
    return NegotiationContext(
        weights=master.weights,
        price=PriceContext(
            p_effective=listing.price + listing.shipping_cost,
            p_target=master.p_target,
            p_limit=master.p_limit
        ),
        time=TimeContext(
            t_elapsed=session.elapsed,
            t_deadline=master.t_deadline,
            alpha=master.alpha
        ),
        risk=RiskContext(
            r_score=listing.seller_reputation,
            i_completeness=listing.completeness
        ),
        relationship=RelationshipContext(
            n_success=get_history(user, listing.seller_id),
            n_dispute_losses=get_disputes(user, listing.seller_id),
            n_threshold=master.n_threshold
        )
    )
```

**핵심 포인트:**

- `p_effective`는 리스팅 가격 + 배송비로 계산된다 (판매자마다 다름)
- `r_score`는 판매자의 신뢰 점수로, 리스팅별로 다르다
- `n_success` / `n_dispute_losses`는 해당 판매자와의 과거 거래 이력이다
- `t_elapsed`는 세션별로 독립적으로 진행된다

이 구조 덕분에 **하나의 전략으로 N개의 세션을 동시에 운영**하면서도, 각 세션은 상대방의 특성에 맞춰 독립적으로 효용을 계산하고 의사결정을 수행한다.

### 3.1 MultiIssueMasterStrategy

```typescript
interface MultiIssueMasterStrategy {
  id: string;
  user_id: string;
  issue_schema_ref: string;          // 이슈 스키마 참조
  issue_weights: IssueWeight[];      // 이슈별 가중치

  risk: RiskCostParams;              // 리스크 비용 파라미터
  relationship: RelationshipBonusParams; // 관계 보너스 파라미터

  u_min: number;                     // 최소 수락 효용
  u_0: number;                       // 초기 목표 효용
  beta: number;                      // Faratin beta
  t_deadline: number;                // 총 마감 시간
  stall_threshold?: number;          // 교착 판단 기준 (기본 4)
```
