## 1. 1:N 흐름 (구매자 1, 판매자 N)

구매자 한 명이 여러 판매자를 동시에 탐색하고 병렬 협상하는 시나리오다.
Haggle의 기본 협상 패턴이며, 구매자 측이 오케스트레이션을 주도한다.

### 1.1 전체 흐름

```
사용자 ──────────────────────────────────────────────────────────────────
  │  "아이패드 프로 M4 사고 싶어, 예산 $800"
  ▼
LLM (또는 Stats → 파라미터)
  │  카테고리, 예산, 선호도 분석
  ▼
Master Strategy 생성
  │  p_target=$720, p_limit=$800, w_price=0.5, w_time=0.2, ...
  ▼
리스팅 조회
  │  해당 카테고리 200개 발견
  ▼
사전 필터 (v1.0.1)
  │  r_score_minimum, i_completeness_minimum 적용
  │  200 → 170개 통과
  ▼
Engine Core — batchEvaluate (34ms)
  │  같은 Master Strategy × 170개 리스팅 → 170개 U_total 계산 → 정렬
  ▼
Top 5 세션 생성 ──────────── 나머지 165개 → WAITING 풀
  │
  ├─ Session A: 리스팅 #42  ($750, 평점 4.8)
  ├─ Session B: 리스팅 #87  ($720, 평점 4.5)
  ├─ Session C: 리스팅 #13  ($780, 평점 4.9)
  ├─ Session D: 리스팅 #156 ($740, 평점 4.7)
  └─ Session E: 리스팅 #201 ($760, 평점 4.6)
  │
  ▼
라운드 진행 (Hot Path만, LLM 0회)
  │  각 세션 독립 실행: executeRound() → COUNTER / ACCEPT / REJECT
  │  크로스 프레셔: BATNA를 다른 세션에 주입
  ▼
재평가 트리거 (새 리스팅 등록 or 4시간 경과)
  │  Engine Core 일괄 재평가 → Top 5 재선정
  │  기존 세션 중 순위 밀린 것 → SUPERSEDED
  │  WAITING 풀에서 새 세션 승격
  ▼
Session D: NEAR_DEAL ($735)
  │  사용자에게 수락 여부 확인
  ▼
사용자 수락 → Session D: ACCEPTED
  │
  ├─ Session A, B, C, E → SUPERSEDED (정상 종료)
  └─ WAITING 풀 165개 → 정리 (세션 미생성 상태로 삭제)
```

**핵심 포인트:**
- `batchEvaluate`는 하나의 Master Strategy로 N개 리스팅을 동시에 평가한다.
- 병렬 세션 수(기본 5)는 리소스와 응답 시간의 균형점이다.
- WAITING 풀은 재평가 시점에만 활성화되어 불필요한 세션 생성을 방지한다.
- 전체 Hot Path에서 LLM 호출은 0회이며, 에스컬레이션 시에만 LLM이 개입한다.

## 2. N:1 흐름 (구매자 N, 판매자 1)

하나의 리스팅에 여러 구매자가 접근하는 시나리오다.
판매자 측이 오케스트레이션을 주도하며, 경매와 유사하지만 개별 협상으로 운영된다.

### 2.1 1:N vs N:1 비교

| 항목 | 1:N (구매자가 오케스트레이션) | N:1 (판매자가 오케스트레이션) |
|------|-------------------------------|-------------------------------|
| 일괄 평가 주체 | 구매자 측 | 판매자 측 |
| 크로스 프레셔 방향 | 구매자 → 판매자들에게 | 판매자 → 구매자들에게 |
| 경쟁 효과 | 판매자 가격 하락 | 구매자 가격 상승 |
| Anti-Sniping | 불필요 | 필요 |

- **1:N**에서는 구매자가 여러 판매자 사이에서 최적 거래를 찾으므로 판매자 간 가격 경쟁이 발생한다.
- **N:1**에서는 판매자가 여러 구매자의 제안을 비교하므로 구매자 간 가격 상승 압력이 발생한다.
- N:1에서는 마감 시간 근처의 전략적 지연(스나이핑)을 방지하기 위해 Anti-Sniping이 필수다.

### 2.2 Anti-Sniping

마감 직전 갑작스러운 제안으로 다른 참가자가 대응할 시간을 빼앗는 행위를 방지한다.

```
IF 마감 5분 이내에 새 제안 도착:
  → 마감 3분 연장 (최대 2회, 총 6분 추가 가능)
  → 모든 참가자에게 통지
```

**규칙 상세:**
- 연장은 최대 2회까지만 허용된다. 3번째 마감 시간 이후에 도착한 제안은 거부된다.
- 연장 발생 시 모든 활성 세션 참가자에게 HNP 알림이 전송된다.
- 연장된 시간 내에 추가 제안이 들어오면, 그 시점부터 다시 5분 카운트가 적용된다 (총 연장 횟수 제한 내에서).

## 3. N:M 분해

Haggle은 N:M 다대다 시장을 직접 처리하지 않는다.
대신 여러 개의 **1:N**과 **N:1**로 분해하여 각각을 독립적으로 운영한다.

### 3.1 분해 예시

```
시장:
  판매자 A: 에어팟 $200 | 판매자 B: 에어팟 $180 | 판매자 C: 에어팟 $220
  구매자 X: 예산 $190   | 구매자 Y: 예산 $210   | 구매자 Z: 예산 $170

분해:
  구매자 X → 1:N (A, B 가능)
  구매자 Y → 1:N (A, B, C 모두 가능)
  구매자 Z → 1:N (B만 가능)

  판매자 B 관점: N:1 (X, Y, Z 모두 접근)
  판매자 A 관점: N:1 (X, Y 접근)
  판매자 C 관점: 1:1 (Y만)
```

### 3.2 설계 결정

> **"Haggle은 옥션이 아니라 협상 플랫폼이다. 각 거래는 양 당사자의 자율적 합의여야 한다."**

이 원칙에 따라:
- 각 세션은 1:1 양자 협상이다. 다자 경매 방식은 채택하지 않는다.
- 크로스 프레셔는 정보 제공일 뿐, 자동 수락이나 강제 낙찰이 없다.
- 구매자와 판매자 모두 언제든 자유롭게 세션을 종료할 수 있다.
- N:M 분해의 충돌(같은 리스팅에 복수 구매자가 ACCEPTED)은 선착순 + Anti-Sniping으로 해결한다.

## 4. 크로스 프레셔

크로스 프레셔는 병렬 협상에서 경쟁 정보를 활용하여 더 유리한 합의를 유도하는 메커니즘이다.
Haggle의 핵심 가치인 **공정함**과 **투명함**을 준수하면서 정보 비대칭을 최소화한다.

### 4.1 정보 흐름

```
Session Orchestrator
  │
  ├─ Session A: 현재 제안 $780
  ├─ Session B: 현재 제안 $740  ← 최선 (BATNA)
  └─ Session C: 현재 제안 $810
  │
  │  BATNA = $740 (Session B가 최선)
  │
  ├─→ Session A에 주입: "다른 판매자가 $740 제안 중"
  ├─→ Session C에 주입: "다른 판매자가 $740 제안 중"
  └─── Session B에는 미주입 (이미 최선)
```

오케스트레이터는 `compareSessions()`를 통해 BATNA를 산출하고,
최선이 아닌 세션에만 경쟁 정보를 주입한다.

### 4.2 규칙

| 규칙 | 설명 |
|------|------|
| 최선에는 미주입 | 이미 최선인 상대에게는 크로스 프레셔를 주입하지 않는다. 불필요한 압박을 방지한다. |
| BATNA만 공개 | 구체적 상대방 정보(누구인지, 어떤 리스팅인지)는 비공개다. 가격 수준만 전달한다. |
| 허위 정보 금지 | 존재하지 않는 경쟁자나 허위 가격을 언급하는 것은 금지된다. 실제 BATNA만 사용한다. |
| 세션당 최대 2회 | 하나의 세션에 크로스 프레셔 정보를 주입하는 횟수는 최대 2회로 제한한다. 과도한 압박을 방지한다. |
| 차이 5% 미만이면 미주입 | 현재 제안과 BATNA의 차이가 5% 미만이면 주입하지 않는다. 미미한 차이는 오히려 역효과를 낼 수 있다. |

### 4.3 Engine Core 반영

크로스 프레셔 정보는 두 가지 경로로 협상에 반영된다.

**경로 1: CompetitionContext로 자동 반영**

오케스트레이터가 CompetitionContext를 구성하여 Engine Core에 전달하면,
엔진이 자동으로 V_p를 조정하여 더 강경한 역제안을 생성한다.

```
Orchestrator → CompetitionContext {
  n_competitors: 4,
  batna: $740,
  position: 0.43
}
  → Engine Core: V_p^adjusted 계산
  → 더 강경한 역제안 생성
```

**경로 2: HNP 메시지로 상대에게 전달**

오케스트레이터가 HNP CompetitivePressure 메시지를 상대 에이전트에게 전송하면,
상대의 Strategy Skill이 해당 정보를 해석하여 자체적으로 전략을 조정한다.

```
Orchestrator → HNP CompetitivePressure 메시지
  → 상대 에이전트 수신
  → 상대의 Strategy Skill이 반응 (양보율 증가 또는 무시)
```

**v1.0.1 추가사항:**

`cross_pressure_sensitivity`가 낮은 에이전트(Market Sense 스탯이 낮은)는
크로스 프레셔 정보를 수신해도 CompetitionContext 반영이 약하다.
이는 시장 감각이 부족한 에이전트가 경쟁 정보에 과잉 반응하지 않도록 보호하는 장치다.

```
sensitivity = agent.stats.market_sense / 100
adjusted_batna_weight = base_weight * sensitivity

// market_sense = 30인 에이전트: 크로스 프레셔 영향 30%만 반영
// market_sense = 90인 에이전트: 크로스 프레셔 영향 90% 반영
```
