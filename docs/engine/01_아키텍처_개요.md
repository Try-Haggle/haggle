## 1. 시스템 위치

Haggle Engine Core는 4-Layer Skills 아키텍처의 L2에 위치합니다.

```
+--------------------------------------------------------------+
|  L0: Gateway                                                  |
|  프로토콜 어댑터 (MCP, UCP, REST, AP2)                         |
|  채널 어댑터 (Web App, ChatGPT Apps, WhatsApp)                 |
|  인증 & 제한 (OAuth 2.0, Rate Limiting)                        |
+-----------------------------+--------------------------------+
                              |
+-----------------------------v--------------------------------+
|  L1: Skill Layer                                              |
|  +----------------------------------------------------------+|
|  | Skill Router | Skill Coordinator | Event Bus              ||
|  +----------------------------------------------------------+|
|  | Core:   GoalParser | Strategy | Negotiation | Session     ||
|  | Domain: Listing | Reputation | MarketResearch             ||
|  | Infra:  Settlement | Escrow | Shipping | Dispute          ||
|  | Intel:  Prediction | IntelligenceRouter | StrategyAPI     ||
|  +----------------------------------------------------------+|
|  | v1.0.2 신규:                                              ||
|  | TermRegistry | OpponentPrior | DeadlineManager            ||
|  | SelectionPolicy | TacticRouter                            ||
|  +----------------------------------------------------------+|
+-----------------------------+--------------------------------+
                              |
+-----------------------------v--------------------------------+
|  L2: Engine Core  <-- 이 문서의 핵심                           |
|  +----------------------------------------------------------+|
|  | 다중 Term 효용 계산기 (V_j 가중 합산)                       ||
|  | Term Space: NEGOTIABLE + INFORMATIONAL                     ||
|  +----------------------------------------------------------+|
|  | Decision Maker (규칙 기반)                                 ||
|  | U_total vs 임계값 -> ACCEPT/COUNTER/REJECT/ESCALATE        ||
|  +----------------------------------------------------------+|
|  | 양보 곡선 (효용 공간 Faratin)                               ||
|  | U_target(t) + 동적 베타 + AC_next                          ||
|  +----------------------------------------------------------+|
|  | Offer Inverter (v1.0.2 신규)                               ||
|  | U_target -> 다중 Term 값 조합 역산                          ||
|  +----------------------------------------------------------+|
|  | 이동 분류기 (v1.0.2 확장: 6종)                              ||
|  | + 베이지안 상대방 모델 + Reputation Prior                   ||
|  +----------------------------------------------------------+|
|  | 동적 마감 (v1.0.2 신규)                                    ||
|  | 판매자 마감일 + Faratin t_max 수식                          ||
|  +----------------------------------------------------------+|
|  | 선택 정책 (v1.0.2 신규)                                    ||
|  | 병렬 협상 최적 offer 선택                                   ||
|  +----------------------------------------------------------+|
|  | 일괄 평가기 | 멀티 세션 비교기                               ||
|  +----------------------------------------------------------+|
+-----------------------------+--------------------------------+
                              |
+-----------------------------v--------------------------------+
|  L3: Wire + Data                                              |
|  HNP (Protobuf/gRPC) | Redis (Hot) | PostgreSQL (Cold)       |
+--------------------------------------------------------------+
```

### 레이어별 역할

| 레이어 | 역할 | 특징 |
|--------|------|------|
| L0: Gateway | 외부 프로토콜 → 내부 변환 | UCP, MCP, HNP, REST 지원 |
| L1: Skill Layer | 비즈니스 로직 오케스트레이션 | DB/API/LLM 접근 가능, 이벤트 버스 통신 |
| L2: Engine Core | 순수 수학 계산 | DB/API/LLM 호출 **불가**, 200μs/계산, 외부 의존성 0 |
| L3: Wire + Data | 프로토콜 직렬화 + 데이터 저장 | HNP Protobuf, Redis 캐시, PostgreSQL 영속 |

## 2. 설계 원칙

1. **결정론(Determinism)** — 동일 입력 → 동일 출력. 암묵적 상태 없음, 난수 없음.
2. **제한된 출력(Bounded Output)** — 모든 V_j ∈ [0, 1], U_total ∈ [0, 1]. 예외 없음.
3. **차원 독립성(Dimensional Independence)** — 각 V_j는 서로소인 입력 집합에 의존.
4. **역할 대칭(Role Symmetry)** — 구매자/판매자 수식은 구조적으로 동일. 파라미터 방향만 다름.
5. **스킬 경계 명확성(Skill Boundary Clarity)** — Engine Core는 DB, API, LLM을 호출하지 않음(MUST NOT).
6. **엔진이 곧 깔때기(Engine as Funnel)** — Engine Core 일괄 평가가 리스팅 선별을 수행. 별도 필터 불필요.
7. **하위 호환성(Backward Compatibility)** — 단일 Term(가격만) 모드가 기본값. 다중 Term은 `term_space` 파라미터 제공 시 활성화. (v1.0.2 → v1.1.0 리네이밍: Issue→Term)

## 3. Engine-First, 리액티브 에스컬레이션

**기본은 규칙 엔진. 모르면 LLM에 물어본다.**

전체 협상 라운드의 95%+는 Engine Core(200μs)가 처리한다. LLM은 엔진이 처리할 수 없는 상황을 만났을 때만 호출된다.

```
Hot Path (매 라운드, 95%+ 트래픽):
  상대 역제안 -> Skill Coordinator -> Engine Core (200μs) -> Decision Maker
    -> v1.0.2: 이동 분류 (6종) -> 상대방 모델 업데이트 (베이지안)
    -> 동적 베타 -> 역제안 (효용 공간 + Offer Inverter)
    -> AC_next 체크 -> 응답
  LLM 호출: 0회

Cold Path (초기 전략 수립):
  사용자 목표 -> Strategy Skill -> LLM (Grok 4.1 Fast) -> MasterStrategy
  빈도: 제품당 1회 이상

리액티브 에스컬레이션 (협상 중 "모르면 물어보기"):
  상대 제안 수신 -> Engine Core가 처리 불가 판단 -> ESCALATE
  -> LLM이 상황 해석 + 전략 갱신 -> Engine Core에 재입력
```

### 핵심 비유: Engine Core는 계산기, LLM은 컨설턴트

계산기(Engine Core)가 효용을 계산하는 것은 매우 빠르고 저렴하다(200μs). 하지만 상대가 번들 제안처럼 비표준 요소를 보내면, 계산기는 이를 처리할 수 없다. 이때 컨설턴트(LLM)에게 물어본다.

```
상대 엔진: "번들 제안 — iPad $750 + Case $30 = $780, 단독이면 $810"
    │
    ▼
Engine Core: "번들?" → ESCALATE (처리 불가 요소 감지)
    │
    ▼
LLM (Grok 4.1 Fast, reasoning=ON):
  "번들 가치 분석:
   - 케이스 시장가 ~$40 → $30은 $10 할인
   - 실질 가격: $780 (번들) vs $810 (단독)
   - p_effective = $780으로 재계산 추천"
    │
    ▼
Engine Core: 갱신된 파라미터로 U_total 재계산 (200μs)
    → U_total = 0.78 → NEAR_DEAL
```

**하나의 제품 = 하나의 Master Strategy.** 에스컬레이션은 전략을 **부분 갱신**하거나 **상대 제안을 재해석**하는 것이지, 새 전략을 만드는 것이 아니다.

## 4. v1.0.2 전체 라운드 실행 파이프라인

```
executeRound(session, strategy, incomingOffer, roundData):

  1. 컨텍스트 조립
     - term_space가 있으면: 다중 Term 효용 사용
     - 없으면: 기존 4차원 (NegotiationContext)

  2. 효용 계산 (U_total)
     → 02_효용_함수.md 참조

  3. 핵심 의사결정
     ACCEPT / COUNTER / NEAR_DEAL / REJECT / ESCALATE
     → 05_의사결정_전술.md 참조

  4. COUNTER 또는 NEAR_DEAL인 경우:
     4a. 동적 마감 계산 (listing_deadline 제공 시)
     4b. effective beta 계산
     4c. Faratin 효용 공간 곡선으로 U_target 계산
     4d. Offer Inversion (단일/다중 Term)
     4e. AC_next 체크: 상대 제안 >= 역제안 → ACCEPT
     → 03_양보_곡선_역산.md 참조

  5. 상대 이동 분류 (3종 또는 6종)

  6. 상대 모델 업데이트 (EMA + 베이지안)
     → 04_상대방_모델.md 참조

  7. 전술 조정 적용 (미러링 전략, 선택적)
     → 05_의사결정_전술.md 참조

  8. 발신 HNP 메시지 생성

  9. ESCALATE면 에스컬레이션 요청 생성

  10. RoundResult 반환
```

## 5. Intent-First API 연동 (v1.0.2)

Intent-First API는 다중 Term 협상을 자연스럽게 지원합니다:

```
Explicit params → NEGOTIABLE Term domain
  max_price, target_price → price Term의 domain

Hard constraints → INFORMATIONAL Term 필터
  { key: "battery_cycle", op: "lt", value: 100 }

Soft preferences → Term 가중치 조정
  { key: "original_box", weight: 0.4 }

Well-known keys → 기존 4차원 가중치 시드
  price_sensitivity → w_p 시드

Free context + Intent → GoalParser LLM이 나머지 가중치 보강
  "배터리 상태가 핵심이에요" → battery_health의 w_j 증가
```

GoalParser v1.0.2는 기존 가중치에 더해 `TermSpace`도 생성합니다:

```
GoalParser 출력 (v1.1.0):
  NegotiationGoal {
    // 기존 (보존)
    weights: UtilityWeights
    p_target, p_limit, alpha, beta, ...

    // v1.1.0 (v1.0.2에서 리네이밍)
    term_space?: TermSpace {
      terms: Term[]
      current_values: {...}
    }
  }
```

## 6. 데이터 윤리

### 상대방 모델 데이터 취급

| 데이터 | 공개 여부 | 이유 |
|--------|----------|------|
| 카테고리 집계 beta 평균 | 공개 (Intelligence API) | 카테고리 수준 통계 |
| 개인 상대 beta_posterior | 비공개 (엔진 전용) | 개인 행동 패턴 |
| 추정 상대 가중치 | 비공개 (엔진 전용) | 개인 선호 데이터 |
| 이동 분류 이력 | 비공개 (세션별) | 세션 기밀 |

### 원칙

1. **양쪽 공정성:** 한쪽에만 유리한 데이터 노출 금지.
2. **집계만 공개:** Intelligence API 데이터는 모두 카테고리 수준 집계.
3. **비공개 = 대행 가치:** 개인 패턴 데이터는 Haggle 대행 서비스를 통해서만 활용.
4. **양쪽에 동일 적용:** 판매자 패턴을 구매자에게, 구매자 패턴을 판매자에게 공개하지 않음.
