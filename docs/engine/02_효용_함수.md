## 1. 총 효용 함수

### 1.1 기본 공식

$$U_{total} = \sum_{i} w_i \cdot V_i$$

**제약조건:**
- 모든 w_i >= 0, SUM(w_i) = 1.0
- 모든 V_i ∈ [0, 1] (각 차원 수식에 의해 보장)
- 따라서 U_total ∈ [0, 1]

### 1.1.1 단위 규칙 (2026-04-25)

엔진과 LLM 파이프라인의 모든 가격은 **minor units**를 표준으로 사용한다.

```
$500.00  → 50000
$28,000 → 2800000
```

표시 계층만 locale/currency formatter를 사용한다. LLM 프롬프트, HNP 메시지, DB snapshot, 테스트 fixture에서는 `*_minor` 필드를 우선하고, dollar 단위 필드는 입력 편의용 alias로만 허용한다.

이 규칙의 목적:

1. `$280` / `$28,000` 같은 단위 혼선을 방지한다.
2. JS floating point 반올림과 locale parsing 차이를 제거한다.
3. HNP wire protocol의 `Money { currency, units_minor }` 원칙과 맞춘다.

### 1.2 4차원 모드 (v1.0.0~v1.0.1)

```
U_total = w_p * V_p + w_t * V_t + w_r * V_r + w_s * V_s
```

### 1.3 다중 Term 모드 (v1.0.2 → v1.1.0 리네이밍, 하위 호환)

```
U_total = SUM_j( w_j * V_j(x_j) )    -- j는 모든 Term (NEGOTIABLE + INFORMATIONAL)

제약:
  SUM(w_j) = 1.0   (모든 Term에 걸쳐, 타입 무관)
  모든 V_j ∈ [0, 1]
  따라서 U_total ∈ [0, 1]
```

**하위 호환성:** `term_space`가 제공되지 않으면, 엔진은 기존 4차원 모드를 사용합니다:
- `V_p` (가격) = NEGOTIABLE, PriceContext에서 domain 생성
- `V_t` (시간) = INFORMATIONAL (라운드별 고정)
- `V_r` (리스크) = INFORMATIONAL (라운드별 고정)
- `V_s` (관계) = INFORMATIONAL (라운드별 고정)

### 1.4 가중치 입력원

가중치는 입력으로 제공된다. Engine Core는 가중치를 수정하지 않는다.

```
입력: UtilityWeights {
  w_p: float  // 경제적 가중치
  w_t: float  // 시간 가중치
  w_r: float  // 리스크 가중치
  w_s: float  // 관계 가중치
}

공급자: Strategy Skill
  - 사용자 목표 + 페르소나 + 시장 데이터 기반
  - LLM이 초기 생성, 이후 Skill이 규칙 기반 미세 조정
  - v1.0.1: AgentStats로부터 자동 도출 가능 (06_에이전트_스탯.md 참조)
```

### 1.5 Protobuf

```protobuf
message NegotiationContext {
  UtilityWeights weights = 1;
  PriceContext price = 2;
  TimeContext time = 3;
  RiskContext risk = 4;
  RelationshipContext relationship = 5;
  CompetitionContext competition = 6;  // 선택적. 없으면 1:1 단독 협상.
}

message UtilityWeights {
  float w_p = 1;
  float w_t = 2;
  float w_r = 3;
  float w_s = 4;
}

message UtilityResult {

  float v_p = 2;

  float v_r = 4;
  float v_s = 5;
  string error = 6;        // 비어있으면 성공
  string error_detail = 7;
}
```

## 2. V_p: 경제적 효용 (Economic Utility)

### 2.1 공식

**구매자 (P_target < P_limit):**

$$V_p = \begin{cases} 0 & \text{if } P_{effective} \geq P_{limit} \\ \text{clamp}\left(\dfrac{\ln(P_{limit} - P_{effective} + 1)}{\ln(P_{limit} - P_{target} + 1)}, \; 0, \; 1\right) & \text{otherwise} \end{cases}$$

**판매자 (P_target > P_limit):**

$$V_p = \begin{cases} 0 & \text{if } P_{effective} \leq P_{limit} \\ \text{clamp}\left(\dfrac{\ln(P_{effective} - P_{limit} + 1)}{\ln(P_{target} - P_{limit} + 1)}, \; 0, \; 1\right) & \text{otherwise} \end{cases}$$

**일반화:** P_effective가 마지노선을 초과(구매자) 또는 미달(판매자)하면 V_p = 0.

### 2.2 변수 정의

| 변수 | 정의 | 공급자 |
|------|------|--------|
| P_effective | 실질 가격 (리스팅 가격 + 배송비) | Shipping Skill |
| P_target | 목표 가격 (이상적 가격) | Strategy Skill. v1.0.1: AgentStats의 anchor_ratio로 도출 가능. |
| P_limit | 마지노선 (이 이상/이하 수락 불가) | Strategy Skill (사용자 설정) |

### 2.3 해석

- **V_p = 1:** 목표가 이상. 매우 좋은 거래.
- **V_p ≈ 0.5:** 목표가와 마지노선의 중간.
- **V_p = 0:** 마지노선 도달 또는 초과. 수락 불가.
- **V_p 계산 불능 (P_target = P_limit):** 오류 반환 `ZERO_PRICE_RANGE`.

### 2.4 역할 대칭

| | 구매자 | 판매자 |
|---|---|---|
| 좋은 거래 | P_effective < P_target | P_effective > P_target |
| 마지노선 | P_effective ≥ P_limit → V_p = 0 | P_effective ≤ P_limit → V_p = 0 |
| P_limit 방향 | P_limit > P_target | P_limit < P_target |

Engine Core는 P_target과 P_limit의 대소 관계로 역할을 판별한다.

### 2.5 ln() 함수 선택 이유

| 속성 | 설명 |
|------|------|
| 오목성 | 목표가 근처에서 민감, 마지노선 근처에서 둔감 |
| 마이너스 방지 | `ln(x + 1)` 은 x ≥ 0에서 항상 ≥ 0 |
| 비율 정규화 | 분자/분모 구조로 자연스럽게 [0, 1] 매핑 |

### 2.6 경쟁 조정 (CompetitionContext)

1:N 또는 N:1 토폴로지에서 경쟁 상황을 V_p에 반영한다.

```protobuf
message CompetitionContext {
  int32 n_competitors = 1;          // 활성 경쟁 세션 수
  float best_alternative = 2;       // BATNA: 현재 최선의 대안 가격
  float market_position = 3;        // 이 제안의 시장 내 위치 [0, 1]
}
```

CompetitionContext가 있으면:

$$V_p^{adjusted} = \text{clamp}(V_p \times (1 + \gamma \cdot \ln(N_{competitors} + 1) \cdot M_{position}), \; 0, \; 1)$$

- γ = 경쟁 민감도 (Strategy Skill 제공, 기본 0.1). v1.0.1: AgentStats의 market_sense로 도출.
- CompetitionContext 없으면: V_p는 기본 공식 그대로. 하위 호환 100%.

### 2.7 Protobuf

```protobuf
message PriceContext {
  float p_effective = 1;
  float p_target = 2;
  float p_limit = 3;
}
```

## 3. V_t: 시간 효용 (Time Utility)

### 3.1 공식

$$V_t = \max\left(V_{t,floor}, \; \left(\max\left(0, \; 1 - \frac{t_{elapsed}}{t_{deadline}}\right)\right)^{\alpha}\right)$$

`V_t`는 `urgency = low|medium|high` 같은 정성 enum에서 직접 나오지 않는다. 제품/프로덕션에서는 **남은 시간(`t_deadline - t_elapsed`)과 양보 곡선 파라미터(`α`, `β`, `V_t_floor`)**가 가격을 바꾸는 유일한 표준 경로다.

예: 같은 판매자라도 오늘 마감이면 `t_elapsed/t_deadline`이 커져 시간 효용이 낮아지고, 내일 마감이면 시간 효용 손실이 작다. 이 차이가 utility-space Faratin 역산을 통해 실제 counter price 차이로 이어진다.

### 3.2 변수 정의

| 변수 | 정의 | 공급자 |
|------|------|--------|
| t_elapsed | 경과 시간 (초) | Session Manager Skill. UTC timestamp 차이로 계산 |
| t_deadline | 최대 허용 시간 (초) | Strategy Skill. 사용자 브라우저 타임존의 local end-of-day를 UTC ISO로 정규화 |
| α | 시간 민감도 | Strategy Skill. v1.0.1: AgentStats의 patience로 도출. |
| V_t_floor | 시간 효용 바닥값 [0, 1] | Strategy Skill. v1.0.1: AgentStats의 patience로 도출. |

### 3.2.1 타임존 규칙

사용자와 상대의 지역이 달라도 계산은 UTC timestamp 기준으로 수행한다. 입력 UX에서는 브라우저 타임존을 사용해 날짜/마감 시간을 받되, 저장 시 아래 메타데이터를 함께 보존한다.

```json
{
  "deadline_at": "2026-04-26T05:59:59.999Z",
  "deadline_timezone": "America/Denver",
  "deadline_local_date": "2026-04-25",
  "deadline_local_end_of_day": true
}
```

엔진은 `deadline_at - now_utc`만 사용한다. `deadline_timezone`과 local date는 감사, 재표시, 사용자 설명용이다.

### 3.3 α의 의미

| α 값 | 곡선 형태 | 의미 | 사용 사례 |
|------|----------|------|----------|
| 0.5 | 볼록 | 초반 급락, 후반 둔감 | "빨리 끝내고 싶어" |
| 1.0 | 선형 | 균등 감소 | 기본값 |
| 3.0 | 오목 | 초반 여유, 후반 급락 | "여유 있지만 마감은 지켜야 해" |

### 3.4 V_t_floor의 의미

V_t_floor는 **시간이 아무리 지나도 V_t가 이 값 밑으로 내려가지 않게** 보장한다.

기존 문제: α = 3.0이라도 마감 근처에서 V_t가 0으로 급락한다. 사용자가 원하는 건 "시간에 쫓기지 않는 것"인데, 어떤 α를 써도 t → T이면 V_t → 0이다.

해결: V_t_floor로 바닥을 설정하면, 시간이 협상 결과에 미치는 영향 자체를 제한할 수 있다.

| 모드 | α | V_t_floor | T (마감) | 효과 |
|------|---|-----------|---------|------|
| 급함 | 0.5 | 0.0 | 24시간 | 시간 → 강한 압박 |
| 보통 | 1.0 | 0.0 | 72시간 | 시간 → 일반적 압박 |
| 여유 | 1.5 | 0.5 | 168시간 | 시간 → 약한 압박, 최소 50% 유지 |
| 천천히 | 1.0 | 0.8 | 720시간 | 시간 → 거의 영향 없음, 최소 80% 유지 |
| 무기한 | 1.0 | 0.95 | 2160시간 | 시간 → 사실상 무시 |

```
"천천히" 모드 (V_t_floor = 0.8):

V_t
1.0 ┤──────────────────────────────
    │                               ╲
0.8 ┤─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ── ── ── ── ──  ← 바닥
    │                                 (여기서 멈춤)
0.0 ┤
    └──────────────────────────────────────────────→ 시간
    0                                              T

"보통" 모드 (V_t_floor = 0.0):

V_t
1.0 ┤──────╲
    │        ╲
0.5 ┤         ╲
    │           ╲
0.0 ┤─────────────╲─────── ← 0까지 떨어짐
    └──────────────────────→ 시간
    0                      T
```

**핵심:** V_t_floor = 0.8이면, w_t × V_t의 최소값이 w_t × 0.8이 된다. w_t = 0.20이라면, 시간으로 인한 효용 손실이 최대 0.04에 불과하다.

### 3.5 Protobuf

```protobuf
message TimeContext {
  float t_elapsed = 1;
  float t_deadline = 2;
  float alpha = 3;
  float v_t_floor = 4;    // 시간 효용 바닥값 [0, 1]. 기본 0.0.
}
```

## 4. V_r: 리스크 효용 (Risk Utility)

### 4.1 공식

$$V_r = w_{rep} \times r_{score} + w_{info} \times i_{completeness}$$

$$w_{rep} + w_{info} = 1.0$$

### 4.2 변수 정의

| 변수 | 정의 | 범위 | 공급자 |
|------|------|------|--------|
| r_score | 상대방 평판 점수 | [0, 1] | Reputation Skill |
| i_completeness | 리스팅 정보 완성도 | [0, 1] | Reputation Skill |
| w_rep | 평판 가중치 | [0, 1] | Strategy Skill. v1.0.1: AgentStats의 riskRadar로 도출. |
| w_info | 정보 완성도 가중치 | [0, 1] | Strategy Skill. v1.0.1: AgentStats의 scrutiny로 도출. |

### 4.3 보장

r_score ∈ [0, 1] ∧ i_completeness ∈ [0, 1] ∧ w_rep + w_info = 1.0 → V_r ∈ [0, 1]

### 4.4 w_rep / w_info의 의미

이전 v1.0.0에서는 V_r = r_score × i_completeness (곱셈)이었다. 문제: Gold 판매자(r=0.95)가 리스팅을 대충 작성하면(i=0.30), V_r = 0.285 → 신규 판매자(r=0.50, i=0.95)의 V_r = 0.475보다 낮았다. 50건 무사고 거래를 한 사람이 신규보다 리스크가 높다는 건 직관에 반한다.

가중 평균으로 전환하면, 같은 상황에서 V_r = 0.95×0.6 + 0.30×0.4 = 0.69. 평판의 가치가 보존된다.

| 카테고리 | w_rep | w_info | 이유 |
|---------|-------|--------|------|
| 전자기기 | 0.50 | 0.50 | 사진/스펙 중요, 평판도 중요 |
| 차량 | 0.40 | 0.60 | 차량 상태 정보가 핵심 |
| 의류/패션 | 0.70 | 0.30 | 상태 주관적, 판매자 신뢰가 핵심 |
| 명품 | 0.50 | 0.50 | 진위 확인 + 판매자 신뢰 모두 중요 |
| 기본값 | 0.60 | 0.40 | 대부분의 거래에서 평판이 약간 더 중요 |

### 4.5 Protobuf

```protobuf
message RiskContext {
  float r_score = 1;
  float i_completeness = 2;
  float w_rep = 3;       // 평판 가중치. 기본 0.6.
  float w_info = 4;      // 정보 완성도 가중치. 기본 0.4.
}
```

## 5. V_s: 관계 효용 (Relationship Utility)

### 5.1 공식

$$V_s = \text{clamp}\left(V_{s,base} + \frac{N_{success}}{N_{threshold}} + P_{dispute}, \; 0, \; 1\right)$$

$$P_{dispute} = N_{dispute\_losses} \times (-0.3)$$

### 5.2 변수 정의

| 변수 | 정의 | 공급자 |
|------|------|--------|
| V_s_base | 초면 기본값 [0, 1] | Strategy Skill. 기본 0.5. v1.0.1: AgentStats의 rapport로 도출. |
| N_success | 이 상대와의 성공 거래 수 | Reputation Skill |
| N_threshold | 최대 신뢰 도달 기준 거래 수 | Strategy Skill. v1.0.1: AgentStats의 rapport로 도출. |
| N_dispute_losses | 분쟁 패소 횟수 | Reputation Skill |

### 5.3 V_s_base의 의미

P2P 거래의 90% 이상이 초면이다. 이전 공식(V_s_base = 0)에서는 초면이 V_s = 0.0 → 효용의 20%(w_s=0.20 가정)가 자동 상실되어, 모든 첫 거래가 구조적으로 불이익을 받았다.

V_s_base = 0.5로 설정하면 초면이 "중립"이 된다. 거래를 쌓으면 보너스, 분쟁이 있으면 페널티.

### 5.4 해석

| 상황 | V_s | 해석 |
|------|-----|------|
| 초면 | 0.5 | 중립 (보너스도 페널티도 없음) |
| 1회 성공 (기준 10회) | 0.6 | 약간 신뢰 |
| 3회 성공 | 0.8 | 상당한 신뢰 |
| 5회 이상 성공 | 1.0 | 최대 신뢰 (clamp) |
| 초면 + 1회 분쟁 패소 | 0.2 | 위험 신호 |
| 3회 성공 + 2회 분쟁 패소 | 0.2 | 분쟁이 신뢰를 상쇄 |

### 5.5 Protobuf

```protobuf
message RelationshipContext {
  int32 n_success = 1;
  int32 n_dispute_losses = 2;
  int32 n_threshold = 3;
  float v_s_base = 4;        // 초면 기본값. 기본 0.5.
}
```

## 6. Term Space (v1.0.2 → v1.1.0 리네이밍)

> **v1.1.0 변경:** 기존 "Issue/IssueSpace"를 "Term/TermSpace"로 리네이밍. 프로토콜 표준화 및 개발자 직관성 개선 목적. "deal terms", "contract terms"가 업계 표준 용어.

### 6.1 Term 타입 분류

모든 협상 Term은 두 가지 타입 중 하나로 분류됩니다. 이 분류가 Offer Inverter와 엔진의 조정 범위를 결정합니다.

```
NEGOTIABLE(협상 Term)
  정의: 협상 중 실시간으로 조율 가능한 조건
  속성: domain [min, max]와 direction (lower/higher is better)이 있음
  Offer Inverter: 대상 -- 엔진이 U_target으로부터 목표값 역산
  예시: price, delivery_speed, shipping_cost, warranty_months
  범위: 모든 카테고리에 보편적 (시스템 레벨)

INFORMATIONAL(정보 Term)
  정의: 고정된 사실. 협상 카드가 아닌 것.
  속성: domain 범위 없음. 가중치(w_j)만 있음.
  Offer Inverter: 대상 아님 -- 값 고정, U_total 계산 시 수락/거절 판단에만 영향
  예시: battery_health, scratch_level, original_box, component_included
  범위: 카테고리별, DB에서 계층 상속으로 로드
```

### 6.2 Term 계층 구조 (v1.1.0 신규)

```
Layer 1: Global Terms (모든 카테고리 공통, HNP 스펙에 하드코딩)
  예시: price, quantity, condition, delivery_method, delivery_cost
  → 모든 엔진이 반드시 이해해야 함

Layer 2: Category Terms (카테고리별 표준, TermSchema로 등록)
  예시: [automotive] mileage, trade_in_value, warranty_months
        [real_estate] closing_date, inspection, financing_type
  → 해당 카테고리 협상 시 양쪽 엔진이 TermSchema를 로드

Layer 3: Custom Terms (유저 자유 입력, 표준에 없는 조건)
  예시: "스피커도 같이 줄게", "토요일에만 픽업 가능"
  → 수신 엔진이 EscalationRequest(type: UNKNOWN_TERM) 발생 → LLM 해석
  → 자주 사용되는 Custom Term은 Category Term으로 승격 가능
```

### 6.3 Term 인터페이스

```typescript
interface Term {
  id: string;                          // 예: "price", "battery_health"
  type: 'NEGOTIABLE' | 'INFORMATIONAL';
  layer: 'GLOBAL' | 'CATEGORY' | 'CUSTOM';  // v1.1.0 신규
  weight: number;                      // w_j, 모든 w_j의 합 = 1.0
  domain?: {                           // NEGOTIABLE 전용. INFORMATIONAL은 null.
    min: number;
    max: number;
    direction: 'lower_is_better' | 'higher_is_better';
  };
  description?: string;               // CUSTOM Term 필수: 사람이 읽을 수 있는 설명
  proposed_value_estimate?: number;    // CUSTOM Term 선택: 제안자의 가치 추정
}

interface TermSpace {
  terms: Term[];
  current_values: Record<string, number>;  // 각 Term의 현재 x_j
}
```

### 6.4 V_j 평가 함수

**선형 (NEGOTIABLE 기본):**
```
구매자 (lower_is_better):
  V_j(x_j) = (max_j - x_j) / (max_j - min_j)

판매자 (higher_is_better):
  V_j(x_j) = (x_j - min_j) / (max_j - min_j)
```

**로그 스케일 (V_p, v1.0.0에서 보존):**
```
구매자:
  V_p = ln(P_limit - P_effective + 1) / ln(P_limit - P_target + 1)

판매자:
  V_p = ln(P_effective - P_limit + 1) / ln(P_target - P_limit + 1)
```

**INFORMATIONAL Term:**
값이 고정 (Skill Layer에서 제공). domain 범위 없음. V_j는 데이터를 제공하는 Skill이 계산.

### 6.5 다중 Term 효용 계산

```typescript
function computeMultiTermUtility(termSpace: TermSpace): number {
  let total = 0;
  for (const term of termSpace.terms) {
    const x = termSpace.current_values[term.id];
    const v = evaluateTerm(term, x);
    total += term.weight * v;
  }
  return total;
}
```

4차원 V_p/V_t/V_r/V_s는 Term으로 매핑 가능:
```
{ id: "price",        type: "NEGOTIABLE",    layer: "GLOBAL",    weight: w_p, domain: {min: p_target, max: p_limit, ...} }
{ id: "time",         type: "INFORMATIONAL", layer: "GLOBAL",    weight: w_t }
{ id: "risk",         type: "INFORMATIONAL", layer: "GLOBAL",    weight: w_r }
{ id: "relationship", type: "INFORMATIONAL", layer: "GLOBAL",    weight: w_s }
{ id: "battery",      type: "INFORMATIONAL", layer: "CATEGORY",  weight: 0.05 }
{ id: "shipping_cost",type: "NEGOTIABLE",    layer: "GLOBAL",    weight: 0.08, domain: {min: 0, max: 50, ...} }
```

### 6.6 Skill Layer 연동

```
TermRegistry Skill (v1.0.2 → v1.1.0 리네이밍):
  - 카테고리별 Term 정의 관리 (TermSchema)
  - 카테고리 계층에서 상속: Electronics > Apple > MacBook
  - Engine Core에 Term 가중치와 domain 제공
  - GoalParser가 사용자 의도에 따라 가중치 오버라이드 가능

데이터 흐름:
  GoalParser -> TermRegistry -> Term[] (가중치 포함)
  리스팅 데이터 -> INFORMATIONAL Term 값 (고정)
  Strategy -> NEGOTIABLE Term domain
  -> Engine Core가 완전한 TermSpace 수신
```

### 6.7 Custom Term 처리 프로토콜 (v1.1.0 신규)

```
수신 엔진의 처리 흐름:

Term 수신 → Layer 확인
  GLOBAL/CATEGORY → 유틸리티 함수 있음 → 바로 계산
  CUSTOM → 유틸리티 함수 없음
    → EscalationRequest(type: UNKNOWN_TERM) 발생
    → LLM이 description을 해석 → 임시 TermPreference 생성
    → 이번 세션에서만 유효
```

### 6.8 Term 승격 파이프라인 (v1.1.0 신규)

```
Custom Term이 특정 카테고리에서 1000회+ 사용
  → 해글이 자동 감지
  → 승격 후보로 등록
  → 리뷰 후 Category Term으로 승격
  → TermSchema에 추가 → LLM 에스컬레이션 없이 처리 가능
```

### 6.9 세션 핸드셰이크 (v1.1.0 신규)

```
협상 시작 시 양쪽 엔진이 TermSchema 버전을 교환:

바이어 → 셀러: SessionInit { category, supported_schema_version, custom_terms_supported }
셀러 → 바이어: SessionAck { agreed_schema, custom_terms_supported, unsupported_terms }

스키마 버전이 다르면:
  → 상위 버전 엔진이 하위 호환 모드로 다운그레이드
  → 신규 Term은 Custom Term으로 전송 → LLM 에스컬레이션으로 처리
  → Graceful degradation (웹 브라우저 호환성 패턴)
```
