## 1. Decision Maker

### 1.1 핵심 결정 규칙 (v1.0.0, 보존)

```
입력: UtilityResult + 임계값 파라미터 + 제안 파싱 결과

U_threshold:  수락 가능 최저 효용 (Strategy Skill 설정. v1.0.1: AgentStats의 resolve로 도출.)
U_aspiration: 목표 효용 (Strategy Skill 설정. v1.0.1: AgentStats의 resolve로 도출.)

규칙 (우선순위 순):

  === 파싱 실패 (최우선) ===
  0. 상대 제안에 처리 불가 요소 포함 → ESCALATE (LLM에 해석 요청)
     예: 번들, 조건부, 트레이드인, 비표준 결제 조건 등

  === 정상 처리 ===
  1. u_total ≥ U_aspiration                       → ACCEPT
  2. u_total ≥ U_threshold AND 마감 임박(V_t < 0.1) → ACCEPT
  3. u_total ≥ U_threshold                        → NEAR_DEAL (사용자에게 수락 추천)
  4. u_total > 0                                  → COUNTER (Faratin 곡선으로 역제안)
  5. u_total ≤ 0                                  → REJECT (마지노선 초과)

  === 진행 중 에스컬레이션 ===
  6. 4라운드 이상 교착 (rounds_no_concession ≥ 4) → ESCALATE (전략 재검토)
  7. V_t ≤ V_t_floor + 0.05 AND u_total < U_threshold → ESCALATE (마감 임박 + 합의 불가)
```

### 1.2 v1.0.2 의사결정 흐름 추가

핵심 결정 이후 추가 체크:

```
1. 핵심 결정 (위)
2. COUNTER 또는 NEAR_DEAL인 경우:
   a. effective beta 계산 (동적 + Faratin)
   b. Faratin 효용 공간 곡선으로 U_target 계산
   c. Offer Inverter 실행 (PROPORTIONAL 또는 OPPONENT_AWARE)
   d. AC_next 체크: 상대 제안 >= 역제안 → ACCEPT
3. 상대 이동 분류 (모델 있으면 6종, 없으면 3종)
4. 상대 모델 업데이트 (EMA + 베이지안)
5. 미러링 전략 조정 적용 (선택적, 섹션 2 참조)
```

### 1.3 FORTUNATE 이동 즉시 수락

`classifyMove6`이 FORTUNATE(양쪽 모두 이득)을 반환하면, U_total < u_aspiration이더라도 U_total > u_threshold이면 즉시 수락을 고려합니다.

```
if move_type == 'FORTUNATE' AND u_total > u_threshold:
  → ACCEPT (파레토 개선, 놓치지 않음)
```

### 1.4 스탯 기반 임계값 영향 (v1.0.1)

AgentStats가 적용되면 Decision Maker의 임계값이 스탯에 따라 조정된다:

| 스탯 | 영향 파라미터 | 효과 |
|------|-------------|------|
| Resolve | U_threshold, U_aspiration | 높을수록 수락 기준 엄격 → ACCEPT 감소, COUNTER/REJECT 증가 |
| Patience | V_t_floor | 높을수록 규칙 7 발동 확률 감소 → 시간 에스컬레이션 억제 |
| Tenacity | β (간접) | 높을수록 COUNTER 시 양보폭 감소 → 교착 가능성 증가 → 규칙 6 발동 증가 |
| Rapport | late_round_aggression_modifier | > 60이면 COUNTER 역제안이 더 관대 |

**예시: Tiger vs Dolphin의 Decision Maker 행동 차이**

```
같은 u_total = 0.55에서:

Tiger (Resolve=60):
  U_threshold = 0.58, U_aspiration = 0.73
  0.55 < 0.58 → COUNTER (기준 미달)

Dolphin (Resolve=40):
  U_threshold = 0.47, U_aspiration = 0.62
  0.55 > 0.47 → NEAR_DEAL (수락 추천)

같은 효용이지만 에이전트 성격에 따라 결정이 다르다.
```

### 1.5 ESCALATE의 두 가지 유형

**유형 1: UNKNOWN_PROPOSAL (리액티브)**
상대 엔진이 보낸 제안에 Engine Core가 이해할 수 없는 요소가 있을 때. 가장 흔한 에스컬레이션.

**유형 2: STRATEGY_REVIEW (선제적)**
4라운드 이상 교착, 마감 임박 등 전략 재검토가 필요할 때.

## 2. 전술 엔진 (v1.0.2 신규)

### 2.1 개요

전술 엔진은 엔진이 **얼마나** 양보하는지가 아니라 **어떻게** 양보하는지를 관장합니다. 상대 모델과 세션 상태를 기반으로 협상 전술을 선택하고 적용합니다.

### 2.2 미러링 전략 (Jonker et al. 5.2)

```
1. 상대의 마지막 이동을 분류 (6종)
2. 같은 카테고리의 이동으로 미러링
3. 선택적으로 NICE 이동 추가 (파레토 개선 시도)

예시:
  상대: CONCESSION (가격 +$20)
  우리 응답: CONCESSION (가격 -$10) + NICE (배송 1일 단축)
  → 상대의 협력에 호응하면서, 파레토 개선 시도
```

### 2.3 전술 선택 매트릭스

```
상대 패턴        세션 단계      전술
BOULWARE         초반           미러링 + 버팀
BOULWARE         후반           점진적 양보 + 마감 신호
CONCEDER         초반           버팀 (상대가 알아서 양보)
CONCEDER         후반           소폭 상호 양보
LINEAR           모든 단계      비례적 미러링
UNKNOWN          초반           기본 (PROPORTIONAL 역산)
UNKNOWN          후반           전략 검토 에스컬레이션
```

### 2.4 Offer Inverter와의 연동

전술 엔진은 `inversion_strategy`를 조정하고, Offer Inverter에 전달하기 전에 `U_target`을 수정할 수 있습니다:

```
상대가 BOULWARE (강하게 버팀)?
  → U_target 감소율 낮춤 (우리도 버팀)
  → PROPORTIONAL 역산 사용 (모델을 드러내지 않음)

상대가 CONCEDER (빠르게 양보)?
  → 높은 U_target 유지 (더 많은 가치 추출)
  → OPPONENT_AWARE 역산 사용 (양보 대상 최적화)

FORTUNATE 이동 감지?
  → 역산 생략, 수락 또는 현 위치 유지
```

## 3. 선택 정책 (v1.0.2 신규)

### 3.1 목적

판매자가 여러 구매자와 동시에 협상하고, 여러 offer가 수용 가능 수준에 도달했을 때 선택 메커니즘이 필요합니다.

### 3.2 수식

```
score_i = alpha * U(offer_i) + (1 - alpha) * R(buyer_i)

여기서:
  alpha           = 효용 vs 평판 가중치 (기본: 0.7)
  U(offer_i)      = 해당 offer의 총 효용
  R(buyer_i)      = 구매자의 공개 평판 점수 [0, 1]
```

### 3.3 인터페이스

```typescript
interface SelectionPolicyParams {
  alpha_selection: number;            // 효용 vs 평판 가중치 (기본: 0.7)
  selection_threshold: number;        // 자동 수락 효용 임계값
  dropout_policy: 'TIMEOUT' | 'RV_MISS' | 'EXPLICIT_REJECT' | 'LISTING_EXPIRED';
}

interface OfferCandidate {
  session_id: string;
  u_total: number;
  r_score: number;
  buyer_id: string;
}

function selectBestOffer(
  candidates: OfferCandidate[],
  alpha?: number,
): OfferCandidate;
```

### 3.4 Skill Layer 연동

```
L1: NegotiationSkill이 병렬 세션 관리
    → 어떤 세션이 ACCEPT 가능 상태에 도달하면:
       1. 모든 활성 세션 상태 수집
       2. selectBestOffer() 호출
       3. 최적 offer 수락, 나머지는 거절 또는 보류
       4. 'session.selected' 이벤트 발행
```

## 4. 일괄 평가기 (Batch Evaluator)

### 4.1 목적

하나의 Master Strategy로 N개 리스팅을 한 번에 평가한다. **엔진이 곧 깔때기** — 별도 필터링 레이어가 필요 없다.

### 4.2 사전 필터 (v1.0.1)

AgentStats에서 도출된 r_score_minimum, i_completeness_minimum이 설정되어 있으면, Batch Evaluate 전에 사전 필터링을 수행한다. 이 필터링은 **Skill Layer**에서 수행되며, Engine Core의 수학에는 영향 없음.

### 4.3 Protobuf

```protobuf
message BatchEvaluateRequest {
  UtilityWeights weights = 1;
  TimeContext time = 2;
  float alpha = 3;
  repeated ListingData listings = 4;
}

message ListingData {
  string listing_id = 1;
  float p_effective = 2;
  float p_target = 3;
  float p_limit = 4;
  float r_score = 5;
  float i_completeness = 6;
  int32 n_success = 7;
  int32 n_dispute_losses = 8;
}

message BatchEvaluateResult {
  repeated RankedListing rankings = 1;
  int32 total_evaluated = 2;
  float evaluation_time_ms = 3;
  int32 pre_filtered_count = 4;
}

message RankedListing {
  string listing_id = 1;
  int32 rank = 2;
  UtilityResult utility = 3;
}
```

**성능:** 200개 리스팅 × 200μs = 40ms. 1,000개도 200ms.

### 5.1 목적

동일 제품에 대한 N개 활성 세션의 현재 상태를 비교하여 순위를 매기고, 세션 간 전략 권고를 생성한다.

### 5.2 Protobuf

```protobuf
message SessionCompareRequest {
  repeated SessionSnapshot sessions = 1;
}

message SessionSnapshot {
  string session_id = 1;
  UtilityResult current_utility = 2;
  float last_offer_price = 3;

message SessionCompareResult {

  string session_id = 1;
```

## 6. NEAR_DEAL 밴드 (v2.0 구현)

### 6.1 수락 임계값 R(t)

$$R(t) = \max\{U_{BATNA}, \; U_{min} + (U_0 - U_{min}) \cdot (1 - \tau^\beta)\}$$

### 6.2 NEAR_DEAL 밴드 정의

`near_deal_band` (기본 δ=0.05)로 수락 임계값 아래에 근접 영역을 정의합니다:

```
if u >= R:          → ACCEPT
if u >= R - δ:      → NEAR_DEAL (거래 근접 신호)
if u < R - δ:       → COUNTER 또는 기타
```

**proximity (근접도) 계산:**
```
proximity = (u - (R - δ)) / δ
// 0 = 밴드 하단, 1에 가까울수록 수락 직전
```

### 6.3 우선순위

```
1. u >= R                             → ACCEPT
1.5. R-δ <= u < R                     → NEAR_DEAL
2. rounds_no_concession >= stall      → ESCALATE (STALL)
3. tau > 1 - deadline_critical        → ESCALATE (DEADLINE)
4. u > 0                              → COUNTER
5. u <= 0                             → REJECT
```

NEAR_DEAL은 ESCALATE보다 높은 우선순위 — 거래가 근접하면 교착/마감보다 우선.

### 6.4 near_deal_band=0으로 비활성화

밴드를 0으로 설정하면 NEAR_DEAL이 발동되지 않고 바로 COUNTER로 전환.

## 7. Parallel Session EU (v2.0 구현)

### 7.1 기대 효용

병렬 세션의 기대 효용:

$$EU_s = \hat{P}_{close} \cdot U_{best} + (1 - \hat{P}_{close}) \cdot U_{BATNA} - \kappa \cdot T_s$$

- `P̂_close`: 세션 성사 확률 추정 [0, 1]
- `U_best`: 해당 세션 최선 효용
- `U_BATNA`: 다른 세션에서의 최선 대안 효용
- `κ`: 시간 비용 계수 (기본 0.01)
- `T_s`: 경과 시간 비율 [0, 1]

### 7.2 Dynamic BATNA

$$U_{BATNA}^{dynamic} = \max_{j \neq s} \hat{P}_{close,j} \cdot U_{best,j}$$

다른 모든 세션 중 가장 높은 (성사확률 × 최선효용)이 현 세션의 BATNA.

### 7.3 세션 순위

`rankParallelSessions(sessions)`: EU 내림차순 정렬.

### 7.4 인터페이스

```typescript
interface ParallelSessionInput {
  session_id: string;
  p_close: number;     // 성사 확률
  u_best: number;      // 최선 효용
  u_batna_other: number; // 외부 BATNA
  t_spent: number;     // 경과 시간 비율
  kappa?: number;      // 시간 비용 (기본 0.01)
}
```
