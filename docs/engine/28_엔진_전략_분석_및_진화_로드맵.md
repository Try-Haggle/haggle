## 2. 대기업이 구조적으로 못하는 6가지

### 2.1 플랫폼 간 가격 비교

eBay는 FB Marketplace 매물과 자사 매물을 비교해서 "저쪽이 더 싸니까 양보하세요"라고 판매자에게 말할 수 없다. 자기 매출을 깎는 행위이기 때문이다. 모든 대형 플랫폼은 경쟁 플랫폼 가격을 참조하는 기능을 넣을 **수익 구조적 동기가 없다.**

**엔진 대응:**

| 항목 | 현재 상태 | 필요 작업 |
|------|----------|----------|
| L5 Signals | 비어있음 ("이벤트 기반"만 표기) | `CrossPlatformPriceFeed` 인터페이스 정의 |
| 대상 파일 | `context-assembly.ts` | L5 레이어에 외부 시세 주입 경로 추가 |
| Coach 반영 | Faratin 곡선만 참조 | 시장 데이터 기반 추천가 보정 로직 |

```
Stage 2 CONTEXT 조립 시:
  L5 주입 → "eBay 동일 모델 $1,100 × 3건, FB Marketplace $980"
  → Stage 3 DECIDE가 크로스플랫폼 근거로 제안
  → 어떤 단일 플랫폼 엔진도 할 수 없는 판단
```

**초기 구현:** 수동 크롤링/API로 데이터 수집 → L5 피드로 밀어넣기. 완성도보다 **"플랫폼 간 데이터가 엔진에 들어오는 경로"** 확보가 핵심.

---

### 2.2 양쪽 동시 중립

eBay 수익 = 판매자 수수료. 판매자가 떠나면 죽는다. 따라서 eBay의 AI 가격 추천은 본질적으로 "빨리 팔릴 가격"을 추천하지, 구매자 최적 가격을 추천하지 않는다. FB Marketplace는 광고 수익이 핵심이라 판매자 노출을 우선한다. **모든 기존 플랫폼은 양쪽에 동시 충성할 수 없는 수익 구조적 충돌이 있다.**

**엔진 대응:**

| 항목 | 현재 상태 | 필요 작업 |
|------|----------|----------|
| Referee | 내 에이전트 보호 전용 (V1~V7) | 양쪽 동일 규칙 적용 보장 |
| 대상 파일 | `protocol-rules.ts` | Fairness Constraint 추가 |
| 검증 방법 | 없음 | 세션 시작 시 Referee 버전 해시 교환 핸드셰이크 |

```
Protocol-Level Fairness Constraint:
  세션 초기화 시:
    Buyer Engine → Referee 규칙 해시 전송
    Seller Engine → Referee 규칙 해시 전송
    해시 불일치 → 세션 거부 (or 경고 로그)
  → "양쪽 모두에게 같은 규칙을 적용하는 유일한 중립 인프라"
```

---

### 2.3 협상 과정 데이터 축적

eBay는 최종 판매가만 안다. "처음에 $1,200을 불렀고, 5라운드 만에 $1,050에 합의했고, 3라운드째에 배터리 상태를 조건으로 교환했다" — 이런 **과정 데이터**는 어떤 기존 플랫폼에도 존재하지 않는다. FB Marketplace 메신저 협상은 비정형 텍스트라 구조화 불가.

**엔진 대응:**

| 항목 | 현재 상태 | 필요 작업 |
|------|----------|----------|
| Living Memo | PT, CL, OM 매 라운드 기록 | 세션 종료 후 NIG 적재 파이프라인 |
| checkpoint-store | in-memory | DB 영속화 + 집계 파이프라인 전송 |
| 대상 파일 | Stage 6 PERSIST, `checkpoint-store.ts` | 익명화 → 집계 경로 추가 |

```
Stage 6 PERSIST 확장:
  기존: Memo 갱신 + Phase 전이
  추가: 익명화 → NIG 집계 파이프라인 전송

결과 데이터 예시:
  "iPhone 15 Pro / Like New / CONCEDER 판매자"
  → 평균 4.2라운드, 11.3% 할인, 배터리 조건 교환 빈도 67%
  → 세상 어디에도 없는 데이터
```

---

### 2.4 외부 거래 분쟁 해결

eBay Money Back Guarantee는 eBay 거래 전용. FB Marketplace 사기 → 방법 없음. OfferUp 분쟁 → 마찬가지. **대기업은 자기 울타리 밖 거래의 분쟁을 해결할 인센티브도 메커니즘도 없다.**

**엔진 대응:**

| 항목 | 현재 상태 | 필요 작업 |
|------|----------|----------|
| Shared Memo 해시 | SHA-256 미구현 (`memo-manager.ts` 미구현) | **P0 — 즉시 구현** |
| 온체인 앵커링 | 로드맵에만 존재 | 해시 → x402 트랜잭션 기록 연동 |
| 대상 파일 | `memo-manager.ts` (신규) | 해시 계산 + 온체인 기록 호출 |

```
분쟁 보호 흐름:
  FB Marketplace 매물 → Haggle로 협상
  → 매 라운드 Shared Memo 해시 기록
  → 합의 시 x402 결제 트랜잭션에 해시 앵커링
  → 분쟁 시: 온체인 해시로 협상 기록 무결성 증명
  → "어떤 플랫폼에서 시작된 거래든 Haggle로 협상하면 분쟁 보호"
```

---

### 2.5 P2P 협상에 LLM 비용 투자

Meta가 FB Marketplace 전체 메시지에 LLM을 돌리면 월 수십만 달러. 대기업에게 P2P는 유저 트래픽을 위한 부속 기능이지 핵심 수익원이 아니다. **LLM 비용을 P2P 협상에 정당화할 수 없는 수익 구조.**

**Haggle의 구조적 우위:**

```
Haggle 단가 구조:
  세션당 LLM 비용:  $0.0055
  평균 거래 수수료:  $5.10 (1.5% × $340)
  LLM/수수료 비율:  0.1%
  → 거래 수수료로 직접 회수, 마진 99.9%

외부 API 가격 설정:
  세션당 $0.01 → 어떤 플랫폼이든 자체 개발보다 저렴
  → 대기업이 직접 만들 동기 제거
  → "사는 게 만드는 것보다 싸다"
```

**엔진 대응:** 엔진 코어(Stage 2~4)를 독립 모듈로 분리하여 외부 API로 제공 가능한 구조 확보. 현재 `llm-negotiation-executor.ts`가 13-step 모노리스 → 6-stage 모듈로 재작성 시 이 분리를 고려.

---

### 2.6 느린 합의 허용

Amazon, eBay의 전체 UX/물류/메트릭이 "빠른 전환율"에 최적화. "5라운드 협상 후 합의"는 전환율을 깎는 행위이고, KPI와 정면충돌. **기술적으로 가능해도 사내 정치적으로 통과 안 됨.**

**엔진 대응:**

| 항목 | 현재 상태 | 필요 작업 |
|------|----------|----------|
| Phase 전이 | 모든 거래에 동일 적용 | 카테고리/금액별 파이프라인 분기 |
| 대상 파일 | `config.ts` | 최소 거래금액 임계값 추가 |

```
카테고리별 파이프라인 분기:
  < $100  → 1-2라운드 간소화 모드 (OPENING → CLOSING 직행)
  $100~$500 → 표준 5-Phase
  > $500  → 전체 파이프라인 + Reasoning 활성화
  > $5,000 → 전체 + 멀티턴 시나리오 시뮬레이션

→ "고가 거래 전문 협상 인프라" 포지셔닝
```

## 3. Haggle 4대 철학으로 본 엔진 점검

### 3.1 투명성 — "엔진이 왜 이렇게 결정했는지" 사용자가 알 수 없다

**현재 상태:**

- `ProtocolDecision.reasoning`이 "내부용, 상대방에게 비공개"로 설계
- Living Memo Compressed Codec은 토큰 절약 최적화 — 사람이 읽을 수 없음
- Referee가 V4 SOFT violation을 경고로 넘겼을 때, V1 HARD에서 auto_fix를 적용했을 때, 사용자에게 전달되는 경로 없음

**문제:** "협상의 민주화"를 내세우면서 엔진 내부가 블랙박스면 기존 플랫폼과 다를 게 없다.

**필요 작업: Decision Explainability Layer**

```
라운드별 의사결정 로그 구조:

{
  round: 5,
  explainability: {
    coach_recommendation: {
      price: 85000,
      basis: "Faratin β=1.5, t/T=0.33",
      acceptable_range: { min: 83000, max: 87000 }
    },
    llm_decision: {
      price: 83000,
      tactic: "reciprocal_concession",
      reasoning_summary: "판매자가 2회 연속 양보 → 호혜적 대응"
    },
    referee_result: {
      violations: [
        { rule: "V7", severity: "SOFT", detail: "Faratin 기준 2.4% 초과 양보" }
      ],
      action: "WARN_AND_PASS",
      auto_fix_applied: false
    },
    final_output: {
      price: 83000,
      action: "COUNTER"
    }
  }
}
```

| 항목 | 대상 파일 | 작업 |
|------|----------|------|
| 로그 스키마 | `types.ts` | `RoundExplainability` 인터페이스 추가 |
| 로그 생성 | `referee-service.ts` | Coach→Validate→AutoFix 각 단계 결과 구조화 |
| 로그 저장 | Stage 6 PERSIST | DB 영속화 경로 추가 |
| 사용자 조회 | `/sessions/:id/decisions` | 라운드별 의사결정 로그 API |

---

### 3.2 중립성 — Referee가 "누구의 편"인지 구조적으로 불명확

**현재 상태:**

- V1~V7 전부 "내 에이전트"의 안전 보호 방향 (맞는 설계)
- 하지만 양쪽 엔진에 동일한 규칙 체계가 적용된다는 **검증 메커니즘 부재**
- Shared Memo SHA-256 해시 미구현 → 어느 쪽이든 사후 변조 가능 → 분쟁 시 증거 능력 없음

**필요 작업:**

| 항목 | 우선순위 | 작업 |
|------|---------|------|
| `memo-manager.ts` SHA-256 | **P0** | Shared Memo 해시 계산 + 매 라운드 기록 |
| Referee 버전 핸드셰이크 | P1 | 세션 시작 시 양쪽 Referee 규칙 버전 교환 |
| Validator 규칙 해시 | P1 | `validator.ts` 규칙 셋의 해시를 세션 메타데이터에 기록 |

---

### 3.3 합리성 — L5 Signals가 비어 있어서 "합리적 판단"의 근거가 반쪽

**현재 상태:**

- 6계층 컨텍스트 중 L5(외부 시세, 경쟁 세션)가 미구현
- Faratin 코칭이 순수하게 내부 양보 곡선에만 기반
- 시장 가격 급변, 동일 아이템 경쟁 세션 존재 시 반영 불가

**문제:** "데이터 드리븐 합리성"을 주장하면서 실제로는 세션 내부 데이터만 보고 결정.

**필요 작업:**

```
L5 Signals 인터페이스 정의:

interface L5Signals {
  market: {
    avg_sold_price_30d: number;    // 최근 30일 평균 거래가
    price_trend: 'rising' | 'stable' | 'falling';
    active_listings_count: number;  // 현재 동일 매물 수
    source_prices: { platform: string; price: number }[];
  };
  competition: {
    concurrent_sessions: number;    // 동일 아이템 진행 중 세션
    best_competing_offer: number;   // 경쟁 세션 최고 제안가
  };
  category: {
    avg_discount_rate: number;      // 카테고리 평균 할인율
    avg_rounds_to_deal: number;     // 평균 합의 라운드
    demand_supply_ratio: number;    // 수요/공급 비율
  };
}
```

| 항목 | 대상 파일 | 작업 |
|------|----------|------|
| 인터페이스 정의 | `types.ts` | `L5Signals` 타입 추가 |
| 컨텍스트 조립 | `context-assembly.ts` | L5 레이어 조립 로직 |
| Coach 반영 | `coach.ts` | 시장 데이터 기반 추천가 보정 |

---

### 3.4 미래지향성 — 단일 LLM 벤더 종속

**현재 상태:**

- `xai-client.ts`, `grok-fast-adapter.ts` — xAI Grok 4 Fast 하드코딩
- L1 Model Instructions "모델 교체 시" 갱신이라고 하지만, 실제 교체 시 프롬프트 전체 재작성 필요
- ModelAdapter 패턴이 존재하나, 진짜 교체 가능한 추상화인지 의문

**필요 작업:**

```
Stage별 ModelAdapter 분리:

// 현재: 모든 Stage가 동일 모델
const adapter = new GrokFastAdapter();  // 전역

// 목표: Stage별 독립 모델 배정
const stageAdapters = {
  UNDERSTAND: new LocalSmallAdapter(),    // 파싱 → 소형 모델
  DECIDE:     new GrokFastAdapter(),      // 핵심 판단 → 고성능 모델
  RESPOND:    new LocalSmallAdapter(),    // 생성 → 소형 모델
};
```

| 항목 | 대상 파일 | 작업 |
|------|----------|------|
| 어댑터 추상화 검증 | `grok-fast-adapter.ts` | Grok 전용 최적화가 인터페이스 오염했는지 점검 |
| Stage별 어댑터 | `config.ts` | Stage → ModelAdapter 매핑 설정 |
| 프롬프트 분리 | `prompts/` | 모델 종속적 지시문 vs 모델 독립적 규칙 분리 |

### 3.5 철학 점검 종합

| 철학 | 핵심 결함 | 수정 우선순위 |
|------|----------|-------------|
| 투명성 | 의사결정 로그가 사용자에게 도달하는 경로 없음 | **P0** |

| 미래지향성 | LLM 단일 벤더 종속, Stage별 모델 분리 안 됨 | P1 |

## 4. LLM 발전에 따른 엔진 공동 진화 전략

### 4.1 판단력 향상 → Referee 역할 전환

**현재:** Validator 7개 규칙 = LLM이 실수할 거라는 전제. HARD violation → auto_fix → fallback.

**LLM 발전 시:** HARD violation 발생률이 0에 수렴. Referee의 역할이 "실수 교정"에서 "전략 최적화 검증"으로 전환.

**엔진 변경:**

| 항목 | 현재 | 변경 |
|------|------|------|
| Validator 모드 | 항상 전체 검증 | HARD 히트율 모니터링 → Lite 모드 전환 |
| Lite 모드 | 없음 | V1~V3 HARD만 검증, V4~V7 SOFT 스킵 |
| 전환 기준 | — | HARD 히트율 < 1% 유지 시 Lite 전환 |
| 대상 파일 | `config.ts`, `validator.ts` | `validationMode: 'full' \| 'lite'` 추가 |

```
Validator Lite 모드:
  if (hardViolationRate_30d < 0.01) {
    validationMode = 'lite';  // HARD만 검증, 오버헤드 감소
  }
  
  // Lite에서도 HARD 히트 발생 시 즉시 full로 복귀
  if (hardViolation detected) {
    validationMode = 'full';
    alert('LLM 품질 저하 감지');
  }
```

### 4.2 토큰 단가 하락 → 코드 단계의 LLM 대체

**현재:** 비용 때문에 LLM 호출을 Stage ①③⑤(3회)로 제한. Stage ②④⑥은 결정론적 코드.

**토큰 단가 10x 하락 시 ($0.20/M → $0.02/M):**

| Stage | 현재 | 가능해지는 것 |
|-------|------|-------------|
| ② CONTEXT | 기계적 L0~L5 이어붙이기 | LLM이 "이번 라운드에 중요한 컨텍스트" 선별 조립 |
| ④ VALIDATE | 하드코딩 7개 규칙 | 자연어 규칙 기술 → LLM 해석 (새 카테고리 규칙 추가 시 코드 불필요) |

**엔진 변경:**

```
Stage별 LLM/코드 모드 동적 전환:

// config.ts
const stageMode = {
  UNDERSTAND: 'llm',      // 항상 LLM
  CONTEXT:    'code',      // 기본 코드, 비용 임계값 이하 시 'llm'
  DECIDE:     'llm',       // 항상 LLM
  VALIDATE:   'code',      // 기본 코드, 비용 임계값 이하 시 'llm'
  RESPOND:    'llm',       // 항상 LLM
  PERSIST:    'code',      // 항상 코드
};

// 비용 기반 자동 전환
if (currentTokenCost < THRESHOLD) {
  stageMode.CONTEXT = 'llm';
  stageMode.VALIDATE = 'llm';
}
```

| 대상 파일 | 작업 |
|----------|------|
| `config.ts` | Stage별 모드 설정 + 비용 임계값 기반 자동 전환 |
| 각 Stage 모듈 | LLM/코드 듀얼 실행 경로 |

### 4.3 컨텍스트 윈도우 확대 → Codec 필요성 소멸

**현재:** JSON 1,000토큰 → Codec 390토큰 (61% 절감). 상당한 엔지니어링 투입.

**컨텍스트 1M+ 토큰 시:** 15라운드 전체 히스토리 JSON 원문 = ~15,000토큰. 전체 넣어도 무시할 수준.

**엔진 변경:**

```
Codec/Raw 동적 전환:

// config.ts
memoEncoding: 'auto' | 'codec' | 'raw'

// auto 모드 로직:
if (model.contextWindow > 500_000 && tokenCostPerM < 0.05) {
  encoding = 'raw';    // 원문 전체 투입
  // 보너스: 과거 유사 세션 원문도 함께 투입 가능
} else {
  encoding = 'codec';  // 압축 모드 유지
}
```

**Raw 모드의 추가 가치:**

| 투입 가능 데이터 | Codec 모드 | Raw 모드 |
|----------------|-----------|---------|
| 현재 세션 히스토리 | 압축 요약만 | 전체 대화 원문 |
| 외부 시세 상세 | 숫자만 | 출처 + 맥락 포함 |
| 과거 유사 세션 | 불가 | 익명화된 유사 세션 1~2개 전체 |
| Skill 상세 지식 | 요약만 | 카테고리 전체 도메인 지식 |

| 대상 파일 | 작업 |
|----------|------|
| `memo-codec.ts` | Raw 모드 추가 (패스스루) |
| `config.ts` | `memoEncoding` 설정 + 자동 전환 로직 |
| `context-assembly.ts` | Raw 모드 시 과거 세션 포함 로직 |

### 4.4 멀티모달 → Stage 1 이미지/영상 파싱

**현재:** Stage 1 UNDERSTAND = 텍스트만 처리. 실제 P2P 거래에서 판매자가 보내는 건 사진.

**멀티모달 LLM 적용 시:**

```
Stage 1 UNDERSTAND 확장:

현재:
  "배터리 92%, 기스 없음" (텍스트)
  → { price_offer: 90000, conditions: [...] }

멀티모달:
  [폰 뒷면 사진] + [배터리 스크린샷] + "90000에 드릴게요"
  → { 
      price_offer: 90000,
      conditions: [...],
      visual_assessment: {
        battery_health: 89,        // 스크린샷에서 직접 추출
        cosmetic_grade: "excellent", // 사진에서 판단
        box_included: true,         // 사진에서 확인
        discrepancy: {
          field: "battery_health",
          claimed: 92,              // 텍스트에서 주장
          observed: 89,             // 사진에서 확인
          severity: "minor"
        }
      }
    }
```

**Skill verify() 연동:**

```
NegotiationSkill.verify() 확장:

// 현재: 외부 API 검증 (IMEI, carrier lock)
verify(term: 'imei', input: string): VerificationResult

// 추가: 이미지 기반 검증
verify(term: 'battery_health', input: { 
  claimed: number, 
  screenshot: ImageBuffer 
}): VerificationResult

// 불일치 시 자동 에스컬레이션
if (result.discrepancy) {
  coaching.warnings.push('판매자 주장과 이미지 불일치 감지');
  // → Stage 3 DECIDE에서 가격 조정 근거로 활용
}
```

| 대상 파일 | 작업 |
|----------|------|
| Stage 1 모듈 | 이미지 입력 수용 + visual_assessment 출력 |
| `types.ts` | `VisualAssessment` 타입 추가 |
| `NegotiationSkill` | `verify()` 이미지 오버로드 |

### 4.5 추론 강화 → 멀티턴 시나리오 시뮬레이션

**현재:** Reasoning 모드 트리거 4가지 (gap < 10%, Coach 경고 2+, BOULWARE 상대, SOFT violation 2+). DECIDE 출력 = 현재 라운드 최적 action 1개.

**강력한 추론 모델 적용 시:**

```
ProtocolDecision 확장:

interface ProtocolDecision {
  // 기존
  action: 'COUNTER' | 'ACCEPT' | 'REJECT' | 'HOLD' | 'DISCOVER' | 'CONFIRM';
  price: number;
  reasoning: string;
  tactic_used?: string;
  
  // 신규: 멀티턴 시나리오 트리
  scenario_tree?: ScenarioNode[];
}

interface ScenarioNode {
  my_offer: number;
  predicted_response: {
    price: number;
    probability: number;        // 이 응답이 올 확률
    opponent_pattern: string;   // 예측 근거
  };
  next_moves: ScenarioNode[];   // 재귀: 다음 수
  expected_settlement: number;  // 이 경로의 기대 합의가
  rounds_to_settle: number;     // 이 경로의 기대 라운드 수
}
```

**Coach 연동:**

```
Coach 개선:

현재:
  recommended_price = Faratin(t, β)  // 현재 시점 양보 곡선만

미래:
  recommended_price = weightedAvg(
    Faratin(t, β),                    // 양보 곡선 기본값
    scenario_tree.best_path.my_offer  // LLM 멀티턴 예측 최적값
  )
  
  // LLM이 "3수 앞"을 읽어주면 Coach의 추천가가 전략적으로 변함
```

| 대상 파일 | 작업 |
|----------|------|
| `types.ts` | `ScenarioNode` 타입 추가 |
| `coach.ts` | 시나리오 트리 반영 로직 |
| `config.ts` | Reasoning 트리거에 시나리오 깊이 설정 |

### 4.6 소형 모델 발전 → Stage 분산 실행

**현재:** 모든 LLM 호출이 xAI API 경유. 3회/라운드.

**소형 로컬 모델이 파싱/생성 수준 도달 시:**

```
Stage 분산 실행 구조:

                  클라이언트 (에이전트)          Haggle 서버
                  ─────────────────          ──────────────
Stage 1 UNDERSTAND  [로컬 소형 모델]     →
Stage 2 CONTEXT                          →  [코드]
Stage 3 DECIDE                           →  [고성능 모델]
Stage 4 VALIDATE                         →  [코드]
Stage 5 RESPOND     [로컬 소형 모델]     ←
Stage 6 PERSIST                          →  [코드]

API 호출: 3회/라운드 → 1회/라운드
비용: $0.00034 → ~$0.00012
지연: ~900ms → ~300ms
```

**엔진 변경:**

```
ModelAdapter 확장:

interface ModelAdapter {
  complete(prompt: string, options: CompletionOptions): Promise<LLMResponse>;
  readonly location: 'remote' | 'local';
  readonly capabilities: ('parse' | 'reason' | 'generate')[];
}

class LocalSmallAdapter implements ModelAdapter {
  location = 'local' as const;
  capabilities = ['parse', 'generate'] as const;
  // WASM/ONNX 기반 로컬 실행
}

class GrokFastAdapter implements ModelAdapter {
  location = 'remote' as const;
  capabilities = ['parse', 'reason', 'generate'] as const;
}
```

**대기업 채택 시나리오:** "파싱과 생성은 우리 소형 모델로, 핵심 협상 판단만 Haggle API를 쓰겠다" → 채택 장벽 극적 하락.

| 대상 파일 | 작업 |
|----------|------|
| `ModelAdapter` 인터페이스 | `location`, `capabilities` 추가 |
| `config.ts` | Stage별 어댑터 매핑 |
| 각 Stage 모듈 | 독립 실행 가능한 함수로 분리 |

### 4.7 에이전트 프레임워크 발전 → 엔진이 "도구"에서 "프로토콜"로

**현재:** Claude/GPT가 MCP/Actions로 Haggle을 "도구"로 호출. `create_goal` → 결과.

**에이전트 간 직접 통신 시대:**

```
현재 (에이전트 → Haggle 대행):
  구매 에이전트 → create_goal → Haggle이 전부 처리 → 결과

미래 (에이전트 ↔ 에이전트, HNP 프로토콜):
  구매 에이전트                    판매 에이전트
    Stage 1 (자체 LLM)              Stage 1 (자체 LLM)
    Stage 2 (Haggle Referee)        Stage 2 (Haggle Referee)
    Stage 3 (자체 LLM)              Stage 3 (자체 LLM)
    Stage 4 (Haggle Referee)  ←→   Stage 4 (Haggle Referee)
    Stage 5 (자체 LLM)              Stage 5 (자체 LLM)
    Stage 6 (Haggle Protocol)       Stage 6 (Haggle Protocol)

→ 양쪽 다 자기 LLM으로 UNDERSTAND/DECIDE/RESPOND
→ Referee 규칙과 Phase 전이는 HNP 프로토콜이 강제
→ Haggle = 프로토콜 인프라
```

**엔진 변경:** 6-Stage 파이프라인의 각 Stage를 독립 실행 가능한 함수로 분리.

```
// 현재: 전체 순차 실행
async function executeRound(session, message) {
  const understood = await understand(message);
  const context = assembleContext(session, understood);
  const decision = await decide(context);
  const validated = validate(decision);
  const response = await respond(validated);
  persist(session, validated, response);
}

// 목표: Stage별 독립 호출 가능
export { understand, assembleContext, decide, validate, respond, persist };

// 외부 에이전트 사용:
// "Stage 1, 3, 5는 내 LLM으로, Stage 2, 4, 6은 Haggle 것을 쓰겠다"
```

## 6. 핵심 원칙

**"엔진이 LLM을 교체 가능한 부품으로 취급하는 구조를 지금 만든다."**

Doc 27의 "Brain / Knowledge / Safety" 3대 원칙이 이 방향이지만, 실제 코드에서는 Brain이 Grok 4 Fast에 하드와이어되어 있다. ModelAdapter 인터페이스를 진짜 교체 가능하게 만들고, Stage별로 다른 모델을 쓸 수 있게 하고, 로컬 모델도 끼울 수 있게 하는 것 — 이것이 "LLM이 발전할수록 Haggle이 자동으로 좋아지는" 구조의 전제조건이다.

대기업이 Haggle을 무시할 수 없게 만드는 건 엔진의 정교함이 아니라, **대기업이 구조적으로 만들 수 없는 것**(크로스플랫폼 데이터, 양쪽 중립, 협상 과정 데이터, 외부 분쟁 해결)을 엔진에 심어놓는 것이다.
