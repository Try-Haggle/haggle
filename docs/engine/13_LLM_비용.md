## 1. LLM 모델 선택

### 1.1 선정 모델: Grok 4 Fast

| 항목 | 사양 |
|------|------|
| 모델 | Grok 4 Fast (xAI) |
| API 모델 ID | `grok-4-fast` |
| Input | $0.20 / 1M tokens |
| Output | $0.50 / 1M tokens |
| 컨텍스트 | 2M tokens |
| Structured Output | JSON mode 지원 |
| 벤치마크 | LMArena Thinking #1 (1483 Elo), Non-thinking #2 (1465 Elo) |
| Tool calling | τ²-bench Telecom #1 |

### 1.2 선정 이유

1. **가격 대비 성능 최고** — 경쟁 대비 3-60배 저렴하면서 성능은 동등 이상.
2. **2M 컨텍스트** — 장기 협상 히스토리 전체를 한 번에 처리 가능. 경쟁 모델 대비 5-16배.
3. **Tool calling 1위** — τ²-bench Telecom에서 1위. 에이전트 아키텍처에 최적.
4. **Structured Output** — JSON mode로 파싱 실패율 최소화. 파이프라인 안정성 핵심.
5. **매 라운드 LLM 호출 환경에서 비용 부담 없음** — $0.20/1M input으로 10라운드 해도 $0.005 미만.

## 2. v2.0 6-Stage LLM Native Pipeline

### 2.1 아키텍처 변경

| | v1.x (Engine-First) | v2.0 (LLM Native Pipeline) |
|---|---|---|
| 라운드 처리 | Engine Core (규칙 기반, CPU만) | **LLM 6-Stage Pipeline** |
| LLM 역할 | 전략 생성 + 에스컬레이션만 | **매 라운드 기본 경로** |
| 세션당 LLM 호출 | 1-3회 | **32회** (10라운드 기준) |
| Engine Core 역할 | 의사결정자 | Referee (심판/검증) |
| Hot Path | 95%+ (규칙 기반) | **폐기** — LLM이 기본 |

### 2.2 라운드당 LLM 호출 구조

```
초기화 (1회):
  Stage 0a: Strategy Generation [LLM]     ← 전략 수립
  Stage 0b: Term Analysis [LLM]           ← 조건 분석
  → 2 LLM 호출

매 라운드 (N회 반복):
  Stage 1: UNDERSTAND [LLM]    ← 상대 제안 해석
  Stage 2: CONTEXT [Code]      ← 메모리 + 코칭 조립 (LLM 없음)
  Stage 3: DECIDE [LLM]        ← 의사결정 + 역제안
  Stage 4: VALIDATE [Code]     ← Referee 검증 (LLM 없음)
  Stage 5: RESPOND [LLM]       ← 자연어 응답 생성
  Stage 6: PERSIST [Code]      ← 상태 저장 (LLM 없음)
  → 3 LLM 호출 / 라운드

총 LLM 호출 = 2 + (라운드 수 × 3)
```

## 3. 실측 비용 분석

### 3.1 데모 파이프라인 실측 (4라운드, Grok-4-Fast)

| 항목 | 실측값 |
|------|--------|
| LLM 호출 수 | 14회 (init 2 + 4라운드 × 3) |
| 총 토큰 | 8,284 (prompt 7,126 + completion 1,158) |
| 총 비용 | **$0.0020** |
| 라운드당 평균 토큰 | ~1,700 |
| 라운드당 평균 비용 | ~$0.0004 |

### 3.2 10라운드 세션 추산

| 항목 | 추산값 |
|------|--------|
| LLM 호출 수 | 32회 (init 2 + 10라운드 × 3) |
| 총 토큰 | ~18,500 |
| **총 비용** | **~$0.005** |

### 3.3 비용 검증

```
실측 계산:
  Input:  7,126 tokens × $0.0000002 = $0.001425
  Output: 1,158 tokens × $0.0000005 = $0.000579
  합계:   $0.002004 ← 표시된 $0.0020과 일치

xAI 공식 가격 (2026-04):
  grok-4-fast: Input $0.20/1M, Output $0.50/1M ← 코드 multiplier와 일치
```

### 3.4 토큰 최적화 기법 (구현 완료)

v2.0은 모든 라운드에서 LLM 3회를 호출하지만, 다음 최적화로 토큰을 억제한다:

| 최적화 | 구현 파일 | 효과 |
|--------|----------|------|
| **Living Memo Codec** | `memo/memo-codec.ts` | CoreMemory ~1000 tok → **~390 tok** (60% 감소) |
| **Delta 인코딩** | `adapters/grok-fast-adapter.ts:150` | 이전 라운드 대비 변경분만 전송 |
| **Phase별 토큰 예산** | `types.ts:433` | DISCOVERY 800 / BARGAINING 1500 / CLOSING 2000 |
| **히스토리 윈도우** | `stages/context.ts:46` | 최근 5라운드 fact만 전송 |
| **코칭 레벨 계층화** | `adapters/context-assembly.ts:67` | LIGHT(1줄) / STANDARD(5줄) / DETAILED |
| **예산 초과 자동 truncation** | `adapters/grok-fast-adapter.ts:59` | chars/4 추정, 초과 시 2라운드로 축소 |

#### Living Memo Codec 형식

```
NS:BARGAINING|R3/10|buyer|FULL_AUTO          ← 세션 상태
PT:85000→90000|gap:5000(5.6%)                ← 가격 궤적
CL:rec:87000|tactic:reciprocal|opp:CONCEDER  ← 코칭 지시
RM:R1:85000→92000|R2:86000→91000             ← 최근 라운드
---
SS:t:83000|f:95000|β:1.5                     ← 전략 (비공개)
OM:CONCEDER(0.20)|conv:0.72|shifts:0          ← 상대 모델 (비공개)
```

Raw JSON(~1000 tokens)을 이 형식(~390 tokens)으로 압축. 데모 파이프라인에서도 동일한 codec 사용.

#### 비용 비교

| 비교 | 세션 비용 | 비고 |
|------|-----------|------|
| v1.x (Engine-First) | ~$0.001-0.002 | LLM 1-3회만 |
| v2.0 (최적화 미적용 시) | ~$0.008+ | 추정 |
| **v2.0 (전체 최적화 적용)** | **~$0.005** | **실측** |

토큰 최적화 덕분에 매 라운드 LLM 3회를 호출해도 비용이 억제되며, 수익 구조에 영향 없음.

## 4. 단위 경제 (Unit Economics)

### 4.1 거래 1건당 비용/수익 (iPhone 14 Pro, $620)

| 항목 | 금액 | 수익 대비 비율 |
|------|------|---------------|
| **수익** | | |
| 거래 수수료 (1.5%) | **$9.30** | 100% |
| | | |
| **변동비** | | |
| LLM 비용 (10라운드) | $0.005 | 0.05% |
| CheckMEND IMEI 조회 | $0.35 | 3.8% |
| 가스비 (Base L2 에스크로) | $0.001 | 0.01% |
| 인프라 (per-txn 배분) | $0.02 | 0.2% |
| **변동비 소계** | **$0.376** | **4.0%** |
| | | |
| **기여이익** | **$8.92** | **96.0%** |

> **LLM 비용은 변동비의 1.3%에 불과.** 가장 큰 변동비는 CheckMEND ($0.35).

### 4.2 실패 거래 비용

모든 협상이 성공하지는 않는다. 실패한 거래도 LLM 비용을 소비한다.

| 거래 결과 | 평균 라운드 | LLM 비용 | 수익 |
|-----------|------------|----------|------|
| 성공 (ACCEPT) | 8-10 | $0.004-0.005 | $9.30 |
| 실패 (REJECT, 5라운드에서) | 5 | $0.003 | $0 |
| 실패 (조기 포기, 2라운드) | 2 | $0.001 | $0 |

→ **실패한 거래의 LLM 비용은 $0.001-0.003 수준으로, 성공 1건의 기여이익($8.92)이 충분히 커버.**

## 5. 전환율 기반 P&L 분석

### 5.1 전환율 가정

| 퍼널 단계 | 전환율 | 근거 |
|-----------|--------|------|
| 방문 → 가입 | 10% | C2C 마켓플레이스 평균 |
| 가입 → 리스팅/검색 | 40% | 관심 기반 전환 |
| 리스팅 → 협상 시작 | 50% | AI 자동 매칭 |
| 협상 시작 → 거래 성공 | 20% | Phase 0 보수 가정 |
| **방문 → 거래 (End-to-End)** | **0.4%** | 10% × 40% × 50% × 20% |

> **주의:** 위는 방문 기준. 가입 유저 기준 거래 전환율은 4% (40% × 50% × 20%).

### 5.2 LLM 비용의 실제 부담: 전환율 반영

가입 유저 100명 중:
- 40명이 리스팅/검색 → 이 중 20명이 협상 시작 (LLM 비용 발생)
- 20건 협상 중 4건 성공, 16건 실패

```
LLM 비용 (유저 100명 기준):
  성공 4건: 4 × $0.005 = $0.020
  실패 16건: 16 × $0.002 = $0.032 (평균 4라운드에서 실패)
  총 LLM 비용: $0.052

수익 (유저 100명 기준):
  성공 4건 × $9.30 = $37.20

LLM 비용 / 수익 = $0.052 / $37.20 = 0.14%
```

**전환율을 감안해도 LLM 비용은 수익의 0.14%.**

## 6. 손익분기점 (BEP) 분석

### 6.1 월 고정비 커버에 필요한 거래 수

```
Phase 0 초기 고정비: $60/월
거래당 기여이익: $9.30 - $0.376 = $8.92

BEP = $60 / $8.92 = 6.7건/월 → 월 7건이면 흑자
```

### 6.2 전환율 감안 BEP

```
월 7건 성공 거래 필요
협상 성공률 20% → 35건 협상 필요
리스팅/검색 전환 50% → 70건 활성 유저 필요
가입 전환 40% → 175명 가입 유저 필요

→ 가입자 175명이면 월 흑자 (Phase 0 초기)
```

## 7. 리스크 요약

### 7.1 비용 리스크 매트릭스

| 리스크 | 확률 | 영향 | 대응 |
|--------|------|------|------|
| LLM 가격 인상 | 낮음 | 낮음 | 10배 올라도 수익의 1.4% |
| CheckMEND 가격 인상 | 중간 | 중간 | 대체 API 존재 (Swappa IMEI 등) |
| 전환율 10%로 하락 | 중간 | 높음 | Gate 3 분쟁률 관리로 신뢰 구축 |
| xAI 서비스 중단 | 낮음 | 높음 | ModelAdapter 추상화로 GPT-5 Nano 전환 가능 |

### 7.2 진짜 리스크는 비용이 아니라 전환율

```
시나리오 A: 전환율 20% (기본)     → 순이익률 83.7% (Phase 0 목표)
시나리오 B: 전환율 10% (비관)     → 순이익률 68.4%
시나리오 C: 전환율 5% (최악)      → 순이익률 36.9%
시나리오 D: LLM 비용 10배 (재앙)  → 순이익률 82.3% (거의 변화 없음)
```

**결론: LLM 비용이 10배 올라도 순이익률에 1.4%p 영향. 전환율이 반으로 줄면 15%p 영향.**
→ 사업 집중점은 LLM 비용 최적화가 아니라 **전환율(=제품 품질)과 분쟁률(=신뢰)**에 있다.
