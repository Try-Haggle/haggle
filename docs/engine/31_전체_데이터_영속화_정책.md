## 0. 데이터 해자 비즈니스 배경

### 0.1 이게 해자인 이유

eBay는 최종 판매가만 안다. 과정이 없다. FB Marketplace는 메신저 대화가 비정형 텍스트라 구조화 불가. Mercari, Poshmark, OfferUp 전부 마찬가지 — 거래 결과는 있지만 "어떤 과정을 거쳐서 이 가격에 도달했는지"는 아무도 모른다.

Haggle의 6-Stage 파이프라인은 **모든 메시지가 엔진을 통과하면서 자동으로 구조화**된다. 이건 사후에 만들 수 없다. 1건부터 기록해야 100만 건째에 해자가 된다.

**경쟁자가 오늘 시작해도:** 같은 양의 데이터를 쌓으려면 같은 시간이 걸린다. 데이터는 시간을 살 수 없다.

### 0.2 데이터 축적 타임라인

| 거래 수 | 가능해지는 것 | 데이터 자산 가치 |
|--------|-------------|---------------|
| 100건 | 기본 패턴 확인 | 없음 |
| 1,000건 | 카테고리별 초기 통계 | 낮음 |
| 10,000건 | 전술 성공률 유의미, Intelligence API 베타 | 중간 |
| 100,000건 | 합의 확률 예측 모델 학습 가능 | **자산 시작** |
| 1,000,000건 | 개인화 추천, 크로스카테고리 패턴 | **복제 불가능** |

### 0.3 공개 vs 비공개 집계

**공개 집계 (Haggle Index)** — 카테고리 수준, 개인 식별 불가:

```
iPhone 15 Pro / Like New:
  평균 할인율:        11.3%
  평균 합의 라운드:    4.2
  가장 효과적인 전술:  reciprocal_concession (성사율 +18%)
  가격대별 성사율:     $800-1000 → 42%, $1000-1200 → 31%
```

용도: 분기별 퍼블릭 리포트, Intelligence API, 언론 인용, 데이터 권위 구축.

**비공개 집계 (대행 엔진 전용)** — 개인 패턴 데이터, 절대 외부 미공개:

```
판매자 X (익명 ID):
  양보 패턴:          CONCEDER (β ≈ 0.4)
  추정 마지노선:      시작가의 -15%
  조건 교환 성향:      배송비 양보에 민감
  시간대 패턴:        저녁에 더 빠르게 양보
```

용도: Coach의 L5 Signals 주입, 대행 모드 품질 향상. **이 데이터가 "Haggle 대행이 직접 하는 것보다 나은 이유"의 핵심 원천.**

## 1. 원칙

**"기록되지 않은 것은 존재하지 않는 것이다."**

1. Haggle을 통과하는 모든 데이터는 영속화한다
2. 영속화된 데이터는 변조 불가능하게 보호한다
3. 변조 불가능한 데이터는 법정 증거로 쓸 수 있게 설계한다
4. 데이터는 시간이 지날수록 가치가 올라간다 — 이것이 복제 불가능한 해자다

**결정 우선순위:** 저장 > 효율, 무결성 > 성능, 증거력 > 편의

## 3. 데이터 계층 아키텍처

모든 데이터를 3개 계층으로 분류하고 각각 다른 저장 전략을 적용한다.

```
┌─────────────────────────────────────────────────────────────┐
│  Layer 1: HOT (실시간 접근)                                    │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐       │
│  │   Redis      │  │  In-Memory   │  │  PostgreSQL  │       │
│  │  (세션 상태)  │  │  (파이프라인) │  │  (OLTP)      │       │
│  └──────────────┘  └──────┬───────┘  └──────────────┘       │
│                          │ FLUSH                             │
├──────────────────────────▼──────────────────────────────────┤
│  Layer 2: WARM (분석/집계)                                    │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐       │
│  │  PostgreSQL  │  │  Materialized│  │  S3 (media)  │       │
│  │  (append-    │  │  Views       │  │  (encrypted) │       │
│  │   only log)  │  │  (집계 통계)  │  │              │       │
│  └──────────────┘  └──────────────┘  └──────────────┘       │
│                                                              │
├──────────────────────────────────────────────────────────────┤
│  Layer 3: COLD (증거/감사/해자)                                │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐       │
│  │  Base L2     │  │  S3 Glacier  │  │  PostgreSQL  │       │
│  │  (온체인 해시)│  │  (장기 보관)  │  │  (판례 DB)   │       │
│  └──────────────┘  └──────────────┘  └──────────────┘       │
└─────────────────────────────────────────────────────────────┘
```

### 3.1 Layer 1 → Layer 2 FLUSH 규칙

**핵심: 인메모리 데이터를 DB로 내리는 시점과 방법**

| 인메모리 데이터 | FLUSH 트리거 | 저장 대상 | 형식 |
|---------------|------------|---------|------|
| RoundFact | 매 라운드 Stage 6 완료 시 | `negotiation_round_facts` (신규) | JSONB |
| OpponentPattern | 매 라운드 (갱신 시) | `negotiation_sessions.opponent_model` (신규) | JSONB |
| CoreMemory | 매 라운드 + 세션 종료 시 | `negotiation_sessions.core_memory_snapshot` (신규) | 압축 JSONB |
| Checkpoint | Phase 전이 시 | `negotiation_checkpoints` (신규) | JSONB |
| LLM 텔레메트리 | 매 LLM 호출 완료 시 | `llm_telemetry` (신규) | 구조화 row |
| 인증 서명 (VL) | 인증 완료 시 | `negotiation_verifications` (신규) | row + signature |
| EscalationRequest | 에스컬레이션 발생 시 | `negotiation_escalations` (신규) | JSONB |

## 4. 신규 테이블 설계

### 4.1 `negotiation_round_facts` — 라운드별 전체 과정 기록

Doc 30의 핵심. `SessionMemoryStore.RoundFact`를 영속화.

```sql
CREATE TABLE negotiation_round_facts (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  session_id UUID NOT NULL,
  round_no INTEGER NOT NULL,

  -- 가격
  buyer_offer NUMERIC(18,0),
  seller_offer NUMERIC(18,0),
  gap NUMERIC(18,0),

  -- 전술
  buyer_tactic TEXT,
  seller_tactic TEXT,

  -- 조건
  conditions_changed JSONB,           -- [{term, old_value, new_value, who}]

  -- 코칭
  coaching_recommended_price NUMERIC(18,0),
  coaching_recommended_tactic TEXT,
  coaching_followed BOOLEAN,

  -- 개입
  human_intervened BOOLEAN DEFAULT false,

  -- Phase
  phase TEXT,

  -- 해시 (변조 방지)
  fact_hash TEXT NOT NULL,            -- sha256(canonical JSON of this row)
  prev_fact_hash TEXT,                -- 이전 라운드의 fact_hash → 체인

  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE UNIQUE INDEX ON negotiation_round_facts(session_id, round_no);
CREATE INDEX ON negotiation_round_facts(session_id);
```

**해시 체인**: 각 row의 `fact_hash = sha256(session_id + round_no + buyer_offer + seller_offer + ... + prev_fact_hash)`. 이전 라운드의 해시를 포함하므로 중간 라운드를 변조하면 체인이 깨진다.

### 4.2 `negotiation_verifications` — 인증/검증 기록

라이브 협상 중 발생하는 모든 검증 결과.

```sql
CREATE TABLE negotiation_verifications (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  session_id UUID NOT NULL,
  round_no INTEGER NOT NULL,

  term TEXT NOT NULL,                  -- 'imei', 'battery_health', 'find_my'
  result TEXT NOT NULL,                -- 'CLEAN', 'BLACKLISTED', 'VERIFIED', 'FAILED'
  detail JSONB,                        -- 검증 상세 (carrier, unlock status 등)
  provider TEXT,                       -- 'checkmend', 'carrier_api', 'self_report'
  cost_minor INTEGER DEFAULT 0,        -- 검증 비용

  -- 서명 (법적 증거력)
  attestation_signature TEXT NOT NULL, -- sign(platform_key, hash(payload))
  attestation_payload_hash TEXT NOT NULL, -- sha256(session_id + round + term + result + timestamp)

  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX ON negotiation_verifications(session_id);
CREATE INDEX ON negotiation_verifications(term, result);
```

### 4.3 `negotiation_escalations` — 에스컬레이션 이력

```sql
CREATE TABLE negotiation_escalations (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  session_id UUID NOT NULL,
  round_no INTEGER NOT NULL,

  type TEXT NOT NULL,                  -- 'UNKNOWN_PROPOSAL', 'STRATEGY_REVIEW', 'HUMAN_APPROVAL_REQUIRED'

  strategy_snapshot JSONB,             -- MasterStrategy at escalation time
  recent_rounds JSONB,                 -- 최근 라운드들

  -- 결과
  resolution TEXT,                     -- 'APPROVED', 'REJECTED', 'MODIFIED', 'TIMEOUT'
  resolution_detail JSONB,
  resolved_at TIMESTAMPTZ,

  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX ON negotiation_escalations(session_id);
```

### 4.4 `negotiation_checkpoints` — Phase 복원점

```sql
CREATE TABLE negotiation_checkpoints (
```

## 5. 변조 방지 — 증거 체인 설계

### 5.1 3단계 무결성 보장

```
Level 1: Row-level Hash
  각 레코드에 sha256 해시 저장 (self-contained 검증)

Level 2: Chain Hash
  이전 레코드의 해시를 포함 → 순서 보장 + 중간 변조 감지

Level 3: On-chain Anchor
  세션 종료 시 최종 chain hash를 Base L2에 기록 → 타임스탬프 + 불변성
```

### 5.2 해시 체인 상세

```
Round 1: fact_hash_1 = sha256(session_id + round_1_data + "GENESIS")
Round 2: fact_hash_2 = sha256(session_id + round_2_data + fact_hash_1)
Round 3: fact_hash_3 = sha256(session_id + round_3_data + fact_hash_2)
...
Session End: session_fact_chain_hash = fact_hash_N (마지막 라운드)
             → HaggleDisputeRegistry.anchorDispute(evidenceRootHash: session_fact_chain_hash)
```

**검증**: 임의의 라운드 N을 변조하면 fact_hash_N이 바뀌고, fact_hash_N+1도 바뀌고, ... 최종 chain hash가 온체인과 불일치 → 변조 증명.

### 5.3 온체인 앵커링

| 이벤트 | 앵커 시점 | 저장 대상 |
|--------|---------|---------|
| 세션 종료 (DEAL) | ACCEPTED 전이 직후 | `session_fact_chain_hash` → `HaggleDisputeRegistry` |
| 세션 종료 (REJECT/TIMEOUT) | 종료 전이 직후 | 동일 |
| Attestation 커밋 | 판매자 출하 전 | `commitHash` → `HaggleDisputeRegistry` |
| 분쟁 판결 | DS 패널 투표 완료 | `evidenceRootHash + resolutionHash` → 이미 구현됨 |

### 5.4 법정 증거 요건 체크리스트

| 요건 | 현재 | 이 설계 후 |
|------|------|---------|
| 변조 불가능성 (Integrity) | ❌ | ✅ 해시 체인 + 온체인 앵커 |
| 시점 증명 (Timestamp) | 🟡 DB timestamp만 | ✅ 온체인 block timestamp |
| 출처 증명 (Authenticity) | 🟡 EIP-712 서명 | ✅ 서명 + 앵커 조합 |
| 보관 연속성 (Chain of Custody) | ❌ | ✅ append-only + 해시 체인 |
| 완전성 (Completeness) | ❌ 과정 소멸 | ✅ 전수 저장 |
| 재현 가능성 (Reproducibility) | ❌ | ✅ canonical payload + 해시 검증 |

## 6. 암호화 정책

### 6.1 저장 시 암호화 (Encryption at Rest)

| 데이터 분류 | 암호화 방식 | 키 관리 |
|-----------|-----------|--------|
| PII (IMEI, 연락처) | AES-256-GCM (필드 레벨) | AWS KMS / Vault |
| 미디어 (사진, 영상) | S3 SSE-S3 (버킷 레벨) | AWS 관리 |
| 일반 거래 데이터 | PostgreSQL TDE (디스크 레벨) | DB 레벨 |
| 온체인 데이터 | 공개 (해시만, 원본 아님) | N/A |

### 6.2 전송 시 암호화 (Encryption in Transit)

- 모든 API: TLS 1.3 필수
- 내부 서비스 간: mTLS
- 온체인 트랜잭션: EIP-712 서명

### 6.3 개인정보와 해자의 분리

```
원본 데이터 (PII 포함)
  │
  ├── 암호화 저장 (Layer 2, 보존 기간 제한)
  │     → GDPR 삭제 요청 시 키 폐기로 대응
  │
  └── 익명화 집계 (해자 데이터)
        → 영구 보존, 개인 식별 불가
        → 카테고리별 통계, 전술 성공률, 패턴 분류
        → 삭제 불필요 (개인정보 아님)
```

## 7. 압축 정책

### 7.1 저장 압축

| 데이터 유형 | 압축 방식 | 예상 절감 |
|-----------|---------|---------|
| JSONB 필드 (큰 것) | PostgreSQL TOAST (자동) | 60-75% |
| CoreMemory snapshot | Living Memo Codec 적용 후 저장 | 61% (Doc 26 측정) |
| 미디어 | WebP (사진), H.265 (영상) | 50-70% |
| 장기 보관 (Cold) | S3 Glacier + gzip | 80%+ |

### 7.2 집계 압축 — 원본 vs 요약

| 보존 기간 | 저장 내용 | 예시 |
|----------|---------|------|
| 0-90일 | **원본 전체** | 모든 RoundFact, 전체 CoreMemory |
| 90일-1년 | **원본 + 요약** | 원본 유지 + SessionSummary 생성 |
| 1년-3년 | **요약만** (원본은 Cold 이동) | SessionSummary + 해시 체인 |
| 3년+ | **익명 집계만** (원본 삭제 가능) | 카테고리 통계, 전술 효과 |

**예외**: 분쟁이 발생한 세션은 원본 **영구 보존** (판례 DB의 일부).

## 8. 보존 정책

### 8.1 데이터 유형별 보존 기간

| 데이터 | 최소 보존 | 최대 보존 | 근거 |
|--------|---------|---------|------|
| 거래 원본 (라운드, 세션) | 3년 | 7년 | 미국 상거래법 + 해자 |
| 분쟁 관련 원본 | 영구 | 영구 | 판례 DB = 해자 핵심 |
| 미디어 (사진/영상) | 90일 | 1년* | 스토리지 비용 (*분쟁 시 영구) |
| LLM 텔레메트리 | 1년 | 3년 | 비용 최적화 분석 |
| 온체인 앵커 | 영구 | 영구 | 블록체인 특성 |
| 익명 집계 통계 | 영구 | 영구 | 해자 (개인정보 아님) |
| PII (IMEI, 연락처) | 거래 완료 +90일 | 1년 | GDPR/CCPA |

### 8.2 삭제 프로토콜

```
사용자 삭제 요청 시:
  1. PII 암호화 키 폐기 → 원본 복호화 불가
  2. 익명 집계에는 영향 없음 (이미 비식별)
  3. 해시 체인은 유지 (해시만으로 개인 식별 불가)
  4. 온체인 앵커는 삭제 불가 (블록체인 특성) — 해시만 존재, 원본 없으면 무의미
```
