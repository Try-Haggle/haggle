# 24. 남용 탐지 정책

**문서:** Haggle Engine Architecture v1.0.2 — Abuse Detection Policy
**범위:** 정상 사용자 동시 유입은 허용하면서, 전략 추출형 probing과 자동화 남용을 탐지하는 신호와 대응 계층

## 1. 목표

Haggle의 기본 보호는 `per-principal` 동시 세션 및 반복 시도 제한이다.  
하지만 공개 링크 트래픽이 커질수록, 정상적인 다중 유입과 남용성 자동화를 구분하는 별도 계층이 필요하다.

이 문서의 목표는 아래 두 가지다.

1. 정상 사용자의 동시 진입과 협상은 가능한 한 방해하지 않는다.
2. probing, session spray, 다중 가면 우회, 대량 자동화만 골라서 늦추거나 차단한다.

---

## 2. 기본 원칙

1. `listing-global cap` 같은 단순 고정 상한은 기본 정책으로 두지 않는다.
2. 대신 행동 신호를 점수화해 `정상`, `주의`, `의심`, `고위험` 으로 나눈다.
3. 탐지 결과는 즉시 차단만 하지 않고, 관찰 -> 감속 -> 제한 -> 차단 순서로 대응한다.
4. 단일 신호보다 복합 신호를 우선한다.

---

## 3. 주요 남용 시나리오

### 3.1 세션 스프레이

- 짧은 시간 안에 동일 listing 또는 동일 seller에 대해 세션을 대량 생성
- principal, alias, credential을 돌려가며 동시 진입

### 3.2 전략 probing

- 가격을 조금씩만 바꿔 seller counter 경계를 반복 측정
- 규칙적인 increment/decrement로 reserve band를 추정

### 3.3 세션 방치

- 세션은 많이 열지만 실제 offer, complete, cancel 비율이 낮음
- active session만 점유하고 응답은 거의 없음

### 3.4 다중 가면 우회

- 같은 네트워크/브라우저 계열인데 principal만 반복 변경
- 인증 없는 익명 유입을 여러 개로 분산

### 3.5 고속 자동화

- 사람 반응 속도로 보기 어려운 간격으로 create/offer/cancel 반복
- round 간 시간 간격이 지나치게 짧고 규칙적

### 3.6 전역 seller 스캔

- 한 seller의 여러 listing에 대해 동시에 비슷한 패턴으로 진입
- seller의 전역 협상 정책을 역추론하려는 패턴

---

## 4. 핵심 탐지 신호

| 신호 | 의미 | 기본 해석 |
|------|------|----------|
| `sessions_per_listing_per_principal` | 동일 listing에서 principal별 세션 생성량 | probing 또는 무의미한 재시도 |
| `sessions_per_seller_per_principal` | 동일 seller 전체에서 세션 생성량 | seller 스캔 가능성 |
| `time_to_first_offer` | 세션 생성 후 첫 제안까지 시간 | 너무 짧거나 너무 길면 자동화/방치 신호 |
| `offers_per_session` | 세션당 제안 수 | 너무 낮으면 spray, 너무 높으면 probing 가능성 |
| `cancel_rate` | 생성 대비 취소 비율 | 실구매 의도 부족 가능성 |
| `distinct_aliases_per_principal` | principal당 별칭 수 | body alias 교체를 통한 탐색 시도 |
| `distinct_principals_per_network_fingerprint` | 동일 네트워크/UA에서 principal 수 | 다중 가면 우회 |
| `median_round_interval_seconds` | 라운드 간 시간 간격 | 지나치게 짧고 일정하면 자동화 |
| `price_probe_step_pattern` | 제안 가격 변화 패턴 | 규칙적 격자 탐색 여부 |
| `read_to_write_ratio` | 상태 조회 대비 실제 제안/완료 비율 | 텔레메트리 수집형 탐색 |

---

## 5. 의심 행동 분류

### 5.1 낮은 위험

- 동일 listing에서 몇 번 협상
- 제안 간격이 자연스럽고 불규칙
- 최종 완료 또는 자연스러운 취소 비율이 높음

### 5.2 중간 위험

- 짧은 시간 안에 반복 진입
- 비슷한 가격대만 계속 시험
- 상태 조회 비율이 비정상적으로 높음

### 5.3 높은 위험

- 다중 principal 회전
- 규칙적인 price probing
- seller 여러 listing 동시 스캔
- 링크 랜딩 없이 협상 endpoint만 직접 때리는 패턴

---

## 6. 권고 대응 단계

### 단계 A: 관찰

- 로그 적재
- principal / listing / seller / network fingerprint 수준의 signal snapshot 저장

### 단계 B: 감속

- 응답 지연 추가
- state polling 응답 빈도 제한
- auto-negotiate나 buyer-engine step 호출 속도 낮춤

### 단계 C: 제한

- principal별 동시 세션 제한 강화
- 일정 기간 create/offer 제한
- detail telemetry 강등 (`full` -> `summary`)

### 단계 D: 차단

- 일정 시간 세션 생성 차단
- seller 단위 보호 모드
- API key 또는 agent identity 정지

---

## 7. 오탐 방지 원칙

1. 단일 지표만으로 바로 차단하지 않는다.
2. 바이럴 링크 유입, 공동구매, 라이브 커머스 상황은 별도 예외 정책을 둔다.
3. 정상 구매 전환율이 높은 principal은 더 보수적으로 차단한다.
4. seller가 직접 허용한 partner agent는 완화된 정책을 적용할 수 있다.

---

## 8. 구현 우선순위

1. `signal logging` 추가
2. principal / network fingerprint / listing 축 집계
3. `risk score` 계산
4. low-impact 대응부터 적용
5. 운영 데이터 기반 threshold 조정

---

## 9. 최종 판단

Haggle의 기본 보호는 `같은 principal의 반복/병렬 시도 제한` 이다.  
그러나 공개 링크 기반 서비스가 커질수록, 진짜 필요한 것은 전체 상한보다 `행동 기반 남용 탐지` 이다.

따라서 권고 기준은 아래와 같다.

1. 기본 정책은 `per-principal` 로 유지
2. `listing-global cap` 은 기본 비활성
3. 남용 방어는 별도의 `abuse detection` 계층에서 수행
4. 정상 대량 유입보다 `탐색 패턴` 자체를 탐지 대상으로 삼는다
