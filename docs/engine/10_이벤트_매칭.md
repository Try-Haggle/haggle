## 1. 대기 의도 (WaitingIntent)

사용자가 특정 상품을 찾고 있지만 아직 적절한 상대방이 존재하지 않을 때, 시스템은 **대기 의도(WaitingIntent)**를 저장한다. 이후 매칭 조건을 충족하는 상대방이 등장하면 이벤트 기반으로 자동 매칭이 이루어진다.

### 1.1 WaitingIntent Protobuf 정의

```protobuf
message WaitingIntent {
  string intent_id = 1;
  string user_id = 2;
  string role = 3;                       // "BUYER" 또는 "SELLER"
  string category = 4;
  repeated string keywords = 5;
  MasterStrategy strategy = 6;
  float min_u_total = 7;                 // 이 이상이어야 세션 시작 (기본 0.3)
  int32 max_active_sessions = 8;         // 동시 활성 세션 상한 (기본 5)
  google.protobuf.Timestamp created_at = 9;
  google.protobuf.Timestamp expires_at = 10;
  enum Status {
    ACTIVE = 0;
    MATCHED = 1;
    FULFILLED = 2;
    EXPIRED = 3;
    CANCELLED = 4;
  }
  Status status = 11;
}
```

### 1.2 필드 설명

| 필드 | 타입 | 설명 |
|------|------|------|
| `intent_id` | string | 대기 의도 고유 식별자 |
| `user_id` | string | 의도를 등록한 사용자 ID |
| `role` | string | `"BUYER"` 또는 `"SELLER"` |
| `category` | string | 상품 카테고리 (예: `"electronics"`, `"fashion"`) |
| `keywords` | string[] | 검색 키워드 목록 |
| `strategy` | MasterStrategy | 사전 생성된 협상 전략 (LLM 호출로 생성) |
| `min_u_total` | float | 매칭 최소 효용값 기준. 이 값 이상이어야 세션 생성 (기본 `0.3`) |
| `max_active_sessions` | int32 | 동시에 활성화할 수 있는 세션 상한 (기본 `5`) |
| `created_at` | Timestamp | 의도 생성 시각 |
| `expires_at` | Timestamp | 의도 만료 시각 |
| `status` | Status | 현재 상태 |

### 1.3 Status 생명주기

```
ACTIVE ──→ MATCHED ──→ FULFILLED
  │
  ├──→ EXPIRED
  └──→ CANCELLED
```

- **ACTIVE**: 매칭 대기 중. 매칭 엔진이 주기적으로 조회한다.
- **MATCHED**: 상대방과 매칭 성공. 세션이 생성되었으나 협상 미완료.
- **FULFILLED**: 협상이 성공적으로 완료되어 거래가 체결됨.
- **EXPIRED**: `expires_at`을 경과하여 자동 만료.
- **CANCELLED**: 사용자가 수동으로 취소.

## 2. 이벤트 흐름

매칭 시스템은 이벤트 드리븐 방식으로 동작한다. 새로운 리스팅이나 구매 의도가 등록될 때 매칭 엔진이 트리거되며, Engine Core의 효용 계산은 LLM 호출 없이 순수 수학으로 처리된다.

### 2.1 시나리오 1: 구매자 먼저, 판매자 나중

구매자가 먼저 의도를 등록하고, 이후 판매자가 리스팅을 올리는 경우.

```
Day 1                                    Day 3
─────                                    ─────

구매자 등록                              판매자 리스팅 등록
    │                                        │
    ▼                                        ▼
   LLM                                  new_listing 이벤트
    │                                        │
    ▼                                        ▼
MasterStrategy 생성                     매칭 엔진 트리거
    │                                        │
    ▼                                        ├─① 해당 카테고리 ACTIVE
WaitingIntent                            │    구매자 Intent 조회 → 발견
  (ACTIVE)                               │
  저장                                    ├─② Engine Core 1회 호출 (200μs)
                                         │    구매자 전략 + 리스팅 데이터
                                         │
                                         ├─③ U_total = 0.72 > min_u_total(0.3)
                                         │    → 매칭 성공
                                         │
                                         ├─④ 세션 자동 생성
                                         │    → 협상 시작
                                         │
                                         └─⑤ LLM 호출: 0회
                                              (전략이 이미 존재)
```

**핵심 포인트:** 구매자의 `MasterStrategy`는 Day 1에 LLM으로 이미 생성되어 있으므로, Day 3 매칭 시점에서 추가 LLM 호출이 필요 없다. Engine Core의 효용 계산만으로 매칭을 판단한다 (200μs).

### 2.2 시나리오 2: 판매자 먼저, 구매자 나중

판매자가 먼저 의도를 등록하고, 이후 구매자가 등록하는 경우. **양방향 매칭**이 필요하다.

```
판매자 의도 등록                         구매자 등록
    │                                        │
    ▼                                        ▼
MasterStrategy 생성                     MasterStrategy 생성
    │                                        │
    ▼                                        ▼
WaitingIntent                            매칭 엔진 트리거
  (ACTIVE)                                   │
  저장                                       ├─① ACTIVE 판매자 Intent 조회 → 발견
                                             │
                                             ├─② 구매자 전략으로 리스팅 평가
                                             │    → U_total (구매자 관점)
                                             │
                                             ├─③ 판매자 전략으로 구매자 조건 평가
                                             │    → U_total (판매자 관점)
                                             │
                                             ├─④ 양쪽 모두 min_u_total 초과
                                             │    → 양방향 매칭 성공
                                             │
                                             └─⑤ 세션 생성 → 즉시 협상 시작
```

**핵심 포인트:** 판매자 먼저 시나리오에서는 양방향 효용 평가가 필요하다. 구매자 관점의 `U_total`과 판매자 관점의 `U_total` 모두 `min_u_total`을 초과해야 매칭이 성립한다.

### 2.3 시나리오 3: 한 리스팅에 대기 구매자 여러 명

복수의 구매자가 대기 중일 때 새 리스팅이 등록되는 경우. **동시 다중 세션**이 생성될 수 있다.

```
대기 중인 구매자들:
  ┌─ Buyer A: budget $800, Fox 스타일
  ├─ Buyer B: budget $850, Tiger 스타일
  └─ Buyer C: budget $750, Dolphin 스타일

새 리스팅 등록: $780
    │
    ▼
매칭 엔진 트리거
    │
    ▼
Engine Core × 3 호출 (총 600μs)
    │
    ├─ Buyer A: U_total = 0.72 ≥ 0.3 → ✅ 매칭
    ├─ Buyer B: U_total = 0.85 ≥ 0.3 → ✅ 매칭
    └─ Buyer C: U_total = 0.35 ≥ 0.3 → ✅ 매칭

결과: 3개 세션 동시 생성
      판매자 관점: N:1 토폴로지 자동 감지
```

**핵심 포인트:**
- `batchEvaluate`를 통해 3명의 구매자를 일괄 평가한다 (총 600μs).
- 모든 구매자의 `U_total`이 `min_u_total(0.3)` 이상이므로 3개 세션이 동시에 생성된다.
- 판매자 측에서 **N:1 토폴로지**가 자동 감지되며, `compareSessions`를 통해 BATNA(최선의 대안)가 실시간으로 업데이트된다.
- 판매자는 복수의 구매자와 동시에 협상하면서 더 유리한 조건을 확보할 수 있다.

## 3. 매칭 트리거

시스템에서 발생하는 다양한 이벤트가 매칭 엔진을 트리거한다. 각 이벤트 유형별 처리 방식은 다음과 같다.

| 이벤트 | 방향 | 처리 |
|--------|------|------|
| 새 리스팅 등록 | 리스팅 → 대기 구매자들 | 해당 카테고리 ACTIVE Intent 조회 → 일괄 평가 |
| 새 구매 의도 등록 | 구매 의도 → 기존 리스팅들 | 해당 카테고리 리스팅 조회 → 일괄 평가 |
| 리스팅 가격 변경 | 가격 변경 → 대기 구매자 재평가 | 변경된 가격으로 재평가 |
| 리스팅 삭제/판매완료 | → 관련 세션 SUPERSEDED | 활성 세션 정리 |
| 전략 변경 | → 기존 리스팅 재평가 | 새 전략으로 일괄 재평가 |

### 3.1 새 리스팅 등록

```
new_listing 이벤트
    │
    ├─ 카테고리별 ACTIVE WaitingIntent 조회
    ├─ 키워드 매칭 (선택적 필터링)
    ├─ Engine Core batchEvaluate 호출
    ├─ U_total ≥ min_u_total인 Intent 필터링
    ├─ max_active_sessions 미달인 Intent만 선택
    └─ 세션 생성 + Intent status 업데이트
```

### 3.2 새 구매 의도 등록

```
new_buyer_intent 이벤트
    │
    ├─ 카테고리별 기존 리스팅 조회
    ├─ 카테고리별 ACTIVE 판매자 Intent 조회
    ├─ Engine Core batchEvaluate 호출
    ├─ U_total ≥ min_u_total인 리스팅 필터링
    └─ 세션 생성 (max_active_sessions 제한 내)
```

### 3.3 리스팅 가격 변경

가격 변경은 기존에 매칭되지 않았던 대기 구매자에게 새로운 기회가 될 수 있다. 매칭 엔진은 변경된 가격으로 모든 관련 ACTIVE Intent를 재평가한다.

### 3.4 리스팅 삭제/판매완료

리스팅이 삭제되거나 판매가 완료되면 해당 리스팅과 연관된 모든 활성 세션의 상태가 `SUPERSEDED`로 전이된다. 이를 통해 더 이상 유효하지 않은 협상이 자동으로 정리된다.

### 3.5 전략 변경

사용자가 협상 전략을 수정하면, 해당 사용자의 모든 ACTIVE WaitingIntent에 새 전략이 반영된다. 매칭 엔진은 새 전략을 기반으로 기존 리스팅들을 재평가하여, 이전에는 매칭되지 않았던 조합이 새로 매칭될 수 있다.
