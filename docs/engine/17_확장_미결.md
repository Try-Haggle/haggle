## 1. 확장 포인트

### 1.1 V_m: 시장 컨텍스트 (예약, 미구현)

5번째 효용 차원 **V_m**(Market Context)은 향후 사용을 위해 예약되어 있으며, v1.0.0의 일부가 **아닙니다**.

현재 엔진은 V_p(가격), V_t(시간), V_f(공정성), V_r(리스크)의 4차원 효용 공간에서 동작합니다. V_m은 시장 가격 변동성, 수요-공급 비율, 카테고리별 계절성 등 외부 시장 데이터를 효용 계산에 반영하기 위한 차원입니다.

**포함 기준 (모두 충족 시에만 도입):**

1. **V_p, V_t로부터의 독립성 공식 증명** — V_m이 기존 차원과 상관계수 |ρ| < 0.3임을 통계적으로 증명해야 합니다. 기존 차원으로 대체 가능한 정보를 중복 반영하면 가중치 해석이 불가능해집니다.

2. **결정론적 데이터 신뢰도 임계값** — V_m 계산에 사용되는 시장 데이터의 신뢰도가 정의된 임계값 이상이어야 합니다. 불완전한 시장 데이터로 인한 비결정론적 결과를 방지하기 위함입니다.

3. **1,000건 시뮬레이션에서 파라미터 조정 이상의 결과 개선 입증** — 단순히 기존 w_i 가중치를 조정하는 것 대비, V_m 도입이 통계적으로 유의미한 협상 결과 개선(p < 0.05)을 보여야 합니다. 파라미터 튜닝만으로 동일한 효과를 낼 수 있다면 복잡도 증가가 정당화되지 않습니다.

4. **Protobuf 하위 호환 확장** — HNP 프로토콜의 Protobuf 메시지에 V_m 필드를 추가할 때, 기존 클라이언트와의 하위 호환성이 보장되어야 합니다. `optional` 필드로 추가하고, V_m이 없는 메시지는 기존 4차원 로직으로 폴백합니다.

### 1.2 EvoEngine 통합 (예약)

자기 진화 협상 지능(EvoEngine)은 **Skill Layer**에서 운용됩니다. Engine Core의 수학 공식을 변경하지 않습니다.

EvoEngine은 거래 이력으로부터 패턴을 학습하여 전략 파라미터를 자동 최적화하는 모듈입니다. 핵심 원칙은 다음과 같습니다:

- **Engine Core 불변성** — `computeUtility`, `makeDecision`, `computeCounterOffer` 등의 수학 공식은 EvoEngine 도입 후에도 동일하게 유지됩니다.
- **Skill Layer 운용** — EvoEngine은 Engine Core 위의 Skill Layer에서 `MasterStrategy`의 가중치(w_i), 양보 곡선 파라미터(β), 임계값(T_accept 등)을 조정합니다.
- **입력만 변경, 공식은 불변** — EvoEngine이 변경하는 것은 Engine Core에 전달되는 *입력 파라미터*이지, Engine Core 내부의 *계산 로직*이 아닙니다.

### 1.3 AgentStats 확장 (v1.0.1 예약)

향후 스탯 체계 확장을 위해 예약된 항목들입니다.

#### 9번째 스탯 후보: Creativity (창의력)

번들 제안, 조건부 제안 등 창의적인 제안을 생성하는 능력을 나타내는 스탯입니다. 현재는 LLM 에스컬레이션에 의존하는 제안 생성을 규칙 기반으로 보조하여, LLM 호출 비용을 절감하면서도 다양한 제안 전략을 구사할 수 있도록 합니다.

- **높은 Creativity** → 번들 제안, 보너스 아이템 추가, 결제 조건 변경 등 다양한 역제안 생성
- **낮은 Creativity** → 단순 가격 조정 위주의 역제안

#### 동적 스탯 조정 (EvoStats)

거래 이력에 따라 스탯이 자동 조정되는 시스템입니다. 에이전트의 승리 패턴을 학습하여 강점을 강화하고 약점을 보완합니다.

- 최근 N건의 거래 결과를 분석하여 각 스탯의 기여도를 평가
- 양(+) 기여 스탯은 점진적으로 상향, 음(-) 기여 스탯은 하향
- 변동 폭 제한(±0.5/주기)으로 급격한 변화 방지
- 사용자 승인 모드: 자동 조정 전 사용자에게 변경 사항 확인 요청

#### 스탯 시각화

사용자 대시보드에서 스탯 관련 정보를 시각적으로 표시합니다.

- **스탯 분포 차트** — 레이더 차트로 8개 스탯의 균형 표시
- **프리셋 비교** — 여러 프리셋 간 스탯 차이를 오버레이하여 비교
- **시뮬레이션 결과** — 스탯 변경 시 예상 협상 결과 변화를 미리보기

### 1.4 OpponentModel 확장 (v1.0.2 예약)

현재 `OpponentModel`은 가격 이동만 추적합니다(`classifyMove` → `CONCESSION` / `SELFISH` / `SILENT`). 향후 다음과 같은 확장이 계획되어 있습니다.

#### 다차원 상대 모델링

가격 외에 추가적인 행동 신호를 추적하여 상대의 전략 유형을 분류합니다.

| 신호 | 설명 | 활용 |
|------|------|------|
| 응답 시간 | 빠른 응답 vs 지연 응답 | 긴급도/관심도 추정 |
| 메시지 길이 | 짧은 응답 vs 상세 설명 | 협상 의지 추정 |
| 감정 패턴 | 긍정/부정/중립 어조 변화 | 만족도/불만 감지 |
| 역제안 패턴 | 고정값 vs 변동값 | 전략 유형 분류 |

이를 통해 상대를 **강경형(Hardliner)**, **협력형(Collaborator)**, **탐색형(Explorer)**, **급매형(Urgent)** 등으로 분류하고, 분류 결과에 따라 양보 전략을 조정할 수 있습니다.

#### 적응형 EMA 팩터

현재 고정된 EMA(Exponential Moving Average) 팩터 α를 협상 진행 상황에 따라 동적으로 조정합니다.

- **초반 (라운드 1-3):** α를 높게 설정 (예: 0.5) → 빠른 학습, 상대 패턴을 신속히 파악
- **안정화 후 (라운드 4+):** α를 낮게 설정 (예: 0.2) → 노이즈 필터링, 일시적 변동에 과민 반응 방지
- **급변 감지 시:** α를 일시적으로 높여 새로운 패턴에 빠르게 적응

#### Dynamic Beta 자동 튜닝

`computeDynamicBeta`의 핵심 파라미터 κ(경쟁자 민감도), λ(양보율 민감도)를 거래 이력 기반으로 최적화하는 메타러닝 시스템입니다.

- 카테고리별 최적 κ, λ 값을 학습 (전자기기 vs 의류 vs 차량 등)
- 베이지안 최적화로 파라미터 공간 탐색
- A/B 테스트 프레임워크와 통합하여 실제 거래에서 검증

## 2. 미결 사항

아래 10개 항목은 현재 결정이 보류된 사항으로, 추가 데이터 수집과 논의가 필요합니다.

### 2-1. V_p 곡률

`ln(|차이|+1)` 공식의 오목성(concavity)이 P2P 거래에 최적인가? 로그 함수는 가격 차이가 클수록 한계 효용이 감소하는 특성을 가지지만, 모든 거래 카테고리에서 이 특성이 바람직한지 검증되지 않았습니다.

**대안:** 조정 가능한 γ를 가진 거듭제곱 함수 `|차이|^γ`. γ < 1이면 오목(현재와 유사), γ = 1이면 선형, γ > 1이면 볼록. 카테고리별 γ 최적값을 A/B 테스트로 결정해야 합니다.

### 2-2. 분쟁 패널티 크기

`P_dispute = 패소당 -0.3`의 실증적 근거가 필요합니다. 현재 값은 직관적 설정이며, 실제 분쟁 데이터를 기반으로 검증되지 않았습니다.

**결정 필요:** 표준 고정값으로 유지할 것인지, 구현체(플랫폼)가 설정 가능하도록 할 것인지. 패널티가 너무 작으면 억제 효과가 없고, 너무 크면 정당한 분쟁 제기를 위축시킵니다.

### 2-3. 에스컬레이션 전략 변경 폭

LLM이 제안한 w_i 변경에 대한 **±0.15 제한**이 적절한가?

- **너무 보수적인 경우:** 에스컬레이션의 의미가 퇴색합니다. LLM이 유의미한 전략 전환을 제안해도 실행되지 못합니다.
- **너무 공격적인 경우:** 사용자가 설정한 원래 전략 의도를 크게 벗어나는 결과를 초래할 수 있습니다.
- **검토 필요:** 카테고리별/라운드별 차등 적용, 사용자 승인 기반 상한 완화 등의 대안.

### 2-4. max_active_sessions 기본값

기본값 **5개**가 적절한가? 카테고리별 차등 적용 필요 여부를 검토해야 합니다.

| 카테고리 | 제안 기본값 | 근거 |
|----------|-------------|------|
| 전자기기 | 5 | 비교 쇼핑이 일반적 |
| 차량 | 3 | 고가, 집중 협상 필요 |
| 의류 | 10 | 저가, 다수 동시 탐색 |

동시 세션이 너무 많으면 상대방 입장에서 성의 없는 협상으로 인식될 수 있고, 너무 적으면 구매자의 BATNA 확보가 어려워집니다.

### 2-5. Master Strategy 만료

생성 후 **7일**이 지난 전략이 여전히 유효한가? 시장 상황, 재고 변동, 경쟁 환경이 변할 수 있습니다.

**필요 정책:**
- 자동 만료 + 재생성 트리거 조건 정의
- 만료 경고 → 사용자에게 갱신 또는 폐기 선택 요청
- 활성 세션이 있는 전략의 만료 처리 방법 (세션 종료? 이전 전략으로 계속?)

### 2-6. 부동소수점 결정론

효용 계산 결과의 **±0.001 허용오차**가 크로스 플랫폼 협상 임계값 판단에 충분한가?

서로 다른 플랫폼(또는 다른 하드웨어)에서 동일한 입력에 대해 미세하게 다른 결과가 나올 수 있습니다. 임계값 경계에서의 판단(`ACCEPT` vs `COUNTER`)이 플랫폼에 따라 달라지면 프로토콜 일관성이 깨집니다.

**검토 항목:** 고정소수점 연산 도입, 임계값 주변 데드존 설정, 결정론적 라운딩 규칙 표준화.

### 2-7. N:M 공정성

복수 구매자 대 복수 판매자 시나리오에서의 우선순위 정책이 미결입니다.

| 정책 | 장점 | 단점 |
|------|------|------|
| First-come | 단순, 투명 | 네트워크 지연에 불공정 |
| 가격 기반 | 경제적 효율 | 부유한 구매자 유리 |
| 평판 우선 | 신뢰도 향상 | 신규 사용자 불이익 |

복합 정책(가중 평균)도 고려 가능하나, 복잡도와 투명성 간 트레이드오프가 존재합니다.

### 2-8. BATNA 증명과 프라이버시

구매자가 BATNA(Best Alternative to a Negotiated Agreement)를 증명하기 위해 플랫폼 서명을 제시할 때, 이 서명이 간접적으로 다른 사용자의 행동을 노출하는 문제가 있습니다.

예를 들어, "다른 판매자로부터 $X 제안을 받았음"이라는 BATNA 증명은 해당 판매자의 가격 전략을 간접적으로 드러냅니다. 영지식 증명(ZKP) 등의 프라이버시 보존 기법 적용 필요성을 검토해야 합니다.

### 2-9. WAITING → ACTIVE 승격 지연

`max_active_sessions` 제한으로 WAITING 상태에 있던 세션이 Top 세션 탈락(종료/만료) 후 ACTIVE로 승격될 때, **상대 리스팅의 유효성 검사**가 필요합니다.

- 대기 중 상대 리스팅이 삭제되었거나 가격이 변경되었을 수 있음
- 승격 시점에 리스팅 재검증 → 실패 시 WAITING 풀의 다음 세션으로 폴백
- 승격 지연 시간의 상한 설정 (무한 대기 방지)

### 2-10. 양방향 매칭 동시성

구매자 Intent와 판매자 리스팅이 동시에 등록될 때, 이벤트 처리 순서에 따라 매칭 결과가 달라질 수 있습니다.

- **이벤트 순서 의존성:** Intent 먼저 등록 vs 리스팅 먼저 등록에 따라 다른 매칭 결과
- **멱등성 보장 필요:** 동일한 Intent-Listing 쌍에 대해 중복 매칭이 발생하지 않아야 함
- **검토 필요:** 분산 락, 이벤트 소싱, CRDT 등의 동시성 제어 기법 적용
