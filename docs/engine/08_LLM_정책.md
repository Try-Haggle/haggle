## 1. LLM 호출 지점

Haggle Engine은 **Cost-Aware** 원칙에 따라 LLM 호출을 최소화합니다. 전체 협상 파이프라인에서 LLM이 개입하는 지점은 정확히 **3가지**입니다.

```
┌─────────────────────────────────────────────────────────────┐
│                    LLM 호출 지점 개요                         │
│                                                             │
│  ① Cold Path ─────── 초기 전략 수립 (세션 시작 전)            │
│  ② Reactive ─────── 상대 제안 해석 (라운드 중, 필요 시)       │
│  ③ Strategy Review ─ 진행 중 전략 재검토 (교착 시)            │
│                                                             │
│  나머지 모든 계산 = Engine Core (순수 수학, ~200μs)           │
└─────────────────────────────────────────────────────────────┘
```

---

### 1.1 ① Cold Path: 초기 전략 수립

사용자의 자연어 요청을 Master Strategy 파라미터로 변환하는 단계입니다. 세션이 시작되기 **전**에 1회 호출됩니다.

**빈도:** 상품당 1회 (복잡한 경우 2-3회)

#### LLM 경로 (자연어 입력)

```
┌──────────────────────────────────────────────────────────────────────┐
│  사용자 입력                                                         │
│  "아이패드 프로 M4 256GB 사고 싶어, AppleCare 남은 것, 예산 $700-850" │
└───────────────────────────┬──────────────────────────────────────────┘
                            │
                            ▼
                 ┌─────────────────────┐
                 │   Strategy Skill    │
                 └──────────┬──────────┘
                            │
                            ▼
              ┌───────────────────────────┐
              │  LLM (Grok 4.1 Fast)     │
              │  reasoning=ON             │
              └─────────┬─────────────────┘
                        │
           ┌────────────┴────────────┐
           │                         │
           ▼                         ▼
  ┌─────────────────┐     ┌──────────────────────┐
  │ 1차 호출         │     │ 2차 호출 (필요 시)    │
  │ intent 파싱      │     │ 전략 파라미터 생성     │
  │ + 카테고리 ID    │     │                      │
  └────────┬────────┘     └──────────┬───────────┘
           │                         │
           └────────────┬────────────┘
                        │
                        ▼
              ┌───────────────────┐
              │  Master Strategy  │
              │  ─────────────── │
              │  p_target: $720   │
              │  p_limit:  $850   │
              │  w_price:  0.55   │
              │  w_quality: 0.25  │
              │  w_time:   0.10   │
              │  w_risk:   0.10   │
              │  beta:     0.7    │
              └───────────────────┘
```

#### 프리셋 경로 (v1.0.1: LLM 불필요)

v1.0.1에서 프리셋(AgentStats)을 사용하면 Stats→Parameter 변환이 **결정론적**이므로 LLM 호출이 필요 없습니다.

```
┌──────────────────────────────────────────────────────────┐
│  사용자 입력                                              │
│  "Tiger로 아이패드 사고 싶어, 예산 $800"                   │
└───────────────────────────┬──────────────────────────────┘
                            │
                            ▼
              ┌───────────────────────────┐
              │  Market Research Skill    │
              │  (API 호출, LLM 아님)      │
              │  시세: $750-$900          │
              └─────────────┬─────────────┘
                            │
                            ▼
              ┌───────────────────────────┐
              │  Stats→Parameter 변환기    │
              │  (순수 함수, 결정론적)      │
              │                           │
              │  Tiger 프리셋:             │
              │    resolve=0.7            │
              │    aggression=0.6         │
              │    patience=0.5           │
              │         │                 │
              │         ▼                 │
              │  p_target, p_limit,       │
              │  w_i, beta 자동 산출       │
              └─────────────┬─────────────┘
                            │
                            ▼
              ┌───────────────────┐
              │  Master Strategy  │
              │  LLM 호출: 0회    │
              └───────────────────┘
```

### 1.2 ② Reactive Escalation: 상대 제안 해석

Engine Core가 상대방의 제안에서 **처리할 수 없는 요소**를 감지하면, LLM에 해석을 요청합니다.

**핵심 원칙:** *"Engine Core는 자기가 모르는 걸 안다."*

Proposal Parser가 미인식 요소를 감지하면 즉시 `ESCALATE(UNKNOWN_PROPOSAL)`을 반환합니다. LLM은 **"번역기"** 역할로, 미지의 요소를 4차원 효용(가격/품질/시간/리스크)으로 변환합니다.

#### 예시: 번들 제안 수신

```
  Round 4: 상대방 제안
  "$830 + 무료 케이스 + 스크린 필름"
          │
          ▼
┌──────────────────────────┐
│  Proposal Parser         │
│  ────────────────────── │
│  가격: $830  ✅ 파싱 성공 │
│  케이스:     ❌ 미인식     │
│  스크린필름: ❌ 미인식     │
└──────────┬───────────────┘
           │
           ▼ ESCALATE(UNKNOWN_PROPOSAL)
┌──────────────────────────────────────────────┐
│  LLM 해석 요청                                │
│  ──────────────────────────────────────────── │
│  "상대가 $830 + 케이스 + 스크린필름 제안함.     │
│   각 부가 항목의 시장 가치와                    │
│   가격 할인 등가값을 추정해줘."                  │
└──────────────┬───────────────────────────────┘
               │
               ▼
┌──────────────────────────────────────────────┐
│  LLM 응답                                     │
│  ──────────────────────────────────────────── │
│  케이스: ~$30 가치                             │
│  스크린필름: ~$15 가치                          │
│  → p_effective = $830 - $30 - $15 = $785      │
└──────────────┬───────────────────────────────┘
               │
               ▼
┌──────────────────────────────────────────────┐
│  Engine Core 재계산                            │
│  ──────────────────────────────────────────── │
│  p_effective = $785 기준으로                    │
│  computeUtility() → makeDecision()            │
│  (순수 수학 경로 복귀)                          │
└──────────────────────────────────────────────┘
```

### 1.3 ③ Strategy Review: 진행 중 전략 재검토

4라운드 이상 교착 상태가 지속되면 `ESCALATE(STRATEGY_REVIEW)`가 트리거되고, LLM에 전략 조정을 요청합니다.

#### 예시: 4라운드 교착

```
┌───────────────────────────────────────────────────────────┐
│  에스컬레이션 컨텍스트                                      │
│  ─────────────────────────────────────────────────────── │
│  "4라운드째 양보 없음.                                     │
│   상대 최근 3개 제안: $810, $805, $803.                    │
│   내 전략: p_target=$720.                                 │
│   현실적 재조정 필요?"                                     │
└───────────────────────────┬───────────────────────────────┘
                            │
                            ▼
              ┌───────────────────────────┐
              │  LLM 전략 검토            │
              │  ──────────────────────  │
              │  권장 조정:               │
              │  p_target: $720 → $760   │
              │  beta:     0.7 → 0.5     │
              │  (양보 속도 증가)          │
              └───────────────────────────┘
```

## 2. 에스컬레이션 트리거 전체 목록

| 트리거 | 유형 | LLM 필요? | 설명 |
|--------|------|-----------|------|
| 미인식 요소 (번들, 트레이드인 등) | `UNKNOWN_PROPOSAL` | 항상 | Proposal Parser가 파싱 불가한 요소 감지 |
| 비표준 결제 조건 (할부, 에스크로 등) | `UNKNOWN_PROPOSAL` | 항상 | 가격 외 결제 조건이 비표준 형식 |
| 수량 할인 제안 | `UNKNOWN_PROPOSAL` | 항상 | 단가 × 수량 구조를 효용으로 변환 필요 |
| 4라운드+ 교착 (`rounds_no_concession ≥ 4`) | `STRATEGY_REVIEW` | 가능 | 양쪽 모두 양보 없이 4라운드 이상 경과 |
| 시장가 10%+ 급변 | `STRATEGY_REVIEW` | 가능 | 외부 시장 데이터에서 급격한 가격 변동 감지 |
| `V_t < 0.05` AND 합의 불가 | `STRATEGY_REVIEW` | 예 | 마감 임박 + 현재 전략으로 합의 불가능 |
| 사용자 전략 변경 요청 | `STRATEGY_REVIEW` | 가능 | 사용자가 세션 도중 전략 조정 요청 |

## 3. 에스컬레이션 제어 장치

LLM 호출의 무분별한 증가를 방지하기 위한 제어 장치입니다.

### 3.1 세션당 최대 호출 횟수

```
┌─────────────────────────────────────────┐
│  세션 LLM 호출 한도: 총 8회             │
│  ───────────────────────────────────── │
│  UNKNOWN_PROPOSAL:  최대 5회            │
│  STRATEGY_REVIEW:   최대 3회            │
│  ───────────────────────────────────── │
│  초과 시: 사용자 수동 개입 요청           │
│  "LLM 호출 한도에 도달했습니다.          │
│   직접 판단을 입력해주세요."             │
└─────────────────────────────────────────┘
```

### 3.2 쿨다운 규칙

| 유형 | 쿨다운 | 비고 |
|------|--------|------|
| `UNKNOWN_PROPOSAL` | 없음 | 동일 유형은 캐시 활용 (섹션 4 참조) |
| `STRATEGY_REVIEW` | 최소 3라운드 | 연속 호출 방지, 전략 변경 효과 관찰 필요 |

### 3.3 전략 변경 제한

전략 파라미터 변경 시 사용자 확인이 필요한 경우:

```
┌─────────────────────────────────────────────────────┐
│  전략 변경 보호 장치                                  │
│  ───────────────────────────────────────────────── │
│                                                     │
│  w_i 변경 > ±0.15  → 사용자 확인 필요                │
│    예: w_price 0.55→0.35 (차이 0.20) → 확인 요청     │
│    예: w_price 0.55→0.50 (차이 0.05) → 자동 적용     │
│                                                     │
│  p_limit 변경      → 항상 사용자 확인 필요             │
│    ※ 마지노선은 사용자 권한                            │
│    예: p_limit $850→$900 → 반드시 확인 요청           │
│                                                     │
└─────────────────────────────────────────────────────┘
```

## 4. 해석 결과 캐싱

동일한 유형의 제안이 반복되면 LLM을 다시 호출하지 않고 캐시된 결과를 재사용합니다.

### 4.1 캐시 구조

```python
class ProposalInterpretationCache:
    """제안 해석 결과 캐시. 동일 유형 반복 시 LLM 재호출 방지."""

    def __init__(self):
        self._cache: dict[str, dict] = {}
        self._timestamps: dict[str, float] = {}
        self.TTL = 3600  # 1시간

    def _make_key(self, proposal_type: str, proposal_params: dict) -> str:
        """제안 유형 + 핵심 파라미터로 캐시 키 생성."""
        sorted_params = tuple(sorted(proposal_params.items()))
        return f"{proposal_type}:{hash(sorted_params)}"

    def get_interpretation(
        self, proposal_type: str, proposal_params: dict
    ) -> dict | None:
        """캐시된 해석 결과 조회. 없거나 만료 시 None 반환."""
        cache_key = self._make_key(proposal_type, proposal_params)
        if cache_key not in self._cache:
            return None
        if time.time() - self._timestamps[cache_key] > self.TTL:
            del self._cache[cache_key]
            del self._timestamps[cache_key]
            return None
        return self._cache[cache_key]

    def cache_interpretation(
        self, proposal_type: str, proposal_params: dict, interpretation: dict
    ) -> None:
        """해석 결과를 캐시에 저장."""
        cache_key = self._make_key(proposal_type, proposal_params)
        self._cache[cache_key] = interpretation
        self._timestamps[cache_key] = time.time()
```

### 4.2 캐시 활용 예시

```
Round 4: "$830 + 케이스 + 스크린필름"
  → 유형: BUNDLE
  → 캐시 미스 → LLM 호출
  → 결과: 케이스=$30, 스크린필름=$15
  → 캐시 저장: BUNDLE:{케이스, 스크린필름}

Round 6: "$820 + 케이스 + 스크린필름"
  → 유형: BUNDLE
  → 캐시 키 동일 (유형 + 항목 동일, 가격만 다름)
  → 캐시 히트 → LLM 호출 없음
  → p_effective = $820 - $30 - $15 = $775

Round 8: "$810 + 트레이드인(아이패드 에어 5)"
  → 유형: TRADE_IN
  → 캐시 미스 (새로운 유형) → LLM 호출
  → 결과: 트레이드인 가치=$200
  → p_effective = $810 - $200 = $610
```

## 5. LLM 호출 빈도 예측

| 시나리오 | Cold Path | Reactive | Review | 합계 |
|----------|:---------:|:--------:|:------:|:----:|
| 단순 거래 (가격만 협상) | 1 | 0 | 0 | **1** |
| 단순 거래 + 프리셋 (v1.0.1) | 0 | 0 | 0 | **0** |
| 일반 거래 (10라운드, 순조로움) | 1 | 0 | 0 | **1** |
| 번들 제안 1회 수신 | 1 | 1 | 0 | **2** |
| 복잡한 상대 (번들 + 트레이드인) | 1 | 2 | 1 | **4** |
| 장기 + 시장 급변 + 트레이드인 | 2 | 2 | 2 | **6** |
| **일반 평균** | **1** | **0.3** | **0.2** | **1.5** |
| **프리셋 사용 시 평균 (v1.0.1)** | **0** | **0.3** | **0.2** | **0.5** |

```
┌─────────────────────────────────────────────────────────────┐
│  비용 함의                                                   │
│  ─────────────────────────────────────────────────────────  │
│                                                             │
│  일반 거래: ~1.5회 LLM 호출 / 세션                           │
│  프리셋:   ~0.5회 LLM 호출 / 세션                            │
│                                                             │
│  Hot Path 비율 (Engine Core만으로 처리):                      │
│    일반: ~85% 라운드가 LLM 없이 처리                          │
│    프리셋: ~95% 라운드가 LLM 없이 처리                        │
│                                                             │
│  ※ 세션당 최대 8회 제한으로 비용 상한선 보장                   │
└─────────────────────────────────────────────────────────────┘
```
