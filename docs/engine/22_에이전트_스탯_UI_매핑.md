## 1. 목적

[06_에이전트_스탯.md](./06_%EC%97%90%EC%9D%B4%EC%A0%84%ED%8A%B8_%EC%8A%A4%ED%83%AF.md) 는 `AgentStats -> Engine Parameters` 변환을 정의한다.  
하지만 실제 제품에서는 아래 질문이 별도로 필요하다.

- 사용자가 이 능력치를 어떤 이름으로 이해할까?
- 판매자와 구매자에게 같은 스탯을 똑같이 보여줘야 할까?
- 어떤 스탯은 기본 UI에서 노출하고, 어떤 파라미터는 고급 설정으로 내려야 할까?
- `beta`, `u_threshold` 같은 내부 파라미터를 직접 보여줄까, 아니면 사용자 언어로 번역할까?

이 문서는 그 UI 계층을 정의한다.

---

## 2. 핵심 원칙

### 2.1 스탯은 보여주되, 엔진 파라미터를 기본값으로 직접 노출하지 않는다

기본 UI는 아래 순서를 따른다.

1. 프리셋 선택
2. 8개 스탯 확인 또는 조정
3. 스탯이 의미하는 협상 성향 설명
4. 고급 사용자가 원할 때만 파라미터 프리뷰 노출

즉 기본 UX는:

`Tiger / Fox / Owl / ...`  
`앵커링 85, 끈기 80, 결단력 60`

고급 UX는:

`beta = 2.62, u_threshold = 0.58, weights = { ... }`

### 2.2 같은 스탯이라도 판매자/구매자 문구는 달라진다

예를 들어 Anchoring 은 내부적으로 동일한 성질이지만, 사용자에게는 이렇게 다르게 보인다.

- 판매자: "처음 얼마부터 부를지"
- 구매자: "처음 얼마나 낮게 시작할지"

### 2.3 사용자는 스탯을 조정하고, 시스템은 전략을 조립한다

사용자가 `stats` 를 조정하면 시스템이 자동으로 아래를 계산한다.

- weights
- alpha / beta
- threshold / aspiration
- risk filter
- relationship bias

즉 `엔진을 만든다`는 경험은 실제로는 `스탯과 프리셋을 조정해서 전략을 조립한다`는 경험이어야 한다.

## 3. 기본 정보 구조

능력치 UI는 3단계 구조를 따른다.

### 3.1 레벨 1: 프리셋

가장 먼저 보여줄 것:

- Tiger
- Fox
- Turtle
- Owl
- Dolphin
- Eagle

각 프리셋은 아래만 즉시 보여준다.

- 한 줄 설명
- 적합 상황
- 대표 장단점

### 3.2 레벨 2: 스탯 카드

프리셋 선택 후 보여줄 것:

- 8개 스탯 이름
- 현재 값
- 한 줄 해석
- 높을 때/낮을 때의 행동 변화

### 3.3 레벨 3: 엔진 프리뷰

고급 모드에서만 보여줄 것:

- utility weights
- alpha / beta
- threshold / aspiration
- risk filter minimum
- information completeness minimum
- use_utility_space 여부

## 4. 8개 스탯의 UI 매핑

| 엔진 스탯 | 기본 표시 이름 | 판매자용 설명 | 구매자용 설명 | 기본 노출 |
|---|---|---|---|---|
| Anchoring | 시작 가격 공격성 | 첫 호가를 얼마나 세게 부를지 | 첫 제안을 얼마나 낮게 던질지 | 예 |
| Tenacity | 양보 속도 | 얼마나 천천히 깎아줄지 | 얼마나 천천히 올릴지 | 예 |
| Resolve | 수용 기준 | 어느 선 아래는 안 팔지 | 어느 선 위로는 안 살지 | 예 |
| Market Sense | 시장 반영 | 경쟁 매물과 시장 흐름을 얼마나 읽을지 | 시세/대안 가격을 얼마나 반영할지 | 예 |
| Risk Radar | 상대 위험 민감도 | 저신뢰 구매자를 얼마나 경계할지 | 저신뢰 판매자를 얼마나 경계할지 | 예 |
| Scrutiny | 정보 검증 수준 | 구매자와 대화할 때 정보 요구 수준 | 판매자 리스팅의 정보 부족을 얼마나 싫어할지 | 예 |
| Patience | 시간 압박 저항 | 시간이 지나도 얼마나 버틸지 | 급하지 않을 때 얼마나 오래 기다릴지 | 예 |
| Rapport | 관계 중시도 | 거래 관계와 톤을 얼마나 중시할지 | 좋은 판매자/반복 거래를 얼마나 높게 볼지 | 예 |

> **고급 연결 파라미터:** 각 스탯이 매핑되는 엔진 파라미터(anchor_ratio, beta, u_threshold 등)는 [06_에이전트_스탯.md](./06_에이전트_스탯.md) §2~§5를 참조하세요.

## 5. 스탯별 상세 표시 규칙

### 5.1 Anchoring

기본 라벨:

- `시작 가격 공격성`

판매자 문구:

- 낮음: "시장가에 가깝게 시작"
- 높음: "강하게 높은 가격에서 시작"

구매자 문구:

- 낮음: "무난한 가격에서 시작"
- 높음: "처음부터 세게 깎기 시작"

보여줄 값:

- 기본: 0~100 느낌의 바/슬라이더
- 고급: `anchor_ratio`, `p_target`

### 5.2 Tenacity

기본 라벨:

- `양보 속도`

판매자/구매자 공통 문구:

- 낮음: "빨리 타협"
- 높음: "쉽게 물러서지 않음"

보여줄 값:

- 기본: `빠름 ↔ 느림`
- 고급: `beta`

### 5.3 Resolve

기본 라벨:

- `수용 기준`

판매자 문구:

- 낮음: "거래 성사 우선"
- 높음: "조건이 안 맞으면 안 판다"

구매자 문구:

- 낮음: "웬만하면 산다"
- 높음: "좋은 딜만 산다"

보여줄 값:

- 기본: `거래 성사 우선 ↔ 기준 엄격`
- 고급: `u_threshold`, `u_aspiration`

### 5.4 Market Sense

기본 라벨:

- `시장 반영`

설명:

- 낮음: "상대 제안 위주로 본다"
- 높음: "시세, 경쟁 매물, 대안을 적극 반영한다"

고급:

- `gamma`
- `market_utilization`

### 5.5 Risk Radar

기본 라벨:

- `상대 위험 민감도`

설명:

- 낮음: "평판이 낮아도 거래를 시도한다"
- 높음: "신뢰가 낮으면 일찍 거른다"

고급:

- `w_rep`
- `r_score_minimum`

### 5.6 Scrutiny

기본 라벨:

- `정보 검증 수준`

설명:

- 낮음: "사진/설명이 부족해도 본다"
- 높음: "정보가 충분하지 않으면 바로 경계한다"

고급:

- `w_info`
- `i_completeness_minimum`

### 5.7 Patience

기본 라벨:

- `시간 압박 저항`

설명:

- 낮음: "빨리 끝내고 싶다"
- 높음: "좋은 조건이 올 때까지 기다린다"

고급:

- `alpha`
- `v_t_floor`

### 5.8 Rapport

기본 라벨:

- `관계 중시도`

설명:

- 낮음: "가격 중심"
- 높음: "거래 톤, 반복 거래, 신뢰 관계를 더 본다"

고급:

- `v_s_base`
- `n_threshold`
- `late_round_aggression_modifier`

## 6. 판매자 화면에서의 노출

판매자 화면은 아래 질문에 답하는 구조여야 한다.

- 얼마부터 시작할까?
- 어디까지 내려갈까?
- 얼마나 오래 버틸까?
- 상대가 믿을 만하지 않으면 얼마나 빨리 끊을까?

### 6.1 판매자 기본 카드

판매자 기본 화면에서는 아래 5개를 우선 노출한다.

1. 시작 가격 공격성
2. 양보 속도
3. 수용 기준
4. 시간 압박 저항
5. 상대 위험 민감도

관계 중시도와 정보 검증 수준은 `더 보기` 아래로 내려도 된다.

### 6.2 판매자용 문구 예시

- `강경한 시작가`
- `천천히 양보`
- `마지노선 엄격`
- `오래 버틸 수 있음`
- `수상한 상대는 빠르게 거름`

### 6.3 판매자에게 기본으로 숨겨도 되는 것

아래는 고급 패널에서만 보여줘도 된다.

- weights
- alpha / beta
- gamma
- w_rep / w_info

## 7. 구매자 화면에서의 노출

구매자 화면은 아래 질문에 답하는 구조여야 한다.

- 얼마나 낮게 시작할까?
- 어디까지 올릴 수 있을까?
- 품질/정보 부족을 얼마나 싫어할까?
- 시간을 쓸지, 빨리 살지?

### 7.1 구매자 기본 카드

구매자 기본 화면에서는 아래 5개를 우선 노출한다.

1. 시작 가격 공격성
2. 양보 속도
3. 수용 기준
4. 정보 검증 수준
5. 상대 위험 민감도

Patience 와 Market Sense 는 `전략 더 보기` 패널로 내려도 된다.

### 7.2 구매자용 문구 예시

- `처음부터 세게 깎기`
- `쉽게 가격을 올리지 않음`
- `좋은 딜만 수용`
- `정보 부족 상품은 경계`
- `신뢰 낮은 판매자는 피함`

## 8. 프리셋 UI 규칙

프리셋 카드는 단순히 이름만 보여주면 안 된다. 각 카드에는 아래 4가지가 필요하다.

1. 프리셋 이름
2. 한 줄 캐치프레이즈
3. 적합한 상황
4. 가장 높은 스탯 2개 / 가장 낮은 스탯 2개

예:

- `Tiger`: 강한 시작가, 느린 양보. 경쟁 매물이 많은 카테고리에 적합.
- `Fox`: 균형형. 초보자 기본값.
- `Turtle`: 오래 버티는 타입. 급하지 않은 거래.
- `Owl`: 정보/시장 분석형. 전자기기, 차량, 고관여 카테고리.
- `Dolphin`: 관계형. 반복 거래, 커뮤니티 거래.
- `Eagle`: 안전 우선형. 고가 제품, 사기 위험 높은 카테고리.

## 9. 플레이그라운드/프로덕트 권장 노출 구조

### 9.1 기본 모드

- 프리셋 선택
- 8개 스탯 슬라이더
- 예상 협상 성향 한 줄 요약

예:

- `시작은 강하게, 양보는 느리게, 기준은 다소 엄격`

### 9.2 고급 모드

- 전략 프리뷰
- 역할 (`SELLER` / `BUYER`)
- alpha / beta
- threshold / aspiration
- use_utility_space
- utility weights

### 9.3 실시간 협상 중 보여줄 것

실시간 협상 타임라인에는 라운드마다 아래를 보여준다.

- 들어온 제안가
- 엔진 결정
- 카운터오퍼
- 총 utility
- price/time/risk/relation 구성값

즉 스탯 설정 화면과 타임라인 화면은 연결돼야 한다.

- 스탯은 "어떤 성격의 엔진인지"를 보여주고
- 타임라인은 "그 성격이 실제로 어떻게 행동했는지"를 보여준다.

## 10. 하지 말아야 할 것

### 10.1 기본 화면에 raw 파라미터만 노출하지 말 것

사용자에게 아래처럼만 보여주면 안 된다.

- `beta = 2.33`
- `u_threshold = 0.58`
- `w_rep = 0.67`

이 값은 의미를 모르면 무의미하다.

### 10.2 판매자와 구매자에게 동일한 문구를 쓰지 말 것

동일 스탯이라도 역할에 따라 표현이 달라져야 한다.

### 10.3 프리셋 이름만 보여주고 차이를 숨기지 말 것

`Tiger`, `Fox`, `Owl` 만 보여주면 안 된다.  
사용자는 최소한 아래는 즉시 이해해야 한다.

- 누가 더 공격적인가
- 누가 더 안전 지향적인가
- 누가 더 오래 버티는가

## 12. 결론

엔진의 능력치는 이미 문서화되어 있다.  
제품에서 해야 할 일은 그 능력치를 `사용자가 이해할 수 있는 말과 화면`으로 번역하는 것이다.

정리하면:

- 내부 기준: `AgentStats -> Engine Parameters`
- 제품 기준: `프리셋 -> 스탯 -> 전략 프리뷰 -> 실제 행동 타임라인`

이 문서는 그 번역 계층의 기준 문서다.
