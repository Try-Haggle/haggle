## 1. 적합성 테스트 개요

> **범위 구분:** 이 문서는 **Engine Core의 수학 계산 결과**를 검증하는 테스트입니다.
> HNP 프로토콜 메시지 처리(세션 상태 전이, 직렬화 등)의 상호운용 테스트는 [20_HNP_적합성_테스트_부록.md](./20_HNP_적합성_테스트_부록.md)를 참조하세요.

- 모든 적합 구현체는 다음 결과를 **±0.001 허용오차** 내에서 생성해야 한다(MUST).
- 테스트는 Engine Core의 핵심 계산(4차원 효용, 경쟁 조정, 가중치 검증, AgentStats 변환)을 모두 검증한다.

## 2. 테스트 케이스

### 테스트 1: 균형 잡힌 구매자

**입력:**

```yaml
weights:
  w_p: 0.4
  w_t: 0.3
  w_r: 0.2
  w_s: 0.1

price:
  p_effective: 200
  p_target: 180
  p_limit: 220

time:
  t_elapsed: 36000
  t_deadline: 86400
  alpha: 1.0
  v_t_floor: 0.0

risk:
  r_score: 0.85
  i_completeness: 0.90
  w_rep: 0.6
  w_info: 0.4

relationship:
  n_success: 3
  n_dispute_losses: 0
  n_threshold: 10
  v_s_base: 0.5
```

**기대값:**

| 차원 | 값 | 설명 |
|------|------|------|
| `v_p` | **0.820** | 구매자(180<220), 200<220이므로 가드 통과, ln(21)/ln(41) |
| `v_t` | **0.583** | max(0.0, (1 - 36000/86400)^1.0) = 0.583 |
| `v_r` | **0.870** | 0.6×0.85 + 0.4×0.90 |
| `v_s` | **0.800** | clamp(0.5 + 3/10 + 0) = 0.8 |
| **`u_total`** | **0.683** | 0.4×0.820 + 0.3×0.583 + 0.2×0.870 + 0.1×0.800 |

**입력:**

```yaml

  r_score: 0.70

  w_rep: 0.6
  w_info: 0.4

relationship:
  n_success: 0
  n_dispute_losses: 0
  n_threshold: 10
  v_s_base: 0.5
```

**기대값:**

| 차원 | 값 | 설명 |
|------|------|------|
| `v_p` | **0.920** | 판매자(220>180), 210>180이므로 가드 통과, ln(31)/ln(41) |
| `v_t` | **0.965** | max(0.0, (1 - 7200/604800)^3.0) |
| `v_r` | **0.740** | 0.6×0.70 + 0.4×0.80 |
| `v_s` | **0.500** | clamp(0.5 + 0/10 + 0) = 0.5 |
| **`u_total`** | **0.860** | 0.70×0.920 + 0.10×0.965 + 0.15×0.740 + 0.05×0.500 |

### 테스트 3: 경쟁 컨텍스트 적용

**입력:** 테스트 1과 동일 + 다음 경쟁 파라미터 추가

```yaml
competition:
  n_competitors: 4
  best_alternative: 195
  market_position: 0.7

gamma: 0.1
```

**기대값:**

| 항목 | 값 | 설명 |
|------|------|------|
| `v_p_base` | **0.820** | 테스트 1과 동일 |
| `adjustment` | **1.1127** | 1 + 0.1 × ln(5) × 0.7 = 1.1127 |
| `v_p_adjusted` | **0.912** | clamp(0.820 × 1.1127, 0, 1) = 0.912 |
| **`u_total`** | **0.794** | 0.4×0.912 + 0.3×0.583 + 0.2×0.870 + 0.1×0.800 |

### 테스트 5: 오류 — 유효하지 않은 가중치

**입력:**

  w_t: 0.3
  w_r: 0.2

### 테스트 6: Fox 프리셋 스탯 검증 (v1.0.1)

**입력:**

```yaml
stats:
  anchoring: 50
  tenacity: 50
  resolve: 50
  market_sense: 50
  risk_radar: 50
  scrutiny: 50
  patience: 50
  rapport: 50
```

**기대값 (파라미터 변환):**

| 파라미터 | 공식 | 값 |
|----------|------|------|
| `anchor_ratio` | 1.0 - (50/100) × 0.4 | **0.800** |
| `beta` | 0.3 + (50/100) × 2.9 | **1.750** |
| `u_threshold` | 0.25 + (50/100) × 0.55 | **0.525** |
| `u_aspiration` | 0.525 + 0.15 | **0.675** |
| `gamma` | (50/100) × 0.2 | **0.100** |
| `w_rep` | 0.3 + (50/100) × 0.5 | **0.550** |
| `w_info` | 1.0 - 0.55 | **0.450** |
| `alpha` | 0.3 + (50/100) × 2.7 | **1.650** |
| `v_t_floor` | (50/100) × 0.85 | **0.425** |
| `v_s_base` | 0.3 + (50/100) × 0.5 | **0.550** |
| `n_threshold` | round(15 - (50/100) × 10) | **10** |
| `late_round_aggression_modifier` | rapport=50 ≤ 60 | **1.0** |

**가중치 도출:**

| 항목 | 공식 | 값 |
|------|------|------|
| `raw_wp` | 50×0.4 + 50×0.3 + 50×0.2 + 50×0.1 | **50.0** |
| `raw_wt` | (100-50)×0.8 + 50×0.2 | **50.0** |
| `raw_wr` | 50×0.5 + 50×0.5 | **50.0** |
| `raw_ws` | 50×0.9 + 50×0.1 | **50.0** |
| `total` | | **200.0** |
| **`w_p`** | 50.0 / 200.0 | **0.250** |
| **`w_t`** | 50.0 / 200.0 | **0.250** |
| **`w_r`** | 50.0 / 200.0 | **0.250** |
| **`w_s`** | 50.0 / 200.0 | **0.250** |

### 테스트 10: Dolphin 프리셋 + 실제 협상 (v1.0.1)

전체 E2E(End-to-End) 테스트. Dolphin 프리셋으로 스탯 → 파라미터 → 효용 계산 → 의사결정까지 전 과정을 검증한다.

#### Step 1: Stats → 파라미터

**입력:**

```yaml
stats:
  anchoring: 25
  tenacity: 30
  resolve: 40
  market_sense: 45
  risk_radar: 50
  scrutiny: 45
  patience: 75
  rapport: 90
```

**기대값 (파라미터 변환):**

| 파라미터 | 값 |
|----------|------|
| `w_rep` | **0.550** |
| `w_info` | **0.450** |
| `alpha` | **2.325** |
| `v_t_floor` | **0.6375** |
| `v_s_base` | **0.750** |
| `n_threshold` | **6** |
| `late_round_aggression_modifier` | **0.8** |
| `u_threshold` | **0.470** |
| `u_aspiration` | **0.620** |

**가중치 도출:**

| 항목 | 값 |
|------|------|
| `raw_wp` | **31.5** |
| `raw_wt` | **26.0** |
| `raw_wr` | **47.5** |
| `raw_ws` | **88.5** |
| `total` | **193.5** |
| **`w_p`** | **0.1628** |
| **`w_t`** | **0.1343** |
| **`w_r`** | **0.2454** |
| **`w_s`** | **0.4574** |

#### Step 2: compute_utility (테스트 1 시장 데이터 사용)

테스트 1과 동일한 시장 데이터(price, time, risk, relationship)에 Dolphin 파라미터를 적용한다.

| 차원 | 값 | 설명 |
|------|------|------|
| `v_p` | **0.820** | 테스트 1과 동일 |
| `v_t` | **0.6375** | max(0.6375, (1 - 36000/86400)^2.325) = max(0.6375, 0.376) = 0.6375 |
| `v_r` | **0.8725** | 0.55×0.85 + 0.45×0.90 = 0.4675 + 0.405 = 0.8725 |
| `v_s` | **1.000** | clamp(0.75 + 3/6 + 0) = clamp(1.25) = 1.000 |

**효용 합산:**

| 항목 | 계산 | 값 |
|------|------|------|
| `w_p × v_p` | 0.1628 × 0.820 | **0.1335** |
| `w_t × v_t` | 0.1343 × 0.6375 | **0.0856** |
| `w_r × v_r` | 0.2454 × 0.8725 | **0.2141** |
| `w_s × v_s` | 0.4574 × 1.000 | **0.4574** |
| **`u_total`** | | **0.891** |

#### Step 3: Decision Maker

```
u_total(0.891) ≥ u_aspiration(0.620) → ACCEPT
```

Dolphin은 관계 중심(w_s = 0.4574) 에이전트이므로, 높은 관계 효용이 전체 u_total을 끌어올려 즉시 수락(ACCEPT) 결정을 내린다.

#### 비교 참고

같은 시장 상황에서 Tiger 프리셋을 적용하면 가중치 분배가 달라(w_p 중심) u_total이 변하고, 이에 따라 다른 의사결정이 나올 수 있다. 이는 AgentStats 시스템이 동일한 시장 데이터에서도 에이전트 성격에 따라 서로 다른 전략적 판단을 생성함을 보여준다.
