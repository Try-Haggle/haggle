# x402_정산_및_분쟁_컨트랙트_아키텍처

## 목적

이 문서는 Haggle 이 `Base + USDC + x402` 를 사용할 때,
결제 정산과 분쟁 앵커링을 어떤 스마트 컨트랙트 경계로 가져갈지 정의한다.

핵심 목표는 세 가지다.

- seller 직접 수령
- Haggle fee 분리
- dispute 결과의 검증 가능성 확보

## 전제

- 기본 결제 rail 은 `x402`
- 기본 네트워크는 `Base`
- 기본 자산은 `USDC`
- buyer authorization 은 `human_wallet` 과 `agent_wallet` 를 모두 지원
- confirmation 은 facilitator + chain verification 의 hybrid 모델

## Payment Protection 구조

물리 상품 거래에서는 배송 확인과 무게 조정이 결제와 길게 얽힌다.
Haggle 은 non-custodial 스마트 컨트랙트 기반 Payment Protection 을 사용하여
자금 보호와 법적 면제를 동시에 달성한다.

> 용어 주의: "escrow" 는 미국에서 Escrow License 가 필요한 규제 용어이므로 사용하지 않는다.
> 대신 **Payment Protection**, **Conditional Hold**, **Settlement Contract** 를 사용한다.

### Settlement Contract 구성

```
Settlement Contract (스마트 컨트랙트):
  ├── 상품 대금              → Phase 1 에서 판매자에게 릴리즈
  ├── 배송비                 → EasyPost/캐리어에 지불
  ├── 무게 조정 버퍼          → Phase 2 에서 정산 후 잔액 판매자 반환
  ├── Haggle 수수료 (1.5%)   → Haggle fee wallet 로 분리
  └── emergency exit         → Timelock 기간 중 사용자 직접 인출 가능
```

Haggle 은 이 Settlement Contract 의 키를 보유하지 않는다 (non-custodial).
자금 릴리즈는 스마트 컨트랙트 로직이 조건 충족 시 자동 실행한다.

### 2단계 정산

- **Phase 1**: 배송 완료 (DELIVERED) + 구매자 확인 (24시간 자동 또는 즉시 확인 버튼) → 상품 대금 릴리즈
- **Phase 2**: 배송 완료 후 14일 → 무게 조정 버퍼 정산 (잔액 판매자 반환)

구매자 검토 기간(24시간)은 **플랫폼 정책으로 고정**이며 협상 대상이 아니다.
이유: 검토 기간을 0으로 네고하면 구매자 보호가 무력화됨.
릴리즈 후에는 non-custodial 구조상 판매자 지갑에서 강제 회수가 불가능하므로,
릴리즈 전 검토 기간이 유일한 구매자 보호 수단이다.

상세 설계는 [배송_추적_모듈_설계.md](./배송_추적_모듈_설계.md) 의 `2단계 정산` 섹션 참조.

### 배송비 = 협상 대상

배송비 부담자는 고정이 아니다.
구매자와 판매자가 협상으로 분담 비율을 결정한다.
Settlement Contract 총액은 협상 결과에 따라 달라진다.

### 무게 조정 버퍼

판매자가 입력한 무게의 한 단계 위 요금과의 차이를 버퍼로 Settlement Contract 에 잡는다.
무게 조정은 판매자 책임이며, 라벨 생성 시 명시적으로 고지한다.

### 첫 거래 무료 디포짓 정책

판매자의 첫 거래 시 분쟁 보증금과 무게 버퍼를 **$0으로 면제**한다 (고가 제품 제외).

- 첫 거래: 디포짓 없이 Settlement Contract 실행
- 추가 요금 발생 시(무게 차이, 분쟁 비용): 판매자에게 사후 청구
- 미납 시 페널티: 7일 리스팅 제한 → 14일 신뢰도 차감 → 30일 계정 정지
- 2회차 거래부터: 디포짓 정상 청구

이 정책은 Settlement Contract 레벨이 아니라 오프체인 비즈니스 로직으로 관리한다.
컨트랙트는 디포짓 유무와 무관하게 동일하게 동작하며,
첫 거래 면제 여부는 API 레이어에서 `deposit_amount = 0` 으로 전달한다.

### 기존 구조와의 관계

- `HaggleSettlementRouter` 는 Conditional Release 실행을 담당한다
- `HaggleDisputeRegistry` 는 분쟁 결과 앵커링을 담당한다
- dispute 는 offchain case system 이 처리하고 결과만 onchain 에 앵커링한다

## 컨트랙트 경계

### 1. HaggleSettlementRouter

역할:

- payment execution request 를 받는다
- seller payout amount 와 Haggle fee amount 를 분리한다
- approval snapshot hash 를 바인딩한다
- optional reservation id 를 바인딩한다
- execution event 를 발생시킨다

필수 입력:

- `order_id`
- `payment_intent_id`
- `buyer_id`
- `seller_id`
- `buyer_authorization_mode`
- `buyer_wallet`
- `seller_wallet`
- `haggle_fee_wallet`
- `gross_amount`
- `seller_amount`
- `haggle_fee_amount`
- `approval_snapshot_hash`

출력:

- `execution_id`
- `router_reference`
- `tx_hash`
- `status`

### 2. HaggleDisputeRegistry

역할:

- dispute case hash 저장
- evidence root hash 저장
- resolution hash 저장
- onchain reference 반환

필수 입력:

- `order_id`
- `dispute_case_id`
- `evidence_root_hash`
- `resolution_hash`

출력:

- `onchain_reference`
- `anchored_at`

## 상태 연결

1. 협상 완료 후 approval snapshot 고정
2. payment intent 생성
3. buyer authorization 완료
4. settlement router 실행
5. 정산 성공 시 payment settled
6. 분쟁이 없으면 종료
7. 분쟁 발생 시 offchain case 진행
8. resolution hash 를 dispute registry 에 앵커링
9. trust ledger 갱신

## hold / reservation 연결

- 일반 `soft hold` 는 router 와 직접 연결되지 않는다
- `reserved` 상태만 `reservation_id` 로 router 에 바인딩될 수 있다
- 즉 가격 보장과 onchain execution 을 섞는 경우에도,
  명시적으로 예약된 상태만 binding 대상이 된다

## human wallet vs agent wallet

### human wallet
- buyer 가 직접 wallet signature 수행
- 일반 사용자 흐름
- wallet UI/지갑 앱 서명이 필요

### agent wallet
- agent 가 programmatic wallet 로 서명
- MCP/UCP 자동 거래 흐름
- 별도 spending policy 와 risk control 필요

컨트랙트/정산 레벨에서는 둘 다 `buyer_authorization_mode` 로 동일하게 기록한다.

## 신뢰 데이터 연결

SettlementRouter 와 DisputeRegistry 는 trust ledger 와 연결된다.

- 승인 후 미결제
- 배송 SLA 미준수
- 분쟁 승리/패배
- 성공 정산

이후 요약값은 `OnchainTrustProfile` 과 `settlement_reliability` 에 반영된다.

## 컨트랙트 거버넌스

### 원칙: Non-Custodial First

Haggle 은 **어떤 상황에서도 사용자 자금에 대한 키를 보유하지 않는다.**
이것은 기술적 설계이자 법적 보호의 핵심이다.

- 자금 흐름: `buyer wallet → smart contract → seller wallet`
- Haggle 은 컨트랙트를 배포하고 오프체인 로직을 운영하지만, 자금을 직접 보관/전송하지 않는다
- 분쟁 해결 시 Haggle 은 oracle 제출자 역할만 수행하고, 컨트랙트가 실행한다

### 업그레이드 모델: Timelock + Multisig

초기부터 컨트랙트 업그레이드 권한과 자금 접근 권한을 완전히 분리한다.

DAO 거버넌스는 채택하지 않는다. 토큰 발행은 증권법 규제 대상이 되므로,
Multisig 기반 Progressive Decentralization 으로 충분한 탈중앙화를 달성한다.

#### Phase 1 — 런칭

```
3-of-5 Multisig (Gnosis Safe)
  + 48시간 Timelock (OpenZeppelin TimelockController)
  서명자: 내부 인원
```

- 초기에는 팀 규모 제약으로 내부자만 구성
- Timelock + emergency exit 으로 사용자 보호는 즉시 작동
- 모든 업그레이드 제안은 온체인 공개

#### Phase 2 — PMF 달성 후

```
4-of-7 Multisig
  + 72시간 Timelock
  서명자: 내부 4명 + 외부 3명
```

- 외부 인원 추가: 보안 감사자, 법률 자문, 업계 전문가
- Timelock 72시간으로 연장

#### Phase 3 — 스케일링

```
5-of-9 Multisig
  + 72시간 Timelock
  서명자: 내부 4명 + 외부 5명 (외부 과반)
```

- 외부 인원이 과반을 차지
- Guardian Multisig 분리 (pause 만 가능한 별도 Multisig)
- 업그레이드마다 외부 감사 의무화

#### Phase 4 — 성숙기

```
7-of-13 Multisig
  + 72시간 Timelock
  서명자: 외부 과반 유지
```

- 서명자 신원 공개 (이름, 역할, 소속)
- 연 1회 서명자 로테이션 정책
- 업그레이드마다 컨트랙트 감사 리포트 공개 의무

#### 왜 DAO 가 아닌가

- 거버넌스 토큰 발행 = 미국 증권법 (Howey Test) 규제 대상
- 토큰 없이도 Multisig 외부 과반 + Timelock + emergency exit 으로
  사용자 보호와 탈중앙화를 충분히 달성 가능
- Gnosis Safe, 초기 Arbitrum, 초기 Optimism 등 선례 다수
- 필요 시 미래에 재검토하되, 현재 로드맵에서는 제외

### 권한 분리 규칙

| 권한 | 보유자 | 비고 |
|------|--------|------|
| 자금 입금 (deposit) | buyer 본인 서명 | Haggle 키 불필요 |
| 자금 해제 (release) | smart contract 로직 | 조건 충족 시 자동 |
| 자금 환불 (refund) | smart contract 로직 | 타임아웃 또는 분쟁 결과 |
| emergency exit | 사용자 본인 | Timelock 기간 중 가능 |
| 컨트랙트 업그레이드 | Timelock + Multisig | 자금 접근 권한 없음 |
| 컨트랙트 일시 중지 | Multisig (긴급용) | pause 만 가능, 자금 인출 불가 |
| 분쟁 결과 제출 | Haggle oracle | 실행은 컨트랙트가 수행 |

### 업그레이드 안전 규칙

1. **자금 인출 함수에 Multisig 키를 절대 넣지 않는다**
2. **Timelock 기간 중 emergency exit 함수는 항상 활성 상태여야 한다**
3. **업그레이드 트랜잭션은 온체인에 공개되어 누구나 검증 가능해야 한다**
4. **Proxy 패턴 사용 시 admin key 와 자금 접근 로직을 완전히 분리한다**
5. **컨트랙트 pause 는 신규 입금만 차단하고, 기존 자금 인출은 항상 허용한다**

### 업그레이드 가능 컨트랙트와 "Control" 해석

업그레이드 권한이 있다고 해서 자금 통제권이 있는 것은 아니다.

- 업그레이드 권한 = 컨트랙트 로직 변경 (Timelock 대기 필수)
- 자금 통제권 = 자금을 직접 이동시키는 능력 (Haggle 은 이 권한 없음)

Timelock 이 이 구분을 보장한다:
- 악의적 업그레이드가 제안되더라도 48시간 동안 실행 불가
- 사용자는 이 기간 동안 emergency exit 으로 자금 회수 가능
- 따라서 업그레이드 권한이 실질적 자금 통제로 이어지지 않는다

### 분쟁 해결 Oracle 패턴

분쟁 시 Haggle 이 자금 방향을 "결정"하지만, "실행"은 컨트랙트가 한다.

```
[분쟁 발생]
  → offchain case system 에서 review
  → Haggle oracle 이 resolution hash 를 컨트랙트에 제출
  → 컨트랙트가 resolution 에 따라 자금 분배 실행
  → Haggle 은 자금에 직접 접근하지 않음
```

이 패턴에서 Haggle 의 역할은 "판사"이지 "금고지기"가 아니다.

## 법적 구조

### 연방 레벨 (FinCEN)

Haggle 의 non-custodial Payment Protection 구조는 FinCEN 의 두 가지 면제에 해당한다.

#### 1. Integral Part 면제 (FIN-2008-R004, FIN-2014-R005)

Payment Protection 이 상품 거래의 필수 불가결한 부분이면 Money Transmitter 가 아니다.

충족 조건:
- **Integral Part Test**: 자금 이동이 본업(상품 거래)에 필수 → Haggle 은 물건 거래 플랫폼
- **Active Facilitation**: AI 협상, 배송 확인, 분쟁 해결 등 적극적 거래 관리
- **Conditional Release**: 스마트 컨트랙트가 조건 충족 시 자동 해제

#### 2. Non-Custodial DApp 면제 (FIN-2019-G001)

> "A DApp developer is not a money transmitter merely by creating an application...
> A developer would have to use that DApp to conduct money transmission
> in order to be considered a money transmitter."

Haggle 은 자금을 accept/transmit 하지 않으므로 이 면제에 해당한다.

### 주 레벨

- **Utah HB 230** (2025.5): 블록체인 활동 보호, 셀프커스터디 보호, MTL 면제
- **New York BitLicense**: Non-custodial 플랫폼에는 직접 적용되지 않음 (Uniswap 선례)
  - 단, NYDFS 가 향후 입장을 바꿀 수 있으므로 모니터링 필요
- **GENIUS Act** (2025.7): 연방 스테이블코인 법. 발행사 규제이며 머천트/마켓플레이스는 대상 아님

### Fiat 온램프 전략

사용자가 달러로 결제하고 싶은 경우를 위한 선택적 경로:

- Stripe 또는 Circle 의 fiat-to-USDC 온램프를 선택적으로 제공
- 이 경우 Fiat↔Crypto 전환의 규제 부담은 Stripe/Circle 이 부담
- Haggle 은 USDC 레이어에서만 동작
- 핵심 결제 rail 은 x402 + USDC 이며, fiat 온램프는 부가 옵션

### 필수 컴플라이언스 (MTL 과 별개)

Money Transmitter 가 아니더라도 아래는 필요하다:

- KYC/AML: BSA 의무로 사용자 신원 확인 필요
- Sanctions screening: OFAC 리스트 확인
- Travel Rule: 일정 금액 이상 거래 시 정보 전달
- 스케일 시 핀테크 변호사 자문 필수

### 핵심 원칙 요약

```
✅ Haggle 은 자금에 대한 키를 보유하지 않는다 (non-custodial)
✅ Payment Protection 은 상품 거래의 필수 기능이다 (integral part)
✅ 업그레이드 권한과 자금 접근 권한은 완전히 분리된다
✅ Timelock + emergency exit 으로 사용자 보호를 보장한다
✅ 핵심 rail 은 x402 + USDC 이며, fiat 온램프는 선택적 부가 옵션이다
```
