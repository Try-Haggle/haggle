컨트랙트 구조
Haggle은 2개의 스마트 컨트랙트로 구성됩니다. 하나는 결제를 처리하고, 다른 하나는 분쟁 기록을 보관합니다.
💡 쉽게 이해하기

정산 라우터는 은행 창구 직원과 비슷합니다. 구매자가 돈을 보내면, 판매자에게 상품 대금을, 플랫폼에 수수료를 각각 나눠서 전달합니다. 단, 이 "창구 직원"은 절대 돈을 자기가 가지지 않습니다 — 모든 돈은 바로 전달됩니다.

분쟁 레지스트리는 공증사무소와 비슷합니다. "이 거래에서 이런 분쟁이 있었고, 이렇게 해결됐다"는 기록을 블록체인에 영구 보관합니다. 나중에 어느 쪽도 "그런 적 없다"고 주장할 수 없습니다.

정산 라우터 (Settlement Router)

결제 담당
구매자의 USDC(스테이블코인)를 판매자와 수수료 지갑으로 나눠서 보냅니다.
핵심: 우리가 사용자 돈을 보유하지 않습니다 — 모든 전송은 즉시 실행됩니다.
구매자 ──[승인]──→ 라우터 (검증) ├──[전송]──→ 판매자 지갑 └──[전송]──→ 수수료 지갑
  • ✓ 구매자 본인만 결제 가능 (사칭 방지)
  • ✓ 백엔드 서명 검증 (파라미터 조작 방지)
  • ✓ 주문당 1회만 결제 (중복 결제 방지)
  • ✓ 허용된 자산만 사용 가능
  • ✓ 수수료 최대 10% 상한
  • ✓ 서명자 교체 시 48시간 대기
  • ✓ 긴급 시 즉시 정지 가능

분쟁 레지스트리 (Dispute Registry)

분쟁 기록 담당
분쟁 해결 결과를 블록체인에 영구 기록합니다.
핵심: 어느 쪽도 기록을 부인하거나 조작할 수 없습니다.
분쟁 해결사 ──[기록]──→ 레지스트리 │ (주문ID, 분쟁ID, 증거, 결과) │ └──→ 영구 보존 기록 (앵커)
  • ✓ 역할 분리 (소유자 ≠ 분쟁 해결사)
  • ✓ 소유권 이전 시 2단계 확인
  • ✓ 주문당 최대 50개 기록 (서비스 방해 방지)
  • ✓ 같은 분쟁 중복 기록 방지
  • ✓ 잘못된 기록 교체/무효화 가능 (감사 추적 유지)

🔗
분쟁 데이터와 유저의 연결 구조
유저 정보는 블록체인에 올라가지 않습니다. 블록체인에는 "지문(해시)"만 저장되고, 실제 데이터는 우리 서버 DB에 있습니다.
💡 쉽게 이해하기

블록체인에는 "이 거래에서 분쟁이 있었고, 이렇게 판결됐다"는 지문(해시)만 남깁니다. "김철수가 구매자다"같은 개인정보는 우리 서버에만 있습니다.

블록체인 기록의 목적은 나중에 Haggle이 판결을 몰래 바꿨다고 할 수 없게 하는 것입니다. 공증사무소에 도장 찍는 것과 같습니다.

유저 레이어 브라우저 / 앱

구매자가 "물건이 안 왔어요"라고 분쟁을 엽니다. → 증거 제출 → 판결 결과 확인

↓ API 호출

서버 레이어 Haggle API + PostgreSQL

분쟁의 전체 과정이 여기에서 처리됩니다 — 분쟁 오픈, 증거 수집, 검토, 판결까지 전부 오프체인.

dispute_cases (분쟁 내역) dispute_evidence (증거) dispute_resolutions (판결 결과) commerce_orders (주문 → buyer_id, seller_id) trust_penalty_records (신뢰 페널티)
↓ 분쟁 해결 후 해시값만 전송

블록체인 레이어 Base L2

분쟁이 해결된 후에만 최종 결과의 해시를 기록합니다. 진행 중인 분쟁은 블록체인에 올라가지 않습니다.

keccak256(주문ID) keccak256(분쟁ID) keccak256(증거 머클루트) keccak256(판결 결과)
ℹ️ 왜 해결 후에만 기록하나요?

분쟁이 열린 시점에는 아직 증거도 다 안 모였고 판결도 안 났습니다. 블록체인에 올리는 건 "이런 증거를 검토해서 이렇게 판결했다"는 최종 결과의 공증이므로, 모든 절차가 끝난 후에 기록하는 것이 맞습니다.

🌳
머클 트리 신뢰도 시스템
유저의 신뢰 점수는 우리 서버(오프체인)에서 관리하고, 주기적으로 "요약 지문" 1개만 블록체인에 기록합니다. 유저가 몇 명이든 블록체인 비용은 동일합니다.
💡 머클 트리란?

데이터 여러 개를 2개씩 묶어서 요약하고, 또 묶어서 요약하고... 최종적으로 "지문 1개"로 만드는 방법입니다.
도서관에 책 1,000권이 있을 때, 전부 비교하는 대신 "지문 1개"만 보관하면 어떤 책이든 원본인지 증명할 수 있습니다.

핵심: 유저가 1만 명이든 1억 명이든, 블록체인에 저장하는 건 항상 32바이트 해시 1개뿐입니다.

💡 왜 첫 디포짓 무료인가?

판매자 온보딩 장벽 최소화 — 디포짓은 첫 판매자에게 심리적 부담.
첫 거래를 무료로 시작하게 하고, 문제가 생기면 사후 청구. 대부분의 첫 거래는 정상 완료되므로 플랫폼 리스크는 낮으면서 판매자 전환율은 크게 높아집니다.

📌 예시: 조작된 증거로 생성된 기록 → 무효 처리 + 사유 기록
