# 신뢰_원장_업데이트_규칙

## 목적

이 문서는 Haggle의 `trust ledger` 가 어떤 이벤트로 갱신되는지와,
그 결과가 `settlement_reliability` 및 온체인 신뢰 프로필에 어떻게 반영되는지 정의한다.

## 원칙

- trust ledger 는 payment / shipping / dispute 의 후행 결과를 기록한다
- 단순 협상 행동이 아니라 `승인 이후 실제 이행` 중심으로 기록한다
- penalty 는 actor 별로 누적된다
- 누적 결과는 `settlement_reliability` 로 정규화된다

## penalty 트리거

### payment
- `buyer_approved_but_not_paid`
  - buyer penalty
  - 이유: `BUYER_APPROVED_BUT_NOT_PAID`

- `seller_approved_but_not_fulfilled`
  - seller penalty
  - 이유: `SELLER_APPROVED_BUT_NOT_FULFILLED`

### shipping
- `shipment_input_sla_missed`
  - seller penalty
  - 이유: `SHIPMENT_INFO_SLA_MISSED`

### dispute
- `dispute_loss`
  - 패소한 actor penalty
  - 이유: `DISPUTE_LOSS`

### positive events
- `successful_settlement`
  - successful settlement 카운트 증가

- `dispute_win`
  - dispute_wins 카운트 증가

## penalty 기본 점수

- `BUYER_APPROVED_BUT_NOT_PAID` = `0.35`
- `SELLER_APPROVED_BUT_NOT_FULFILLED` = `0.40`
- `SHIPMENT_INFO_SLA_MISSED` = `0.20`
- `DISPUTE_LOSS` = `0.30`

이 점수는 trust ledger 의 raw record 로 남고,
최종 공개값은 누적 정규화된 `settlement_reliability` 로 사용한다.

## settlement reliability 계산

권장 계산식:

`reliability = (successful_settlements + dispute_wins * 0.2) / (successful_settlements + approval_defaults * 1.5 + shipment_sla_misses * 1.0 + dispute_losses * 1.2 + dispute_wins * 0.2)`

의미:
- approval default 는 가장 강한 부정 신호
- shipment SLA miss 는 중간 수준 부정 신호
- dispute loss 는 강한 부정 신호
- dispute win 은 소폭의 신뢰 보정

## 온체인 프로필 반영

아래 값은 온체인 프로필에 앵커링될 수 있다.

- `reputation_score`
- `settlement_reliability`
- `successful_settlements`
- `approval_defaults`
- `shipment_sla_misses`
- `dispute_wins`
- `dispute_losses`
- `expertise[]`

## 전문 분야 반영

분쟁/이행 결과는 전체 점수뿐 아니라 도메인별 배지에도 반영할 수 있어야 한다.

예:
- 전자기기 거래 성공 누적 → `electronics.successful_orders` 증가
- 명품 분쟁 패소 누적 → `luxury.dispute_losses` 증가

즉 같은 actor 라도 분야별 신뢰 프로필은 다를 수 있다.
