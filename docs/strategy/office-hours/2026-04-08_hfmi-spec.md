# HFMI (Haggle Fair Market Index) — Technical Spec v0

Generated 2026-04-08 as companion to `2026-04-08_week1-2_work-list.md`
Status: ENGINEERING SPEC — READY FOR BUILD
Scope: Phase 0 v0, iPhone 14 Pro single SKU, US market

---

## 1. Purpose

HFMI is Haggle's open fair-price index for used Apple devices. Two roles:

1. **Consumer-facing anchor**: Display on listing pages so buyers and sellers negotiate against a shared reference point.
2. **North Star attribution**: `savings = max(0, HFMI_median − final_price)` — the metric that proves Haggle delivers value.

Secondary role (Phase 1+): **B2B API product** — refurb shops, insurers, accountants, developers query HFMI programmatically.

---

## 2. Design Principles

1. **Open methodology** — formula, coefficients, R², sample size all public. Trust comes from transparency.
2. **Single source in v0** — multi-source weighted averages create structural bias (Gazelle buyback is 30-50% below market, Back Market refurb is a ceiling). Better to be narrow and honest than broad and biased.
3. **Hedonic regression, not grid median** — grid approach (model × storage × battery × cosmetic) has cell-level data sparsity. Regression pools information across dimensions.
4. **Internal data is the moat, not the v0** — in cold start (<50 txns) internal weight is 0. As transactions accumulate, internal data becomes the dominant signal by Phase 1.
5. **Consumer UI always free, always current** — differentiation is API access (Phase 1), never freshness or accuracy on consumer side. Fairness principle (CLAUDE.md #4).

## 3. Formula

### 3.1 Hedonic OLS Regression

```
log(price_usd) = β₀
               + β₁ · I(storage=256)
               + β₂ · I(storage=512)
               + β₃ · I(storage=1024)
               + β₄ · battery_health_pct
               + β₅ · I(cosmetic=B)
               + β₆ · I(cosmetic=C)
               + β₇ · I(carrier_locked)
               + β₈ · days_since_listing
               + ε
```

Baseline (all indicator vars = 0): **128 GB, cosmetic grade A, unlocked, listed today**.

**Why log(price)**:
- Price effects are multiplicative (10% premium for 256GB, not flat $60)
- Residuals closer to normal
- Coefficients interpretable as percent changes

**Why these variables only**:
- `days_since_listing` captures depreciation drift (even within 30-day window)
- `battery_health_pct` is continuous — linear assumption acceptable in 80-100 range
- Color excluded (weak price signal, high cardinality)
- Seller reputation excluded (scraped eBay data doesn't normalize cleanly)

### 3.2 Prediction

```typescript
function hfmiMedian(params: {
  storage: 128 | 256 | 512 | 1024
  battery_health: number  // 0-100
  cosmetic: 'A' | 'B' | 'C'
  carrier_locked: boolean
}): { median: number; ci_low: number; ci_high: number } {
  const coef = latestCoefficients('iphone_14_pro')
  const log_p =
      coef.intercept
    + coef.storage_256   * (params.storage === 256 ? 1 : 0)
    + coef.storage_512   * (params.storage === 512 ? 1 : 0)
    + coef.storage_1024  * (params.storage === 1024 ? 1 : 0)
    + coef.battery       * params.battery_health
    + coef.cosmetic_b    * (params.cosmetic === 'B' ? 1 : 0)
    + coef.cosmetic_c    * (params.cosmetic === 'C' ? 1 : 0)
    + coef.carrier_locked * (params.carrier_locked ? 1 : 0)
    + coef.days_since_listing * 0  // t=0 = today

  const median = Math.exp(log_p)
  const ci_low  = Math.exp(log_p - 1.96 * coef.residual_std)
  const ci_high = Math.exp(log_p + 1.96 * coef.residual_std)
  return { median, ci_low, ci_high }
}
```

## 4. Data Sources

### 4.1 Primary — eBay Sold Listings

**API**: eBay Finding API `findCompletedItems` (legacy but still live)

**Query**:
```
categoryId=9355  (Cell Phones & Smartphones)
aspectFilter:
  - Model: "Apple iPhone 14 Pro"
  - Network: "Unlocked" | "T-Mobile" | "Verizon" | "AT&T"
  - Storage Capacity: "128 GB" | "256 GB" | "512 GB" | "1 TB"
itemFilter:
  - SoldItemsOnly: true
  - LocatedIn: US
  - ListingType: FixedPrice
  - EndTimeFrom: now - 30 days
sortOrder: EndTimeSoonest
paginationInput: entriesPerPage=100, pageNumber=1..N
```

**Fetch cadence**: Hourly cron.

**Rate limit**: Finding API allows 5,000 calls/day per app ID. Hourly fetch with 5 paginated pages per SKU variant = ~20 calls/hour = 480/day. Well within limit.

**Known limitations**:
- Finding API deprecation risk (eBay announced migration to Browse/Buy APIs, but `findCompletedItems` still active as of 2026-04)
- Listing title noise (sellers misuse "14 Pro" for 14 and 14 Pro Max)
- Condition mapping eBay's taxonomy → Haggle's A/B/C grading (see §5.1)

**Fallback**: If Finding API returns empty or errors, use cached coefficients from previous nightly fit. Alert on-call if stale > 48h.

### 4.2 Marketplace Insights API — Phase 0.5

- Application submission: **Day 1 of Week 1** (critical path)
- Approval timeline: 2-4 weeks
- Access: sold listings with structured data (no scraping needed)
- Migration path: swap ingestion source, keep same `hfmi_price_observations` schema

### 4.3 Anchor Sources (Sanity Checks Only, Weight = 0)

**Gazelle** (`gazelle.com`):
- Method: Manual weekly sampling by ops team (10 SKU configs)
- Use: `alert if HFMI_median < gazelle_trade_in_price` — means our index is broken
- Rationale: Gazelle trade-in is floor price. Any C2C median should be above it.

**Back Market** (`backmarket.com`):
- Method: Manual weekly sampling (refurbished, "Excellent" grade)
- Use: `alert if HFMI_median > backmarket_refurb_price` — means our index is broken
- Rationale: Back Market refurb is ceiling price. C2C private sale should be below.

Both anchors surfaced in `/admin/hfmi-sanity` dashboard. No automatic ingestion in v0.

### 4.4 Haggle Internal — Phase 0.5+

- Enters regression when Haggle completed transactions ≥50 for 14 Pro
- Replaces eBay as primary source when ≥500 transactions (Gate 2 threshold)
- Transition: weight interpolation, not hard cutover
  ```
  w_haggle = min(0.60, n_haggle / (n_haggle + n_ebay + 50))
  w_ebay   = 1 - w_haggle
  ```
- Implemented as per-observation weight in regression (WLS, not OLS) when `w_haggle > 0`

## 5. Data Normalization

### 5.1 eBay Condition → Haggle Cosmetic Grade

| eBay Condition | Haggle Grade | Notes |
|---|---|---|
| New | Excluded | Sealed retail, different market |
| Open Box | A | Used but minimal wear |
| Certified Refurbished | Excluded | Back Market territory |
| Excellent - Refurbished | Excluded | Back Market territory |
| Very Good - Refurbished | Excluded | Back Market territory |
| Good - Refurbished | Excluded | Back Market territory |
| Seller Refurbished | B | Uncertified, treat as mid-tier |
| Used | B | Default C2C grade |
| For Parts or Not Working | Excluded | Different market |

### 5.2 Storage Parsing

Regex on listing title + eBay aspect `Storage Capacity`:
- `/\b(128|256|512)\s*gb\b/i` → 128, 256, 512
- `/\b1\s*tb\b/i` → 1024
- Listings without parseable storage → **excluded**

### 5.3 Battery Health

**Problem**: eBay listings rarely include battery health.

**v0 approach**: When battery_health not in listing, **impute using model mean** (first fit without battery var, then fit with battery var on subset that has it, then propagate mean to rest).

**Better approach** (Phase 0.5): Ops team manually extracts battery % from listing photos for a labeled sample (100-200 listings), fits a listing-title-based classifier to predict battery band for unlabeled.

**Haggle internal listings**: Battery health is required at attestation (screenshot), so 100% coverage after transition.

### 5.4 Title Noise Filtering

Exclude listings where title contains (case-insensitive):
- `case`, `charger`, `cable`, `adapter` alone (accessories)
- `broken`, `cracked`, `for parts`, `not working`
- `lot of`, `bulk`, `wholesale` (bulk listings)
- `bad esn`, `blacklisted`, `icloud locked`, `activation locked`
- Any listing with final price < $200 or > $1,500 (outliers, 14 Pro realistic range $350-1,100)

### 5.5 Outlier Removal

After normalization, compute rolling MAD (median absolute deviation) per `(storage, cosmetic)` cell. Drop observations where `|log(price) − median(log(price))| > 3 · MAD`.

## 6. Fitting Pipeline

### 6.1 Nightly Job

Cron: `0 3 * * *` (3 AM UTC, low traffic window)

```
1. Load hfmi_price_observations WHERE
     source = 'ebay_sold'
     AND model = 'iphone_14_pro'
     AND observed_at >= now - 30 days
     AND price BETWEEN 200 AND 1500
     AND cosmetic IS NOT NULL
     AND storage IS NOT NULL

2. Apply §5.5 outlier removal

3. Fit OLS regression (simple-statistics or mathjs)
   - Solve (X'X)⁻¹X'y via Cholesky
   - Compute β̂, R², residual std, sample size

4. Insert row into hfmi_model_coefficients
   - fitted_at = now
   - model = 'iphone_14_pro'
   - coefficients = {intercept, storage_256, ..., residual_std}
   - r_squared, sample_size

5. If R² < 0.50 OR sample_size < 30:
     - Do NOT update active coefficients
     - Alert on-call
     - Continue serving previous coefficients

6. Regenerate tryhaggle.ai/hfmi methodology page (static MDX rebuild)
```

### 6.2 Active Coefficient Selection

```sql
SELECT * FROM hfmi_model_coefficients
WHERE model = 'iphone_14_pro'
  AND r_squared >= 0.50
  AND sample_size >= 30
ORDER BY fitted_at DESC
LIMIT 1
```

In-memory cache refresh on pod start + hourly TTL.

### 6.3 Health Metrics

Track on `/admin/hfmi-health`:
- Latest fit timestamp, R², sample size
- 7-day R² trend (should be stable 0.75-0.90)
- Gazelle/Back Market sanity check status
- Coefficient drift (flag if any |β_new − β_old| / |β_old| > 20%)

## 7. Schema (PostgreSQL via Drizzle)

```typescript
// packages/db/src/schema/hfmi-observations.ts
export const hfmiPriceObservations = pgTable('hfmi_price_observations', {
  id: uuid('id').primaryKey().defaultRandom(),
  source: text('source', { enum: ['ebay_sold', 'gazelle', 'backmarket', 'haggle_internal'] }).notNull(),
  model: text('model').notNull(),                     // 'iphone_14_pro'
  storageGb: integer('storage_gb'),                   // 128 | 256 | 512 | 1024
  batteryHealthPct: integer('battery_health_pct'),    // nullable in v0
  cosmeticGrade: text('cosmetic_grade', { enum: ['A', 'B', 'C'] }),
  carrierLocked: boolean('carrier_locked').notNull().default(false),
  observedPriceUsd: numeric('observed_price_usd', { precision: 10, scale: 2 }).notNull(),
  observedAt: timestamp('observed_at').notNull(),
  externalId: text('external_id'),                    // eBay item ID for dedup
  rawPayload: jsonb('raw_payload'),
  createdAt: timestamp('created_at').notNull().defaultNow(),
}, (t) => ({
  sourceModelIdx: index('hfmi_obs_source_model_idx').on(t.source, t.model, t.observedAt),
  externalIdUnique: unique('hfmi_obs_external_id_unique').on(t.source, t.externalId),
}))

// packages/db/src/schema/hfmi-coefficients.ts
export const hfmiModelCoefficients = pgTable('hfmi_model_coefficients', {
  id: uuid('id').primaryKey().defaultRandom(),
  model: text('model').notNull(),
  fittedAt: timestamp('fitted_at').notNull(),
  coefficients: jsonb('coefficients').notNull(),       // { intercept, storage_256, ..., residual_std }
  rSquared: numeric('r_squared', { precision: 5, scale: 4 }).notNull(),
  sampleSize: integer('sample_size').notNull(),
  fitVersion: text('fit_version').notNull(),           // 'v0.1.0' etc, bump on formula change
  createdAt: timestamp('created_at').notNull().defaultNow(),
}, (t) => ({
  modelFittedIdx: index('hfmi_coef_model_fitted_idx').on(t.model, t.fittedAt),
}))
```

Migrations live in `packages/db/migrations/`. No writes to existing tables.

## 11. Open Questions (Bob's Decisions)

1. **OLS implementation**: `simple-statistics` only supports simple linear regression (one predictor). For multi-variable we need `ml-regression-multivariate-linear` or manual Cholesky via `mathjs`. Prefer pure TS to avoid Python microservice. **Recommend**: `ml-regression-multivariate-linear` (1.6M downloads, MIT).
2. **eBay client**: `ebay-api` npm package exists but is heavyweight. A focused 200-line client is probably cleaner. **Bob picks.**
3. **Battery health imputation**: proceed with model-mean approach in v0? Yes, document as limitation on methodology page.
4. **`hfmi-core` package vs inline in `apps/api`**: new package gives isolation + test boundary. **Recommend**: new package, follow `engine-core`/`dispute-core` pattern.
5. **Caching**: Redis for coefficient cache vs in-memory per-pod? In-memory with 1h TTL is simpler and fine for v0.
